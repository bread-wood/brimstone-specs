# breadwinner v0.3.0 — Iron Harvest

**Theme:** Backtesting engine, IV infrastructure, options lifecycle automation, and signal
pipeline.

---

## Overview

Iron Harvest turns breadwinner from a portfolio tracker into a systematic income engine.
Adds four primitives for disciplined premium selling: a **backtesting engine** to validate
strategies on history, an **IV infrastructure layer** to time entries correctly,
**options position management** to enforce proven rules (21 DTE, 50% profit, delta breach),
and a **signal pipeline** to surface opportunities and produce a daily actionable brief.
A lightweight **composer research loop** ties it together so strategy research sprints
produce testable parameters rather than prose.

**Prerequisite:** v0.2.0-option-forge must complete (options positions, exit rules,
scanner, and execution pipeline must exist before the backtester and signal pipeline
can use them).

## Success Criteria

- [ ] `bw backtest run <strategy> --start <date> --end <date>` replays the plan engine
  over a historical date range and prints annualized Sharpe, max drawdown, win rate,
  and benchmark comparison (SPY)
- [ ] `bw bars fetch <ticker> --start --end` pre-populates a local OHLCV bar cache;
  the backtester runs entirely from local cache with no live API calls during replay
- [ ] `bw iv fetch` collects ATM implied volatility for all watchlist tickers and stores
  it in `iv_history`; `bw iv rank <ticker>` returns IV rank and IV percentile
- [ ] `bw options scan` returns a ranked list of CSP and covered-call candidates meeting
  configurable delta, DTE, and IV rank thresholds
- [ ] The daily heartbeat flags open option positions that have hit the 50% profit
  target, crossed 21 DTE, or breached the configured delta threshold
- [ ] `bw brief` produces a plain-text morning summary covering: positions needing
  attention, IV rank for watchlist tickers, top technical signals, upcoming earnings
  within 14 days — formatted for Telegram dispatch, under 4096 characters
- [ ] `bw backtest export <run-id>` produces a markdown summary of a completed backtest
  run readable by composer research agents
- [ ] All new features covered by unit tests; `make test && make lint` pass on mainline

## Scope

**Included:**

- **Backtesting engine**: local bar cache (`bars fetch`), simulated execution with
  configurable slippage (default 0.1%), equity curve, metrics output, backtest export
- **IV infrastructure**: daily IV snapshot collection (`iv fetch`), IV rank and percentile
  computation, integration into portfolio snapshot and scanner
- **Options lifecycle**: opportunity scanner with IV rank gate, 21-DTE / 50%-profit /
  delta-breach lifecycle rules, roll plan generator
- **Signal pipeline**: technical signal scan (RSI, MACD, Bollinger Bands), morning brief
  aggregator (`bw brief`)
- **Composer research loop**: backtest result schema, `bw backtest export` command

**Explicitly excluded:**

- moot deliberation integration (v0.4.0-crossing)
- Live automated execution (all signals surface to `bw execute-trades`, not auto-dispatched)
- Multi-leg complex options beyond CSP, covered call, vertical spreads
- Real-time intraday signals (all end-of-day, daily bar based)
- Portfolio optimization / mean-variance allocation
- Web UI or TUI dashboard

## Failure Modes

**Backtesting:**
- Plan engine call fails for a historical date (e.g. data gap) → log warning, skip that
  date, continue the replay; include skipped dates in the export summary
- Bar cache miss during replay → abort the backtest with an error message directing the
  user to run `bw bars fetch` first; never fall back to live API during replay
- Backtest produces no trades → complete normally; report zero trades in metrics

**IV fetch:**
- Alpaca option chain unavailable for a ticker → skip that ticker; log warning; do not
  fail the entire fetch
- ATM strike selection heuristic ambiguous → use the nearest strike to current price,
  log the strike selected; document the heuristic in the code

**Morning brief:**
- A signal source fails (heartbeat, IV rank, technicals) → include results from
  available sources; note missing sections in the brief with a warning line
- Brief exceeds 4096 characters → truncate with a "…(truncated)" trailer; log the
  overflow

## Interface Contracts

**Backtest run output (schema for `export` command):**

```json
{
  "run_id": "<uuid>",
  "strategy": "<strategy_name>",
  "start": "<YYYY-MM-DD>",
  "end": "<YYYY-MM-DD>",
  "metrics": {
    "sharpe": 1.32,
    "max_drawdown_pct": -12.4,
    "win_rate": 0.68,
    "total_trades": 47,
    "benchmark_sharpe": 0.91
  },
  "equity_curve": [[date, value]],
  "trade_log": [{"date", "action", "symbol", "pnl"}]
}
```

**`bw brief` output:**

Plain text, Telegram-safe, ≤ 4096 characters. Sections:
1. Positions needing attention (exit rules triggered, roll candidates)
2. IV rank for watchlist (sorted by rank descending)
3. Top technical signals (RSI oversold/overbought, MACD crosses)
4. Upcoming earnings within 14 days

## Constraints

- Backtester must use the existing `plan_engine.generate_plan()` and `exit_engine`
  without forking them; time-travel via injected `as_of_date` parameter
- SQLite remains the only persistence layer
- IV fetch integrates into `update-state` as an optional step, not a separate scheduled job
- Slippage model is configurable (default 0.1%); commissions not modeled in v0.3.0
- `bw brief` output: plain text, Telegram-safe, under 4096 characters

## Key Unknowns

1. **[P1]** IV data availability via Alpaca: does Alpaca's option snapshot endpoint
   provide reliable 30-day ATM IV per ticker? Specifically: which expiry and strike
   selection heuristic yields a consistent IV proxy, and is the data available for all
   watchlist tickers (including less-liquid names)? If unreliable, an alternative
   (yfinance, CBOE data) may be needed. Research stage must confirm before IV
   infrastructure is designed.

2. **[P1]** Plan engine time-travel compatibility: the existing `planning.py` six-phase
   pipeline makes several calls that assume "today" (market clock, latest bars, account
   state). Can these be injected with a historical `as_of_date`, or do they require
   mocking? The answer determines whether the backtester wraps the plan engine or
   re-implements a subset. Research stage must audit `planning.py` and its dependencies
   before the backtesting engine is designed.
