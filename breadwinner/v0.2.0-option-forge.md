# breadwinner v0.2.0 — Option Forge

**Theme:** Build the full options layer on top of the existing equity engine.

---

## Overview

Adds comprehensive options trading to the breadwinner engine. The existing equity
pipeline (planning, execution, exit rules, heartbeat) is extended to handle options
positions end-to-end: scanning candidates, evaluating exits, placing multi-leg orders,
tracking fills, and monitoring Greeks. Dividend tracking and a rebalancing engine round
out the equity side. An integration test suite validates the full stack against a real
Alpaca paper account.

## Scope

- Strategy YAML schema extension for options and crypto
- `dividends.py`: corporate actions fetch, ex-date calendar, yield-on-cost

- `services/execution.py`: multi-leg order placement, fill tracking, roll logic

- Integration test suite in `tests/integration/`
- Rename `src/breadwinner/llm/` to `src/breadwinner/prompts/`; the directory contains
  prompt builders, response parsers, and domain context types — not LLM provider code;

## Failure Modes

**Options execution:**
- None-price in option chain → log warning, skip order rather than crash; no silent failures
- Multi-leg order partial fill → record partial fills; heartbeat flags incomplete rolls
- Assignment detected at reconciliation → update position state and log; surface in heartbeat

**Options scanning:**
- Option chain unavailable for a ticker → skip that ticker; log warning; do not fail the scan
- Earnings blackout overlap not detectable (no calendar data) → conservatively reject;
  log that earnings data was unavailable

**Rebalancing:**
- Wash sale detection false negative → rebalancing proposals include a disclaimer;
  compliance layer flags for manual review
- Drift below threshold → no rebalancing proposals; log that portfolio is within bounds

## Interface Contracts

**`OptionProposal` (planning → execution):**

```python
class OptionProposal(BaseModel):
    strategy_type: OptionStrategyType
    underlying: str
    legs: list[OptionLeg]
    net_debit_credit: float  # negative = credit
    rationale: str
    urgency: str
    action: Literal["open", "close", "roll"]
```

**`OptionScanResult` (scanner → planning context):**

```python
@dataclass
class OptionScanResult:
    underlying: str
    strategy_type: OptionStrategyType
    current_price: float
    suggested_strike: float
    suggested_expiry: date
    est_premium: float | None
    rationale: str
    constraints_met: list[str]
    constraints_failed: list[str]
```

## Key Unknowns

1. **[P1]** Roll atomicity via Alpaca: does Alpaca's options API support simultaneous
   close + open as a single order, or must rolls be two separate orders? The answer
   determines whether `services/execution.py` can treat a roll as atomic or must manage
   the two-leg sequence explicitly (with failure handling between legs). Research stage
   must check Alpaca docs / paper account behavior before execution design is finalized.

2. **[P2]** Covered-call delta tracking: when a call goes deep ITM and delta approaches
   1.0, should the exit rule trigger on the option's delta alone, or on the net delta of
   (short call + underlying)? The answer affects how `evaluate_option_exit_rules()`
   queries live Greeks. Research stage should confirm the correct risk measure for
   covered calls vs naked puts.
