# wowbot v0.6.0 — Sentinel

**Theme:** The bot watches itself. Behavioral evasion becomes active.

## Overview

Adds an active behavioral self-monitoring layer that tracks the bot's own
session patterns — kill rate, movement entropy, session duration, action
timing distributions — and introduces deliberate variation when patterns
drift toward bot-like regularity. Where earlier versions minimise individual
action detectability (humanized input, natural movement), Sentinel addresses
the aggregate: a session that looks statistically indistinguishable from a
player over hours and days. Adds configurable rest cycles, session length
bounds, and a telemetry stream that surfaces behavioral metrics for human
review.

---

## Success Criteria

- [ ] `wowbot farm --zone <zone> --spec <spec> --session-max 90m` stops the
  session after 90 minutes regardless of remaining work and exits 0
- [ ] `wowbot farm` with `--session-max` and `--session-min` picks a random
  session length uniformly between the two bounds for each invocation
- [ ] After each session, `wowbot` writes a behavioral metrics report to
  `WOWBOT_LOG_DIR` including: kill rate mean and coefficient of variation,
  movement path entropy score, action timing jitter, session duration
- [ ] When kill rate CV falls below `WOWBOT_MIN_KILL_RATE_CV` (indicating
  too-regular farming), the bot inserts a randomized pause of 30–120 s before
  the next pull
- [ ] `wowbot status` includes a behavioral summary: current kill rate,
  CV trend (rising/falling/stable), session elapsed, configured max
- [ ] An anomalous death rate (> `WOWBOT_DEATH_ALERT_RATE` deaths/hour)
  triggers a Telegram alert with zone, spec, and recent behavioral metrics
- [ ] All existing `wowbot` commands (farm, gather, level, dungeon) support
  `--session-max` and `--session-min` flags
- [ ] `make test` passes clean with no live WoW instance

---

## Scope

**Included:**

- Session length bounds: `--session-max` and `--session-min` flags across all
  bot modes; actual session length chosen uniformly at random within bounds on
  each invocation
- Kill rate tracking: rolling window kill rate with coefficient of variation;
  automatic pause insertion when CV falls below threshold
- Movement entropy tracking: measure variance in path taken between pull
  positions; surface as a score in behavioral metrics
- Action timing jitter: measure deviation of action timing from the baseline
  distribution set in v0.1.0; surface as a metric
- Behavioral metrics report: JSON file written to `WOWBOT_LOG_DIR` at end of
  each session
- `wowbot status` behavioral summary: live view of key metrics during a session
- Death rate alerting: Telegram notification when deaths/hour exceeds threshold
- All metrics computed from internal bot telemetry only; no network calls to
  external services for behavioral analysis

---

## Failure Modes

**Behavioral monitoring:**
- Kill rate CV cannot be computed (fewer than 10 kills in window) → skip CV
  check for this window; do not insert pause; log warning
- Movement entropy score unavailable (navigation not used in current mode) →
  omit from metrics report; not an error

**Session management:**
- `--session-min` > `--session-max` → `error: session-min must be less than session-max` to stderr, exit 1
- Session ends mid-combat → finish current combat; then exit cleanly

**Alerting:**
- Telegram alert fails → log to `WOWBOT_LOG_DIR`; continue session; do not
  halt on alert delivery failure

## Interface Contracts

**CLI additions (all modes):**

```
wowbot <mode> ... [--session-min <duration>] [--session-max <duration>]
```

| Flag | Default | Purpose |
|------|---------|---------|
| `--session-min` | `--session-max` | Minimum session length (if set without max, exact duration) |
| `--session-max` | unlimited | Maximum session length; actual length randomized between min and max |

**Behavioral metrics report (written to `WOWBOT_LOG_DIR/metrics-<timestamp>.json`):**

```json
{
  "session_at": "<ISO8601>",
  "duration_seconds": 3450,
  "mode": "farm",
  "zone": "The Ringing Deeps",
  "spec": "frost-mage",
  "kill_rate": {
    "mean_per_hour": 312,
    "cv": 0.18
  },
  "movement_entropy_score": 0.74,
  "action_timing_jitter_ms": 42,
  "pauses_inserted": 3,
  "deaths": 1,
  "death_rate_per_hour": 1.04
}
```

**Environment variables (new in v0.6.0):**

| Variable | Required | Default | Purpose |
|----------|----------|---------|---------|
| `WOWBOT_MIN_KILL_RATE_CV` | No | `0.12` | CV below which a randomized pause is inserted |
| `WOWBOT_DEATH_ALERT_RATE` | No | `3.0` | Deaths/hour above which a Telegram alert is sent |
| `TELEGRAM_BOT_TOKEN` | No | — | Telegram bot token for death rate alerts |
| `TELEGRAM_CHAT_ID` | No | — | Telegram chat ID for alerts |

## Constraints

- Depends on v0.5.1 for the full mode set (farm, gather, level, dungeon);
  `--session-max` and `--session-min` must work uniformly across all modes
- All behavioral metrics are derived from internal telemetry; no external
  behavioral analysis services
- The behavioral monitoring layer must not add more than 5% overhead to the
  main bot loop
- Pause insertion must not occur during active combat or while the bot is
  navigating a critical path (e.g., mid-corpse-run)
- macOS only in this version

---

## Key Unknowns

1. **[P1]** Can the bot accurately measure its own behavioral regularity —
   kill rate variance, movement path entropy, action timing jitter — using
   only internal telemetry, or does accurate self-assessment require comparison
   against an external baseline of human player behavior to calibrate what
   "sufficiently irregular" means? The answer determines whether the behavioral
   monitoring layer can be self-contained or requires a reference dataset.

2. **[P2]** Does introducing deliberate pauses and session length randomization
   meaningfully reduce the statistical signal available to server-side behavioral
   analysis — given that the underlying action patterns (rotation, movement,
   loot) remain bot-generated — or does evasion at the session level require
   changes to the lower-level action patterns that would undo the consistency
   gains from v0.1.0 humanization? The answer determines whether Sentinel's
   approach addresses a real detection surface or the wrong layer.

---

## Non-Goals

- Dynamic rotation variation (deliberately suboptimal play to appear human)
- AI-generated behavioral patterns (ML model trained on human session data)
- Account-level analysis across characters or sessions (single-session scope only)
- Automatic detection of Warden scan events
- Windows support
