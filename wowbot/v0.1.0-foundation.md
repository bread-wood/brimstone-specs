# wowbot v0.1.0 — Foundation

**Theme:** Read the game. Send input. Everything else builds on this.

## Overview

Establishes the two capabilities every subsequent version depends on: reading the game
window and sending humanized input. Ships a `wowbot status` command that reads and prints
current game state — confirming the screen reading pipeline works — and a fixture
infrastructure that lets every future version run its tests without a live game instance.
No bot behaviour yet; this version proves the platform is viable before any automation
is built on top of it.

---

## Success Criteria

- [ ] `wowbot status` prints current game state (character name, zone, health %, mana %,
  whether the character is in combat) to stdout as JSON and exits 0
- [ ] `wowbot status` when WoW is not running prints `error: WoW process not found` to
  stderr and exits 1
- [ ] `wowbot status` when no character is logged in prints `error: character not in
  world` to stderr and exits 1
- [ ] `wowbot status --fixture <path>` produces output matching the fixture's expected
  JSON without a live WoW instance and exits 0
- [ ] `wowbot record --output <path>` records a session fixture from a live game session
  until stopped with Ctrl-C and exits 0
- [ ] `wowbot` with no subcommand prints usage to stderr and exits 1
- [ ] `make test` passes clean with no live WoW instance

---

## Scope

**Included:**

- Game window detection: locate the WoW process and its window
- Screen reading: capture game window contents and extract structured game state
  (character name, zone, health, mana, combat status)
- Humanized input primitives: mouse movement, mouse click, keyboard press with
  randomized timing and natural movement characteristics
- Fixture infrastructure: record a live session to disk; replay it offline for testing
- Session logging: timestamped action log; screenshots saved on extraction failure
- `wowbot status`: read and print current game state
- `wowbot record`: record a session fixture

---

## Failure Modes

**Game state:**
- WoW process not found → `error: WoW process not found` to stderr, exit 1
- Character not in world → `error: character not in world` to stderr, exit 1
- Game state extraction fails (unrecognised UI layout) →
  `error: game state unreadable; screenshot saved to <path>` to stderr, exit 1

**Fixture:**
- Fixture file not found or unreadable → `error: fixture not found: <path>` to stderr, exit 1
- Fixture format invalid → `error: invalid fixture: <path>` to stderr, exit 1

## Interface Contracts

**CLI:**

```
wowbot status [--fixture PATH]
wowbot record --output PATH
```

| Stream | Content |
|--------|---------|
| stdout | JSON game state on success |
| stderr | `error: <description>` on any failure |
| exit 0 | Success |
| exit 1 | Any error |

**Status JSON schema:**

```json
{
  "read_at": "<ISO8601>",
  "realm": "<realm-name>",
  "character": "<name>",
  "zone": "<zone-name>",
  "health_pct": 100,
  "mana_pct": 85,
  "in_combat": false
}
```

**Environment variables:**

| Variable | Required | Purpose |
|----------|----------|---------|
| `WOWBOT_WOW_PATH` | No | Path to WoW installation; auto-detected if absent |
| `WOWBOT_LOG_DIR` | No | Session logs and error screenshots (default: `~/.wowbot/logs`) |

## Constraints

- Targets WoW Midnight (retail); UI layout assumptions are Midnight-specific
- No memory reading; all data extracted from the game window
- Input simulation must include humanized timing — no fixed-interval events
- Must not inject code, hook game functions, or modify game files
- Fixture replay must run without a live WoW instance (required for CI)
- The screen reading and input approaches must be designed for the full performance
  range of the roadmap: low-frequency AH interactions in v0.1.1–v0.2.1 and
  high-frequency combat inputs in v0.4.x
- macOS only in this version

---

## Key Unknowns

1. **[P1]** What screen reading approach is undetectable by Warden on macOS across
   both the low-frequency AH interactions in v0.1.1 and the high-frequency reads
   required by the combat rotation engine in v0.4.x? The answer determines the capture
   architecture; an approach that works for AH but cannot meet combat timing requirements
   would require a redesign at v0.4.x.

2. **[P1]** What input simulation approach is indistinguishable from human behaviour
   by Warden across both sparse AH interactions and rapid combat key sequences? The
   answer determines the input layer design; if the humanisation strategy that passes
   for AH is detectably different from what passes for combat, the two use cases need
   separate designs from the start.

---

## Non-Goals

- AH navigation or listing extraction (v0.1.1)
- Any market analysis or transactions (v0.2.x)
- Waypoint navigation or gathering (v0.3.x)
- Combat or farming (v0.4.x)
- Windows support
