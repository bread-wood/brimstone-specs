# wowbot v0.0.x — Threat Model

**Theme:** Understand the detection surface before building anything.

## Overview

A research-only version that produces a threat model covering every detection surface
the bot will encounter across the full v0.1–v0.4 roadmap. Ships no code. Ships a threat
model document that each subsequent spec's Key Unknowns draw from, so that Warden
questions are answered once with rigour rather than re-asked and partially answered at
every version. No implementation begins until this threat model is complete.

---

## Success Criteria

- [ ] `docs/threat-model.md` exists in the wow-bot repo and covers all five threat
  surfaces defined in Scope
- [ ] Each threat surface has a documented answer with an explicit confidence level:
  `[DOCUMENTED]`, `[INFERRED]`, or `[UNKNOWN]`
- [ ] Every `[UNKNOWN]` claim has a stated reason why it could not be determined and
  a recommended mitigation under uncertainty
- [ ] Each answered question includes at least one concrete constraint it places on
  the implementation — not just what Warden does, but what the bot must do differently
  because of it
- [ ] The threat model is reviewed against the Key Unknowns of v0.1.0 through v0.4.1
  and each one is marked resolved or escalated with a rationale

---

## Scope

**Included — five threat surfaces to document:**

- **Screen reading**: which capture approaches are visible to Warden on macOS; what
  distinguishes them from legitimate application window access; whether capture
  frequency affects detectability
- **Input simulation**: how Warden distinguishes simulated input from human input;
  what timing, trajectory, and event-level signals are inspected; what humanisation
  is sufficient vs. insufficient
- **Movement patterns**: what movement signatures Warden or server-side analysis flags;
  how flight path regularity, waypoint deviation, and turn behaviour are evaluated;
  what variation is required
- **Behavioural analysis**: how sustained repetitive behaviour (AH scanning, gathering
  loops, combat rotations, farming sessions) is distinguished from legitimate play;
  what cadence, duration, and activity-switch patterns avoid detection
- **Session and account analysis**: what server-side signals feed into ban decisions;
  how session duration, gold accumulation rate, kill rate, and AH transaction volume
  are monitored; what thresholds or patterns are known to trigger review

---

## Failure Modes

This version ships a document, not code. Failure modes are document quality failures:

- A threat surface is documented without any actionable implementation constraint →
  the surface is marked incomplete and must be revisited before the first version
  that touches it is dispatched
- A claim is marked `[DOCUMENTED]` without a citable source → claim is downgraded
  to `[INFERRED]`
- The threat model is completed but a downstream spec's Key Unknown is not resolved
  by it → that Key Unknown becomes a follow-up research issue before that spec's
  design stage begins

## Constraints

- This version produces no code; no implementation issues are filed until the threat
  model is accepted
- Claims must be grounded in observable evidence (public ban reports, reverse
  engineering disclosures, known detection events) — not speculation about what
  Warden might do
- The threat model must cover the full v0.1–v0.4 roadmap, not just the first version;
  a threat model that only considers AH scanning would be incomplete
- Where evidence is genuinely unavailable, the document must say so explicitly and
  specify a conservative safe-by-default constraint rather than assuming safety

## Key Unknowns

1. **[P1]** What is Warden's current capability and scan frequency on macOS Midnight —
   specifically, does it inspect window capture API calls, process memory access patterns,
   or only in-game UI state? The answer establishes the outer boundary of what is and
   is not detectable and determines whether any off-process approach is viable.

2. **[P1]** What input event properties distinguish simulated from human input at the
   driver, OS, and application layer on macOS — and has Warden historically acted on
   any of these signals, or only on behavioural patterns in aggregate? The answer
   determines whether low-level input fidelity matters or whether timing distribution
   is the only signal that counts.

3. **[P1]** What server-side behavioural signals are used in Blizzard's ban process,
   and how do they interact with client-side Warden findings — specifically, does
   a clean Warden report protect against a server-side behavioural flag, or are the
   two independent detection paths? The answer determines whether Warden evasion is
   sufficient or whether server-side behavioural constraints must be designed for
   independently.

---

## Non-Goals

- Any implementation, scaffolding, or code
- Documenting threats outside the macOS / WoW Midnight scope
- Comprehensive reverse engineering of Warden internals beyond what is necessary to
  answer the five threat surfaces
