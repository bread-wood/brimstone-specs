# wowbot v0.5.2 — Fixer

**Theme:** Sell the result, not the items. Service economy at scale.

## Overview

Orchestrates paid carry runs: advertises boosting services in trade chat,
accepts whispered commissions, verifies gold payment, invites the buyer into
a group, runs them through the configured content, and confirms delivery. The
bot interacts directly with other players through the in-game chat and group
system — the highest-risk, highest-reward mode in the campaign. Gold per hour
ceiling is an order of magnitude above farming because the product is guaranteed
completion of content players cannot clear themselves. Requires explicit threat
model review before dispatch; a dedicated threat model addendum is a prerequisite
for this version's design stage.

---

## Success Criteria

- [ ] `wowbot fixer --content <dungeon> --price <gold>` posts an advertisement
  in trade chat on a configured interval, accepts the first valid whispered
  commission, invites the buyer, clears the content with them in the group,
  and exits 0 with a run report
- [ ] Run report includes: content cleared, gold collected, buyer character name,
  duration, deaths
- [ ] The bot verifies gold payment via trade window before the group enters
  the instance; if payment is not completed within `WOWBOT_PAYMENT_TIMEOUT_SECONDS`,
  the group is disbanded and the bot re-advertises
- [ ] The bot detects if the buyer drops from the group mid-run; if the buyer
  leaves before completion, the bot exits 1 with `error: buyer left group`
- [ ] `wowbot fixer --content <dungeon> --price <gold> --runs <N>` completes
  N consecutive carry runs before exiting
- [ ] `wowbot fixer --content <dungeon> --price <gold> --dry-run` posts one
  advertisement, reads incoming whispers, and prints them without accepting or
  forming a group
- [ ] `make test` passes clean with no live WoW instance

## Scope

**Included:**

- Trade chat advertisement: post a configurable message on a configurable interval
- Whisper reading: detect incoming whispers expressing intent to buy
- Commission acceptance: whisper back a confirmation message and payment instructions
- Payment verification: initiate a trade window; wait for the buyer to place the
  correct gold amount; confirm trade
- Group management: invite buyer; confirm group is ready before entering instance
- Instance clear: use v0.5.1 dungeon mode with the buyer present as a group member
- Buyer presence detection: monitor group roster; detect if buyer leaves
- Run report: JSON written to stdout on exit
- Multi-run mode: repeat the full loop N times before exiting

## Failure Modes

**Advertisement:**
- Trade chat throttled (spam filter) → back off exponentially; resume posting
  when throttle clears; log warning

**Commission:**
- Multiple buyers whisper simultaneously → queue in order received; serve one
  at a time; whisper others "currently running, will contact you when available"
- Buyer whispers but does not follow up on payment instructions → timeout after
  `WOWBOT_PAYMENT_TIMEOUT_SECONDS`; cancel; move to next in queue

**Payment:**
- Buyer places incorrect gold amount → decline trade; whisper correct amount;
  wait for retry; timeout if no retry within `WOWBOT_PAYMENT_TIMEOUT_SECONDS`
- Trade window not recognised → save screenshot; log error; exit 1

**In-run:**
- Buyer leaves group mid-run → exit 1 with `error: buyer left group`; do not
  refund (operator decision)
- Instance clear fails (3 wipes) → exit 1 with `error: repeated wipes, halting`;
  refund policy is operator decision; bot does not initiate refund automatically

## Interface Contracts

**CLI:**

```
wowbot fixer --content <dungeon> --price <gold> [--runs <N>] [--ad-interval <seconds>] [--dry-run]
```

| Flag | Default | Purpose |
|------|---------|---------|
| `--content` | required | Dungeon or content to carry |
| `--price` | required | Gold to charge per run |
| `--runs` | unlimited | Number of carry runs before exit |
| `--ad-interval` | `120` | Seconds between trade chat advertisements |
| `--dry-run` | off | Post ads and read whispers without accepting |

**Run report JSON schema:**

```json
{
  "session_at": "<ISO8601>",
  "content": "The Deadmines",
  "runs_completed": 3,
  "runs_attempted": 3,
  "total_gold_collected": 15000,
  "runs": [
    { "buyer": "Goldsworth", "gold_collected": 5000, "duration_seconds": 1640, "deaths": 0 }
  ]
}
```

**Environment variables:**

| Variable | Required | Default | Purpose |
|----------|----------|---------|---------|
| `WOWBOT_WOW_PATH` | No | auto | Path to WoW installation |
| `WOWBOT_LOG_DIR` | No | `~/.wowbot/logs` | Session logs |
| `WOWBOT_PAYMENT_TIMEOUT_SECONDS` | No | `120` | Max wait for payment before cancelling |

## Constraints

- Depends on v0.5.1 for dungeon clearing; the carry run uses the dungeon mode
  engine with a real player in the group instead of all-bot composition
- Depends on v0.1.0 for input primitives (chat reading/writing, trade window)
- **Prerequisite**: a threat model addendum covering chat interaction and
  inter-player social detection must be accepted before this version's design
  stage begins; this is a hard gate, not a soft recommendation
- The bot handles payment collection but does not enforce a refund policy;
  refund decisions are operator responsibility
- Single buyer per run; no group carries (multiple buyers, one run)
- macOS only in this version

## Key Unknowns

1. **[P1]** Can in-game chat reading and writing — both trade chat broadcasts and
   whisper parsing — be performed through screen reading and input simulation
   without triggering Warden's bot detection at a higher rate than other bot
   modes? Chat interaction is uniquely visible: it produces social traces,
   is rate-limited by Blizzard's spam detection, and creates a record observable
   by GMs. The answer determines whether this mode is viable at all or is
   fundamentally too high-risk to operate.

2. **[P1]** Is payment verification via trade window reliable enough to prevent
   gold theft (buyer initiates trade, places wrong amount, bot confirms) —
   specifically, can the bot read the exact gold amount in the trade window
   before confirming, and does the confirmation sequence have enough time for
   the bot to validate? The answer determines the payment verification protocol
   and the financial risk per run.

3. **[P2]** Does running content with a real player in the group — one whose
   movement and actions are not controlled by the bot — create dungeon-clear
   failure modes that the v0.5.1 engine is not designed to handle, such as the
   buyer pulling extra mobs, dying in unusual locations, or triggering mechanics
   the bot has not encountered in solo testing? The answer determines whether
   the dungeon engine requires a "passenger mode" that accounts for unpredictable
   group member behavior.

## Non-Goals

- Group carries (multiple buyers per run)
- Raid carries
- Automated refunds
- Reputation management across buyers
- Cross-character coordination for simultaneous carries
- Windows support
