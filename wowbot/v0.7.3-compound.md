# wowbot v0.7.3 — Compound

**Theme:** The operation runs itself. Capital breeds capital.

## Overview

A capital allocation layer that treats the gold-making operation as a portfolio
with measurable returns per strategy. Tracks total capital — liquid gold plus
AH inventory value plus outstanding order fees — measures gold-per-hour for
each active mode across recent sessions, and automatically scales budgets and
session parameters as capital grows. An AH flipper constrained to 10,000g
budget earns less than it could; Compound removes the constraint by reinvesting
profits back into the strategies producing the best returns. The bot that earns
1,000g/hour today earns 5,000g/hour in a month — not because the strategies
improved, but because the capital compounded.

## Success Criteria

- [ ] `wowbot compound --status` prints current capital breakdown: liquid gold,
  AH inventory value (at current median), outstanding fees, total, and
  30-day P&L
- [ ] `wowbot compound --allocate` reads session history from `WOWBOT_LOG_DIR`,
  computes gold-per-hour per mode, and writes an updated allocation config
  to `--output`; exits 0 with allocation summary to stdout
- [ ] The allocation config produced by `--allocate` is a valid campaign config
  consumable by `wowbot clockwork --campaign`
- [ ] `wowbot compound --allocate` increases the AH flip budget when the flipper's
  30-day gold/hour exceeds the farming mode's gold/hour by more than
  `WOWBOT_REALLOC_THRESHOLD_PCT`
- [ ] `wowbot compound --allocate` reduces budget for a mode whose gold/hour has
  declined more than `WOWBOT_DECAY_THRESHOLD_PCT` from its 7-day peak
- [ ] `wowbot compound --allocate --dry-run` prints the proposed allocation
  without writing it
- [ ] After 30 days of operation with Compound active, total gold/hour across
  all modes is measurably higher than it was on day 1 (verified from session
  logs)
- [ ] `make test` passes clean

## Scope

**Included:**

- Capital ledger: reads liquid gold from game state; reads AH inventory from
  active listing data; reads outstanding work order fees from Contractor
  session logs; computes total capital
- ROI model: reads session reports from all modes in `WOWBOT_LOG_DIR`;
  computes gold/hour per mode using configurable rolling windows (7-day, 30-day)
- Allocation engine: given ROI model, apply allocation rules to produce updated
  budgets and session parameters per mode
- Allocation rules: configurable thresholds for reallocation and decay detection;
  minimum floor allocation per mode (no mode goes to zero)
- Output: allocation config as a valid `wowbot clockwork` campaign config file
- `--status` command: human-readable capital and P&L snapshot
- `--allocate` command: compute and write updated allocation
- Drift detection: flag modes whose ROI has changed significantly since last
  allocation (useful for detecting Sentinel-triggered behavioral changes)

## Failure Modes

**Capital measurement:**
- Liquid gold unreadable (game not running or WoW path not set) → `--status`
  reports `liquid_gold: unknown`; allocation proceeds using last known value
  with a staleness warning
- AH inventory valuation fails (AH scan unavailable) → use last known inventory
  value; flag as stale in `--status` output

**ROI model:**
- Insufficient session history for a mode (fewer than 3 sessions in window) →
  exclude mode from reallocation; log warning `insufficient history for <mode>`;
  do not reduce that mode's budget based on missing data
- All modes have insufficient history → `--allocate` exits 0 with
  `no allocation changes: insufficient history` and makes no writes

**Allocation:**
- Proposed allocation would reduce a mode below its configured floor →
  clamp at floor; log the constraint
- Proposed allocation exceeds total liquid gold → scale all budgets proportionally
  to fit within liquid gold; log the scaling

## Interface Contracts

**CLI:**

```
wowbot compound --status
wowbot compound --allocate [--output <path>] [--dry-run] [--window <days>]
```

| Flag | Default | Purpose |
|------|---------|---------|
| `--status` | — | Print capital snapshot and exit |
| `--allocate` | — | Compute and write updated allocation config |
| `--output` | `~/.wowbot/allocation.json` | Path for allocation config output |
| `--dry-run` | off | Print proposed allocation without writing |
| `--window` | `30` | Days of session history to use for ROI model |

**`--status` output (stdout):**

```
Capital snapshot (as of 2026-03-03 14:22 UTC)

Liquid gold:         248,400 g
AH inventory:         84,200 g  (31 active listings at current median)
Outstanding fees:      6,000 g  (3 work orders in progress)
─────────────────────────────
Total capital:       338,600 g

30-day P&L:          +82,400 g  (+32.2%)

Mode ROI (30-day avg):
  ah-flip:    1,840 g/hr
  farm:         420 g/hr
  de-arb:       680 g/hr
  contractor:   910 g/hr

Last allocation:  2026-02-28  Next suggested: now (decay detected in farm)
```

**Allocation config output:** Valid `wowbot clockwork` campaign config JSON with
updated budget and session parameters per mode.

**Environment variables:**

| Variable | Required | Default | Purpose |
|----------|----------|---------|---------|
| `WOWBOT_LOG_DIR` | No | `~/.wowbot/logs` | Source of session history |
| `WOWBOT_REALLOC_THRESHOLD_PCT` | No | `25` | ROI advantage required to trigger reallocation |
| `WOWBOT_DECAY_THRESHOLD_PCT` | No | `20` | ROI decline from 7-day peak that triggers budget reduction |

## Constraints

- Depends on v0.7.0 (Ledger) for price history used in AH inventory valuation
- Depends on v0.7.1 (Syndicate) for multi-character capital aggregation; on a
  single character, liquid gold is read from game state directly
- Depends on v0.6.1 (Clockwork) for the campaign config format consumed by
  `--allocate` output
- Compound does not run any bot mode itself; it reads history and writes configs
  consumed by other commands
- Allocation runs are operator-initiated (`--allocate`); there is no automatic
  reallocation trigger in this version
- The ROI model uses raw gold/hour from session reports; it does not risk-adjust
  or account for variance between sessions

## Key Unknowns

1. **[P1]** Is a 30-day rolling gold/hour average a sufficient ROI signal for
   allocation decisions — given that mode profitability is cyclical (raid resets,
   patch cycles, time-of-day effects) and a 30-day average may lag behind rapid
   changes — or does the model need to be time-aware to avoid reallocating capital
   away from a mode that's temporarily underperforming due to a known cyclical
   factor? The answer determines the ROI model's time resolution and whether
   seasonal adjustments are required from the start.

2. **[P2]** Does AH inventory valuation at current median price provide a reliable
   capital figure — given that posting a large inventory moves the market and
   the bot may never realise median price on all units — or does the model need
   a sell-through discount (e.g., assume 70% of inventory value is realisable)
   to avoid systematically overstating capital and over-allocating budgets? The
   answer determines the inventory valuation methodology and the margin of safety
   built into allocation decisions.

---

## Non-Goals

- Automatic reallocation without operator invocation
- Risk-adjusted returns (Sharpe ratio, variance modeling)
- Multi-account capital aggregation beyond v0.7.1 Syndicate scope
- Tax or fee modeling beyond AH cut
- Predictive modeling of future ROI (allocation is backward-looking only)
- Windows support
