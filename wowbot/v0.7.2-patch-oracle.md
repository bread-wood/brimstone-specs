# wowbot v0.7.2 — Patch Oracle

**Theme:** Read the future. Trade on what's coming.

## Overview

Monitors public WoW patch notes and PTR data sources for economy-affecting
changes — item buffs and nerfs, recipe additions and removals, vendor price
changes, crafting requirement changes — and translates them into actionable
trade signals before the patch drops. When a new raid tier is announced, mat
prices will spike on patch day; when a crafting recipe is nerfed, its output
loses value; when an item is added to a vendor, the player-crafted version
crashes. Patch Oracle reads these signals days or weeks in advance and positions
the bot's inventory and posting strategy accordingly. The WoW economy's version
of trading on public information.

---

## Success Criteria

- [ ] `wowbot oracle --watch` fetches patch notes and PTR data on a configured
  interval, emits signals for economy-affecting changes, and runs until stopped
- [ ] When a patch note mentions buffing a crafted item, Oracle emits a BUY_MATS
  signal for the relevant crafting reagents within one fetch cycle
- [ ] When a patch note mentions nerfing a crafted item's output, Oracle emits
  a SELL_FINISHED_GOODS signal for current inventory of that item
- [ ] When an item previously only craftable by players is added to a vendor,
  Oracle emits a DUMP signal with urgency `high`
- [ ] `wowbot oracle --status` prints all active signals with their source,
  confidence, and recommended action, then exits 0
- [ ] `wowbot oracle --dry-run` fetches and parses the current patch notes,
  prints detected signals without writing them to the signal store, and exits 0
- [ ] Each signal includes: item or mat affected, signal type, confidence level
  (`[HIGH]`/`[MEDIUM]`/`[LOW]`), source (PTR note URL + patch version), and
  recommended action
- [ ] `wowbot oracle --watch` sends a Telegram alert for every `[HIGH]` confidence
  signal within one fetch cycle of detection
- [ ] `make test` passes clean

## Scope

**Included:**

- Patch note fetching: periodic retrieval from configured public sources
  (PTR patch notes, official patch notes, datamine feeds)
- Change parsing: extract item changes, recipe changes, vendor changes, and
  source-of-truth changes (item moved from craft-only to vendor-sold)
- Impact classification: map each change to an economic signal type:
  `BUY_MATS`, `BUY_FINISHED`, `SELL_FINISHED`, `DUMP`, `HOLD`, `WATCH`
- Confidence assignment: classify confidence based on source reliability and
  signal specificity
- Signal store: persistent local store of active signals with expiry
  (signals expire one week after the patch they reference goes live)
- Signal integration hooks: other bot modes can query the signal store before
  deciding which items to buy, craft, or post
- Telegram alerts for `[HIGH]` confidence signals
- `--status` command: snapshot of current active signals

---

## Failure Modes

**Fetching:**
- Source URL unreachable → log warning; retain last-fetched data; retry on
  next cycle; do not emit stale signals as new

**Parsing:**
- Change text does not match any known pattern → log the raw text as
  `[UNPARSED]`; do not emit a signal; include in `--status` output for
  human review
- Parsed item name not found in local item database → emit `[LOW]` confidence
  signal with item name as-is; flag for human review

**Signal store:**
- Store file corrupt → rebuild from last successful fetch; log warning; emit
  no signals until next successful parse cycle

## Interface Contracts

**CLI:**

```
wowbot oracle --watch [--interval <seconds>]
wowbot oracle --status
wowbot oracle --dry-run
```

| Flag | Default | Purpose |
|------|---------|---------|
| `--watch` | — | Run continuous fetch-parse-signal loop |
| `--interval` | `3600` | Seconds between fetch cycles |
| `--status` | — | Print active signals and exit |
| `--dry-run` | — | Fetch and parse once; print signals without storing |

**Signal schema:**

```json
{
  "signal_id": "<uuid>",
  "detected_at": "<ISO8601>",
  "patch_version": "11.1.5 PTR",
  "source_url": "https://...",
  "item_id": 229001,
  "item_name": "Algari Competitor's Band",
  "signal_type": "BUY_MATS",
  "confidence": "HIGH",
  "rationale": "Item buffed: secondary stat increased 15%. Demand expected to spike.",
  "recommended_action": "Buy crafting mats before patch drops.",
  "expires_at": "<ISO8601>"
}
```

**Signal types:**

| Type | Meaning |
|------|---------|
| `BUY_MATS` | Accumulate crafting mats; finished goods will be more valuable |
| `BUY_FINISHED` | Accumulate the item itself; value will rise |
| `SELL_FINISHED` | Sell current inventory of the item; value will fall |
| `DUMP` | Sell immediately at any price; item is being made obsolete |
| `HOLD` | Do not buy or sell; situation unclear |
| `WATCH` | Monitor price; change detected but impact uncertain |

**Environment variables:**

| Variable | Required | Default | Purpose |
|----------|----------|---------|---------|
| `WOWBOT_LOG_DIR` | No | `~/.wowbot/logs` | Signal store and logs |
| `WOWBOT_ORACLE_SOURCES` | No | built-in list | Comma-separated list of patch note source URLs |
| `TELEGRAM_BOT_TOKEN` | No | — | Telegram bot token for HIGH signal alerts |
| `TELEGRAM_CHAT_ID` | No | — | Telegram chat ID for alerts |

## Constraints

- Oracle does not execute any trades; it produces signals consumed by other modes
- All data sources are public URLs; no authenticated or private data sources
- Signals expire automatically; no manual signal management required
- The item database mapping item names to item IDs is a local data file;
  Oracle does not fetch it at runtime
- Confidence levels are heuristic — `[HIGH]` means the source is a confirmed
  patch note and the item is named explicitly; `[LOW]` means the source is
  a datamine and the item is inferred

## Key Unknowns

1. **[P1]** Can economy-affecting changes be reliably extracted from patch note
   text using pattern matching against known change templates ("X increased by Y%",
   "X is now available from vendor Z") — or is the natural language variation in
   patch notes high enough that reliable extraction requires a more sophisticated
   parsing approach? The answer determines whether signal extraction is a rules
   engine or needs a language model in the parsing pipeline.

2. **[P2]** How far in advance do PTR patch notes reliably appear before a patch
   goes live — and is the lead time consistent enough to act on signals before
   prices move? If the market prices in PTR changes within hours of note
   publication, the Oracle's edge may be narrower than expected. The answer
   determines the practical value of the pre-patch positioning strategy and
   whether the signal expiry window needs to be much shorter than one week.

---

## Non-Goals

- Executing trades directly (signals only; other modes consume them)
- Monitoring community forums or social media for rumours
- Predicting undocumented changes
- Cross-realm price impact modeling
- Windows support
