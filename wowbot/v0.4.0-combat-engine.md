# wowbot v0.4.0 — Combat Engine

**Theme:** Target, fight, loot. Proves the combat loop before sustained farming.

## Overview

Executes a combat loop against a single target: detect the nearest valid mob, engage,
execute the class rotation, loot the corpse, and repeat until stopped. No sustained
farming session management yet — no death recovery, no bank runs, no zone-wide loop.
This version proves that the combat engine works correctly and safely before the full
farming harness is added in v0.4.1. Depends on v0.1.0 for screen reading and input;
builds on the detection foundations from v0.3.1 where available.

---

## Success Criteria

- [ ] `wowbot combat --spec <spec> --duration 5m` targets nearby mobs, executes the
  rotation, loots kills, and runs for 5 minutes; exits 0 with a combat report to stdout
- [ ] Combat report includes: mobs killed, items looted, gold looted, and duration
- [ ] The rotation keeps the character alive against single-target mobs of equal level
  in ≥ 90% of encounters over a 30-minute fixture simulation
- [ ] `wowbot combat --spec <spec>` when the logged-in character does not match the spec
  prints `error: character spec does not match <spec>` to stderr and exits 1
- [ ] `wowbot combat --spec <spec>` when required abilities are not on the action bars
  prints `error: required abilities not found on action bars` to stderr and exits 1
- [ ] `wowbot combat` with no arguments prints usage to stderr and exits 1
- [ ] `make test` passes clean with no live WoW instance

---

## Scope

**Included:**

- Mob targeting: detect the nearest valid, uncontested mob via the game UI; skip mobs
  in combat with another player
- Combat rotation: one hard-coded spec (determined at design time); monitors cooldowns
  and proc indicators from the game UI to sequence abilities
- Health and resource monitoring: trigger defensive cooldowns and consumable use below
  configurable thresholds
- Loot execution: interact with corpse, take all loot, close loot window
- Spec validation: confirm character spec and required abilities match before starting
- Combat report: JSON written to stdout on exit

---

## Failure Modes

**Combat:**
- Multiple mobs aggro → use AoE rotation if spec supports it; disengage toward safe
  position if health drops below 20%
- CC or silence → wait for debuff to expire; resume rotation
- Mob in evade state → abandon pull; blacklist mob position for 60 seconds; target next
- Target immune to all rotation abilities → abandon pull; target next; log warning

**Loot:**
- Loot window does not open within timeout → skip corpse; continue
- Loot contested → skip; continue

**Spec validation:**
- Spec mismatch → `error: character spec does not match <spec>` to stderr, exit 1
- Required abilities missing → `error: required abilities not found on action bars`
  to stderr, exit 1

## Interface Contracts

**CLI:**

```
wowbot combat --spec <class-spec> [--duration <duration>]
```

| Flag | Default | Purpose |
|------|---------|---------|
| `--spec` | required | Class spec identifier (e.g. `frost-mage`) |
| `--duration` | unlimited | Stop after this duration |

| Stream | Content |
|--------|---------|
| stdout | Combat report JSON on exit |
| stderr | `error: <description>` on fatal failure |
| exit 0 | Clean stop (Ctrl-C or duration elapsed) |
| exit 1 | Fatal error: spec mismatch, abilities missing |

**Combat report JSON schema:**

```json
{
  "session_at": "<ISO8601>",
  "duration_seconds": 300,
  "realm": "<realm-name>",
  "spec": "frost-mage",
  "mobs_killed": 28,
  "gold_looted": 120,
  "items_looted": { "Resonite Crystal": 4, "Ironclaw Fang": 9 }
}
```

**Environment variables:**

| Variable | Required | Purpose |
|----------|----------|---------|
| `WOWBOT_WOW_PATH` | No | Path to WoW installation; auto-detected if absent |
| `WOWBOT_LOG_DIR` | No | Session logs and error screenshots (default: `~/.wowbot/logs`) |

## Constraints

- Depends on v0.1.0 for screen reading and input
- One spec supported; spec chosen during design based on Midnight class balance and
  rotation state legibility from the game UI
- Combat rotation is hard-coded for the chosen spec; no dynamic priority system
- Bot must not initiate combat with PvP-flagged players
- All input uses humanized timing; no fixed GCD polling
- The spec design must consider extensibility: adding a second spec in v0.4.2 must not
  require redesigning the rotation engine, only adding a new rotation definition
- macOS only in this version

---

## Key Unknowns

1. **[P1]** Which class spec in Midnight has the highest solo survivability combined with
   the most legible rotation state from the game UI, and is this spec's rotation
   structure representative enough that the engine built for it can extend to a second
   spec without architectural changes? The answer determines which spec to implement first
   and whether the rotation engine is spec-agnostic or spec-specific.

2. **[P2]** Does Warden detect combat bots by GCD timing regularity, spell cast frequency,
   or session duration — and is this the same detection surface as the movement and AH
   automation in earlier versions, or does combat introduce a distinct detection profile?
   The answer determines the humanisation strategy for combat inputs.

## Non-Goals

- Death recovery (v0.4.1)
- Bank runs or inventory management (v0.4.1)
- Sustained farming loop with zone-wide targeting (v0.4.1)
- Multiple specs (v0.4.2)
- Dungeon or group content
- PvP
- Windows support
