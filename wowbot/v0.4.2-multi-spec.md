# wowbot v0.4.2 — Roster

**Theme:** More than one class. The rotation engine earns its generality.

## Overview

Extends the farming loop from v0.4.1 to support multiple class specs selectable
at runtime. In v0.4.0 a single hardcoded spec validated the rotation engine; in
v0.4.2 any spec can be run by supplying a spec configuration file. Adding a new
class spec requires writing a config file only — no code changes. This version
validates that the v0.4.0 rotation engine is genuinely data-driven and not
implicitly coupled to the proof-of-concept spec.

---

## Success Criteria

- [ ] `wowbot farm --zone <zone> --spec warrior-arms --duration 1h` runs the
  farming loop using the warrior arms rotation and exits 0 with a yield report
- [ ] `wowbot farm --zone <zone> --spec frost-mage --duration 1h` runs using
  the frost mage rotation — a spec distinct from the v0.4.0 proof-of-concept
- [ ] `wowbot farm --zone <zone> --spec <unknown>` prints
  `error: unknown spec: <unknown>` to stderr and exits 1
- [ ] A new spec is added by writing a single spec config file with no code
  changes; the new spec is immediately usable via `--spec`
- [ ] The spec config format documents: abilities in priority order, cooldown
  usage conditions, resource thresholds, and any spec-specific state machine
  transitions
- [ ] `wowbot farm --zone <zone> --spec warrior-arms` and
  `wowbot farm --zone <zone> --spec frost-mage` both produce valid yield reports
  using the same yield report schema from v0.4.1
- [ ] `make test` passes clean with no live WoW instance

---

## Scope

**Included:**

- Spec config file format: declarative description of a class rotation —
  ability priority list, cooldown conditions, resource thresholds
- Spec loader: reads the config file for the spec named in `--spec`
- Two shipped spec configs: one melee, one ranged caster (as proof of generality)
- Validation of spec config at load time: unknown abilities, malformed conditions
  produce a clear error before any farming begins
- All v0.4.1 farming loop behavior unchanged

---

## Failure Modes

**Spec loading:**
- Spec name not found → `error: unknown spec: <name>` to stderr, exit 1
- Spec config file malformed → `error: invalid spec config: <reason>` to stderr, exit 1
- Spec config references unknown ability → `error: unknown ability in spec: <name>` to stderr, exit 1

**Farming (inherited from v0.4.1):**
- All v0.4.1 failure modes apply unchanged

---

## Interface Contracts

**CLI:**

```
wowbot farm --zone <zone> --spec <spec-name> [--duration <duration>] [--bank-threshold <pct>]
```

Identical to v0.4.1. The `--spec` argument now resolves to a named spec config
file rather than a hardcoded spec.

**Spec config format (schema):**

```json
{
  "name": "frost-mage",
  "resource": "mana",
  "abilities": [
    { "name": "Frozen Orb", "type": "cooldown", "condition": "cooldown_ready" },
    { "name": "Frostbolt", "type": "filler" }
  ],
  "thresholds": {
    "resource_min_pct": 20
  }
}
```

| Stream | Content |
|--------|---------|
| stdout | Yield report JSON on exit (same schema as v0.4.1) |
| stderr | `error: <description>` on fatal failure |
| exit 0 | Clean stop |
| exit 1 | Fatal error |

**Environment variables:** Same as v0.4.1.

---

## Constraints

- Depends on v0.4.1 for the farming loop and v0.4.0 for the rotation engine
- Adding a new spec must require only a new JSON config file; no code changes
- The spec config schema must accommodate the full range of WoW class mechanics
  (resource types: mana, rage, energy, combo points, runes; cooldown categories;
  proc conditions) without requiring schema changes for each new class
- Two shipped configs must cover at least one melee and one ranged caster to
  validate generality

---

## Key Unknowns

1. **[P1]** Does the v0.4.0 rotation engine — designed around a single proof-of-concept
   spec — accommodate arbitrary class mechanics through configuration alone, or do
   class-specific state machines (e.g., combo point builders, rage generation,
   proc tracking) require code beyond what a declarative priority list can express?
   The answer determines whether the rotation engine is genuinely data-driven or
   needs architectural extension before new specs can be added without code changes.

---

## Non-Goals

- More than two shipped spec configs (community configs are out of scope)
- Spec auto-detection from equipped gear or talent tree
- Per-spec zone recommendations
- Dungeon or group specs — v0.5.1
