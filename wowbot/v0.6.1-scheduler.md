# wowbot v0.6.1 — Clockwork

**Theme:** Right mode at the right time. The bot reads the room.

## Overview

A population-aware scheduling layer that selects which bot mode to run based
on real-time signals about server economic conditions. AH modes are most
profitable when many buyers are active; farming modes are most efficient when
competition for mobs is low; work orders peak during raid prep nights. The
bot infers population and market state from AH activity signals — listing
velocity, bid activity, price compression — and routes accordingly, running
autonomously across mode boundaries without human intervention. A session that
starts as a farmer at 3am transitions to an AH flipper at 7pm without the
operator touching anything.

---

## Success Criteria

- [ ] `wowbot clockwork --campaign <config>` runs autonomously, selecting and
  switching modes based on market signals, for the configured session duration,
  and exits 0 with a session report
- [ ] Session report includes: time spent per mode, gold earned per mode,
  signal readings at each mode transition, total gold
- [ ] Given a simulated signal sequence (high AH velocity → low AH velocity →
  high AH velocity), the clockwork selects AH mode → farm mode → AH mode
  in that order
- [ ] Mode transitions are clean: the active mode completes its current atomic
  operation (current pull, current AH transaction) before the switch occurs
- [ ] `wowbot clockwork --campaign <config> --dry-run` prints the current signal
  readings and the mode it would select, then exits 0 without starting any mode
- [ ] `wowbot clockwork --campaign <config> --observe 60m` collects signal
  readings for 60 minutes, writes a signal log, and exits 0 without running
  any mode — for baseline calibration
- [ ] After 7 days of operation the scheduler's signal-to-mode decisions can
  be replayed from the signal log for audit
- [ ] `make test` passes clean with no live WoW instance

## Scope

**Included:**

- Signal collection: periodic AH scan measuring listing count, price compression
  ratio (spread between cheapest and median), and recent transaction velocity
  (items sold since last scan)
- Population inference: model server population from signal composite — high
  listing count + high velocity = high population; low count + low velocity =
  low population
- Mode routing table: operator-configured mapping of signal conditions to modes;
  configurable per time-of-day window and signal threshold
- Mode lifecycle management: launch, monitor, and cleanly terminate bot mode
  processes; pass session context (zone, spec, budget) from campaign config
- Mode transition: detect when signal conditions have shifted; complete current
  atomic operation; switch to new mode
- `--observe` mode: passive signal collection for calibration
- Signal log: append-only JSONL written to `WOWBOT_LOG_DIR` during operation
- Session report: JSON written to stdout on exit

---

## Failure Modes

**Signal collection:**
- AH scan fails → retain last known signals; log warning; retry on next cycle;
  do not switch modes on stale data older than `WOWBOT_SIGNAL_STALENESS_SECONDS`

**Mode lifecycle:**
- Mode process exits non-zero unexpectedly → log error; wait one signal cycle;
  re-evaluate and launch appropriate mode; do not re-launch the same mode
  immediately (may indicate a systematic failure)
- No mode matches current signal conditions → continue running the last active
  mode; log warning `no mode matches current signals: <signal summary>`

**Mode transition:**
- Active mode does not respond to clean-stop signal within
  `WOWBOT_MODE_TRANSITION_TIMEOUT_SECONDS` → send SIGTERM; log warning; proceed
  with transition

## Interface Contracts

**CLI:**

```
wowbot clockwork --campaign <config> [--duration <duration>] [--dry-run] [--observe <duration>]
```

| Flag | Default | Purpose |
|------|---------|---------|
| `--campaign` | required | Campaign config file (modes, thresholds, time windows) |
| `--duration` | unlimited | Total session duration before exit |
| `--dry-run` | off | Print current signal reading and mode selection; exit |
| `--observe` | off | Collect signals only; write signal log; no mode runs |

**Campaign config schema:**

```json
{
  "modes": [
    {
      "name": "ah-flip",
      "command": "wowbot ah flip --budget 50000",
      "conditions": {
        "population": "high",
        "time_windows": ["17:00-23:00"]
      }
    },
    {
      "name": "farm",
      "command": "wowbot farm --zone ringing-deeps --spec frost-mage --duration 30m",
      "conditions": {
        "population": "low"
      }
    }
  ],
  "signal_interval_seconds": 300
}
```

**Signal log event (JSONL):**

```json
{
  "ts": "<ISO8601>",
  "listing_count": 8420,
  "price_compression_ratio": 0.84,
  "transaction_velocity": 312,
  "inferred_population": "high",
  "active_mode": "ah-flip"
}
```

**Session report JSON schema:**

```json
{
  "session_at": "<ISO8601>",
  "duration_seconds": 18000,
  "total_gold": 28400,
  "modes": [
    { "name": "farm", "duration_seconds": 7200, "gold_earned": 4100 },
    { "name": "ah-flip", "duration_seconds": 10800, "gold_earned": 24300 }
  ],
  "transitions": 3
}
```

**Environment variables:**

| Variable | Required | Default | Purpose |
|----------|----------|---------|---------|
| `WOWBOT_WOW_PATH` | No | auto | Path to WoW installation |
| `WOWBOT_LOG_DIR` | No | `~/.wowbot/logs` | Signal log and session logs |
| `WOWBOT_SIGNAL_STALENESS_SECONDS` | No | `600` | Max age of signals before mode hold |
| `WOWBOT_MODE_TRANSITION_TIMEOUT_SECONDS` | No | `60` | Max wait for clean mode stop |

## Constraints

- Depends on v0.6.0 (Sentinel) for session-length randomization; Clockwork
  inherits `--session-max` and `--session-min` behavior for each sub-mode run
- All sub-modes (farm, flip, gather, etc.) must be runnable as child processes
  with a clean-stop signal; Clockwork does not embed any mode's logic
- Signal collection uses the AH scanner from v0.1.1; no new screen-reading
  surfaces are introduced
- The campaign config mode routing table is static; Clockwork does not learn
  or update thresholds automatically in this version

## Key Unknowns

1. **[P1]** Do AH activity signals — listing count, price compression, transaction
   velocity — provide sufficient resolution to infer server population state
   across the full range of operating conditions (weekday/weekend, patch day,
   raid reset), or are the signals too noisy to distinguish "low population"
   from "low AH participation despite high population" (e.g., when players are
   in a new raid and ignoring the AH)? The answer determines whether the signal
   model produces reliable mode routing or requires operator-defined time windows
   as the primary signal.

2. **[P2]** Is a static threshold-based routing table sufficient for practical
   operation — where the operator configures explicit signal → mode rules — or
   do the thresholds require frequent tuning as the meta shifts between patches,
   making a learned model a practical necessity from the start? The answer
   determines whether Clockwork ships as a rules engine or must include a
   calibration and adaptation mechanism.

---

## Non-Goals

- Learned or adaptive routing (static thresholds only in this version)
- Cross-realm signal collection
- Economic forecasting (that is Patch Oracle's domain)
- Parallel mode execution (one mode active at a time)
- Windows support
