# breadmin-composer v0.3.0 — Rolling Loop

**Theme:** The machine runs itself.

## Overview

Replaces the batch-oriented implementation stage with a self-sustaining dispatch loop.
Today the orchestrator claims a fixed set of issues, waits for the entire batch to
finish, then merges. If one agent hangs or the orchestrator session dies, work stalls
until a human intervenes. Rolling Loop introduces a **rolling dispatch** that refills
concurrency slots as agents complete, an **agent watchdog** that terminates and
re-queues hung agents automatically, and **per-issue failure isolation** so a
repeatedly-failing issue does not block the rest of the queue. Also adds
`composer run --qa` and `composer run --release` so the final two pipeline stages
can execute without manual steps.

---

## Success Criteria

- [ ] `composer run --impl --milestone <m>` fills all concurrency slots at startup
  and refills them within 30 seconds of any agent completing, without human intervention
- [ ] An agent exceeding `COMPOSER_AGENT_TIMEOUT_MINUTES` is automatically terminated;
  its issue is unclaimed and re-queued; its concurrency slot is refilled within one
  watchdog cycle (default: 60 s)
- [ ] An orchestrator process killed mid-pipeline restarts and resumes: issues with
  open PRs are not re-dispatched; issues with `in-progress` and no PR are reclaimed
  and re-queued
- [ ] The orchestrator emits a heartbeat log event every 60 seconds recording: active
  agent count, queue depth, and total cost to date
- [ ] `composer status` prints a live table: active agents (issue #, branch, elapsed
  time), queue depth, completed this session, total cost
- [ ] An agent that fails 3 times on the same issue posts a summary comment on the
  issue, removes `in-progress`, and does NOT block other issues from being dispatched
- [ ] `composer run --qa` runs the full test suite and exercises each spec acceptance
  criterion; files `stage/impl` + `bug` + `P0`/`P1` issues for failures; loops with
  implementation until qa runs clean
- [ ] `composer run --release` creates the git tag, publishes the GitHub release, and
  files the `init: <next version>` pipeline issue
- [ ] `make test` passes clean

## Scope

**Included:**

- **Rolling dispatch loop**: continuously refills concurrency slots as agents complete;
  does not wait for the full batch before starting the next issue
- **Agent watchdog**: background thread that checks agent liveness on a configurable
  interval; terminates and re-queues timed-out agents; SIGTERM → SIGKILL sequence with
  a 5-second grace period
- **Orchestrator crash recovery**: on startup, reconcile `in-progress` issues against
  open PRs to distinguish in-flight from orphaned; reclaim orphaned issues automatically
- **Heartbeat logging**: periodic JSONL events to the conductor log recording orchestrator
  vitals (active agents, queue depth, cost)
- **`composer status`**: live plaintext snapshot of orchestrator state readable from a
  second terminal while the worker runs; reads a small state file written by the
  orchestrator — no new IPC mechanism
- **Per-issue failure isolation**: each issue tracks its own retry count; on exceeding
  `max_retries`, the agent posts a failure summary comment, removes `in-progress`, and
  does not block other issues from being dispatched
- **`composer run --qa`**: qa stage implementation — runs test suite, exercises spec
  acceptance criteria against the running system, files bug issues for failures, loops
  with implementation until clean
- **`composer run --release`**: release stage implementation — confirms zero open P0/P1
  issues, creates git tag, publishes GitHub release, files `init: <next version>` issue

## Failure Modes

**Agent watchdog:**
- Agent subprocess hangs with no output → watchdog sends SIGTERM after timeout; if still
  alive after 5 s, sends SIGKILL; issue is unclaimed and re-queued
- Agent subprocess exits non-zero but produced a PR → orchestrator monitors PR as normal;
  exit code does not determine PR fate
- Watchdog thread itself crashes → orchestrator logs error, attempts restart; if watchdog
  cannot restart, halts dispatch and sends Telegram alert

**Rolling dispatch:**
- GitHub rate limit hit during dispatch → orchestrator backs off exponentially; does not
  drop issues from queue; resumes when limit clears
- All concurrency slots time out simultaneously → orchestrator logs warning, continues
  polling; does not terminate
- Issue claimed by another process between list and claim → orchestrator detects conflict
  on label write, skips issue, moves to next

**Crash recovery:**
- `in-progress` + open PR → resume monitoring PR; do not re-dispatch
- `in-progress` + no PR → unclaim and re-queue
- `in-progress` + merged PR → close issue, free slot
- Checkpoint file missing → start fresh; log warning

**`composer status`:**
- State file absent (orchestrator not running) → print `no active orchestrator session`
  and exit 0
- State file stale (orchestrator crashed) → print last-known state with a staleness
  warning

## Interface Contracts

**`composer status` output:**

```
Active agents (2/5 slots):
  #42  feat/crash-recovery     3m 12s
  #51  feat/heartbeat          1m 04s

Queue: 3 issues pending
Completed this session: 7
Total cost: $1.24
```

**Heartbeat log event (conductor log JSONL):**

```json
{
  "event": "heartbeat",
  "run_id": "<uuid>",
  "ts": "<iso8601>",
  "active_agents": 2,
  "queue_depth": 3,
  "completed": 7,
  "cost_usd": 1.24
}
```

**Environment variables (new in v0.3.0):**

| Variable | Required | Default | Purpose |
|----------|----------|---------|---------|
| `COMPOSER_AGENT_TIMEOUT_MINUTES` | No | `60` | Max wall-clock time per agent before watchdog kills it |
| `COMPOSER_WATCHDOG_INTERVAL_SECONDS` | No | `60` | How often the watchdog checks agent liveness |

## Constraints

- Preserves all v0.2.x security layers: credential proxy, explicit subprocess env,
  `--allowedTools` policy, `CLAUDECODE` nesting guard
- Rolling dispatch respects existing rate-limit concurrency caps
  (Pro: 1–2, Max: 2–3, Max 20x: 3–5) and the pre-dispatch budget check
- Watchdog and dispatch loop do not require a persistent daemon; the orchestrator
  remains a foreground CLI command
- `composer status` works by reading a small state file written by the orchestrator —
  no new IPC mechanism, no sockets
- Checkpoint format remains backward-compatible with v0.2.x checkpoints or provides
  a one-time migration path
- No changes to the research, design, or scoping stages

## Key Unknowns

1. **[P1]** Is wall-clock timeout against the subprocess PID sufficient for agent
   liveness detection, or does the watchdog need to monitor stdout event frequency
   (absence of stream-json for N seconds) to distinguish a genuinely hung agent from
   one doing legitimate slow work? Can the orchestrator reliably kill a Claude Code
   subprocess on macOS without orphaned child processes or locked worktrees?
   The answer determines the watchdog mechanism and whether SIGTERM → SIGKILL
   sequencing is safe.

2. **[P1]** Is `gh issue list --label in-progress` combined with `gh pr list`
   sufficient to distinguish "completed with PR" from "mid-run no PR" after a crash,
   or does reliable recovery require a crash-safe local state file independent of
   GitHub (atomic-write JSON or equivalent)? The answer determines whether a local
   state store is required for correct recovery or whether GitHub itself is the
   source of truth.

---

## Non-Goals

- `composer run --spec-update` (v0.3.1)
- Campaign mode and multi-milestone sequencing (v0.4.0)
- A persistent background daemon (launchd, systemd)
- Changes to research, design, or scoping stages
- A web UI or TUI dashboard beyond `composer status` plaintext
- Multi-repo orchestration in a single session (already in v0.2.x)
- Agent memory or context persistence across retries
