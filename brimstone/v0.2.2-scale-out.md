## Overview

Extends brimstone to orchestrate work across the full platform — multiple repos, multiple milestones, wave-ordered execution — from a single session. Today brimstone operates on one repo at a time; each repo must be driven separately, and cross-repo milestone dependencies are tracked only in human-maintained docs. This milestone adds a platform repo registry (a config-backed catalog of all known repos with their local paths and spec directories), a `brimstone run --campaign` entry point that drives the full platform build from a campaign file routing work to the correct repos per wave, spec write routing so specs land in each repo's own directory rather than a shared external location, and bead-level inter-repo milestone dependency support so the BeadStore can block a WorkBead until a milestone releases in a different repo. Cross-model consensus for research quality validation is also added here, compounding with multi-repo scale. Depends on v0.2.1: multi-model backends must be in place before consensus runs.

**Prerequisite:** v0.2.1-multi-model must complete before this milestone — the backend abstraction is required for cross-model consensus.

## Success Criteria

- [ ] `brimstone run --campaign platform.md --wave 0` dispatches agents to all Wave 0 repos and milestones in parallel from a single session; runner log shows per-repo dispatch
- [ ] `brimstone run --campaign platform.md --wave 1` blocks until all Wave 0 milestones have published GitHub releases, then dispatches Wave 1 repos automatically
- [ ] Module isolation is enforced per repo: the same module label is not claimed in two repos simultaneously
- [ ] One repo's CI failure does not halt the orchestrator; the failing repo is reported in the session summary and all other repos continue
- [ ] `brimstone repo add bread-wood/moot --local ~/dev/moot --specs specs/moot` registers a repo in the registry; `brimstone repo list` shows it
- [ ] A research doc produced by running the same issue through two backends is cross-validated; claims where the backends disagree are marked `[DISPUTED]` in the output doc
- [ ] A `[DISPUTED]` claim triggers a flag in the runner log identifying the claim and the nature of the disagreement
- [ ] A WorkBead blocked on a cross-repo milestone shows `blocked: waiting for <repo>/<milestone> release` in `brimstone status`; it is dispatched within one watchdog cycle after the release appears on GitHub
- [ ] Spec files written by brimstone for a registered repo land in that repo's configured spec directory, not a hardcoded path
- [ ] `brimstone repo add bread-wood/moot` adds yeast-bot as a collaborator (write) to that repo automatically; collaborator invite is visible on GitHub
- [ ] `brimstone repo remove bread-wood/moot` removes the repo from the registry; subsequent campaign runs skip it with a warning
- [ ] `bread-wood/brimstone-beads` and `bread-wood/brimstone-specs` repos exist on GitHub and are registered in the platform repo registry
- [ ] All existing specs migrated from `~/Documents/specs/` into `brimstone-specs` with directory structure preserved; spec write routing resolves to `brimstone-specs` for all registered repos
- [ ] Concurrent bead writes from two sessions do not corrupt `brimstone-beads`; the second write queues behind the first and completes cleanly
- [ ] `make test` passes clean

## Scope

**Included:**

- **Platform repo registry**: a config file (`brimstone.toml` or equivalent) listing all known repos with: GitHub `owner/repo`, local clone path, and spec directory path; `brimstone repo add/remove/list` commands manage entries; `brimstone repo add` automatically adds yeast-bot as a collaborator (write) to the target repo
- **`brimstone run --campaign <path> [--wave N]`**: reads a campaign file; for each wave, identifies which repos and milestones are active; dispatches agents to all repos in the wave in parallel; blocks wave N+1 until all wave N milestones have GitHub releases
- **Spec write routing**: when brimstone writes or modifies spec files, it resolves the destination using the repo registry's `spec_dir` for that repo; no hardcoded spec paths in orchestrator code
- **Bead inter-repo milestone deps**: WorkBead schema extended with an optional `blocked_by` field recording `{repo, milestone}` pairs; BeadStore enforces the block before dispatch; block clears when the target milestone's GitHub release is detected
- **`brimstone-beads` repo creation**: create `bread-wood/brimstone-beads`; migrate BeadStore writes to commit bead state to this repo; introduce a dedicated write queue for `brimstone-beads` commits (separate from MergeQueue, which remains scoped to product repo PR merges)
- **`brimstone-specs` repo creation + migration**: create `bread-wood/brimstone-specs`; migrate all existing specs from local `~/Documents/specs/` into the repo with directory structure preserved; spec write routing resolves to `brimstone-specs` for all registered repos
- **Multi-repo agent dispatch**: single brimstone session dispatches agents to issues across all registered repos; module isolation enforced per repo; per-repo credential and branch scoping
- **Cross-model consensus**: the same research issue is run through two configured backends; a diff step identifies contradictory claims; disagreements are annotated `[DISPUTED]` in the merged output doc for human review

**Explicitly excluded:**

- Rolling dispatch or agent watchdog (v0.3.0)
- `brimstone status` live campaign dashboard (v0.3.0 adds per-session status; v0.4.0 adds campaign view)
- Campaign file authoring or spec generation (v0.5.0-spec-forge)
- Automatic resolution of `[DISPUTED]` claims without human review

## Failure Modes

**Platform repo registry:**
- Repo in campaign file not in registry → log warning; skip that repo for dispatch; continue with registered repos; list unregistered repos in session summary
- Registered repo's local clone path does not exist → log error for that repo; skip dispatch; continue with other repos
- Registry file not writable → `brimstone repo add` fails with `error: registry not writable: <path>`; read-only operations still work

**Multi-repo orchestration:**
- One repo's CI fails → issues remain open for that repo; orchestrator continues with other repos; failing repo reported in session summary
- Module conflict across repos (same module claimed in two repos simultaneously) → orchestrator serialises; does not dispatch second until first completes
- A repo becomes inaccessible mid-session → unclaim that repo's in-progress issues, log error, continue with remaining repos

**Bead inter-repo milestone deps:**
- Target milestone release not found after `BRIMSTONE_RELEASE_WAIT_MINUTES` → emit warning in session log; continue polling; do not abandon the blocked bead
- Target repo not in registry when block is evaluated → log error; treat block as unresolvable; surface in session summary for human action

**Cross-model consensus:**
- Second backend unavailable → skip consensus for this issue; produce single-backend doc without `[DISPUTED]`; log warning
- Both backends agree → doc produced normally with no `[DISPUTED]` markers; expected success case
- Consensus diff step fails to parse one backend's output → skip consensus, fall back to primary backend's doc, log error with both raw outputs saved

**Spec write routing:**
- Repo's `spec_dir` does not exist → create it; log that it was created
- `spec_dir` not writable → fail the spec write with a clear error; do not write to a fallback path silently

## Interface Contracts

**Repo registry schema (`brimstone.toml` or equivalent):**

```toml
[[repos]]
github = "bread-wood/brimstone"
local  = "~/dev/brimstone"
specs  = "specs/brimstone"

[[repos]]
github = "bread-wood/moot"
local  = "~/dev/moot"
specs  = "specs/moot"

[[repos]]
github = "bread-wood/breadwinner"
local  = "~/dev/breadwinner"
specs  = "specs/breadwinner"
```

**`brimstone repo` CLI:**

```
brimstone repo add <owner/repo> --local <path> --specs <path>
brimstone repo remove <owner/repo>
brimstone repo list
```

**WorkBead schema extension:**

```json
{
  "issue": 42,
  "repo": "bread-wood/moot",
  "branch": "42-council-spine",
  "blocked_by": [
    { "repo": "bread-wood/brimstone", "milestone": "v0.2.1" }
  ]
}
```

**Campaign file format** (markdown; parsed by `brimstone run --campaign`):

```markdown
## Wave 0
- bread-wood/brimstone v0.2.0
- bread-wood/moot v0.4.0

## Wave 1
- bread-wood/brimstone v0.2.1
- bread-wood/moot v0.5.0
```

**Environment variables (new in v0.2.2):**

| Variable | Required | Default | Purpose |
|----------|----------|---------|---------|
| `BRIMSTONE_REGISTRY` | No | `~/.config/brimstone/repos.toml` | Path to repo registry file |
| `BRIMSTONE_CONSENSUS_MODEL` | No | — | Secondary backend for cross-model consensus; disabled if absent |
| `BRIMSTONE_RELEASE_WAIT_MINUTES` | No | `60` | How long to poll for a cross-repo milestone release before warning |

## Constraints

- Depends on v0.2.1: multi-model backends must be available before consensus runs
- Depends on v0.2.0: repo-scoped checkpoint schema must be in place; multi-repo sessions must not require a checkpoint migration
- All spec paths in orchestrator code must resolve via the repo registry; no hardcoded filesystem paths to spec directories
- Cross-model consensus is opt-in via `BRIMSTONE_CONSENSUS_MODEL`; applies only to research issues; must not change the research doc format visible to downstream stages
- Multi-repo orchestration must degrade to single-repo behaviour when no campaign file is provided; no behaviour change for existing single-repo users
- Bead `blocked_by` field is optional; existing BeadStore records without it must remain valid without migration
- Cross-repo bead dependencies are milestone-level only (`{repo, milestone}` blocking); sub-milestone bead-to-bead deps across repos are not supported in this version
- MergeQueue remains scoped to product repo PR merges only; `brimstone-beads` write serialisation uses a separate, dedicated write queue primitive — MergeQueue is not extended
- Campaign file wave ordering is positional (Wave 0 before Wave 1); no graph inference — ordering is explicit

## Key Unknowns

1. **[P1]** What diff strategy for comparing two research docs is accurate enough to surface genuine contradictions without false-positiving on paraphrased agreement? The answer must hold as research doc volume grows with multi-repo use.

2. **[P1]** What is the right config file format and location for the platform repo registry — a TOML file in `~/.config/brimstone/`, a `brimstone.toml` at repo root, or a section in `pyproject.toml`? The answer determines how the registry is discovered and whether it can be committed to a repo for team use or must remain machine-local.

3. **[P2]** What multi-repo scheduling policy prevents module conflicts and fairly distributes dispatch capacity across repos when one repo has significantly more open issues than another? The answer determines the scheduler design and must extend cleanly to more repos.

## Non-Goals

- A persistent background daemon
- Rolling dispatch or agent watchdog (v0.3.0)
- `brimstone status` live campaign dashboard (v0.3.0 / v0.4.0)
- Campaign file authoring or structured spec interview (v0.5.0)
- Automatic resolution of `[DISPUTED]` claims without human review
- Non-linear milestone dependency graphs in campaign files (explicit wave ordering only)
- Cross-repo bead visibility: a bead in repo A observing the live state of a bead in repo B (deferred; future milestone)
