# breadmin-composer v0.2.1 — Multi-Model

**Theme:** Run agents on any configured backend, routed by task complexity.

## Overview

Extends the pipeline to support Gemini and GPT-4.1 as drop-in agent backends alongside
Claude. Adds a per-task complexity estimator that selects the appropriate backend based
on inferred task stakes and difficulty, so high-risk tasks get a capable model and
routine tasks get a cheaper one automatically. Also adds Telegram notifications for
stage transitions and a human approval gate before release. Depends on v0.2.0: the
credential proxy must be active before non-Claude backends are dispatched, and the
crash recovery checkpoint must be in place before parallel multi-backend sessions run.

## Success Criteria

- [ ] `composer run --research --milestone <m>` completes using the Gemini backend when
  `COMPOSER_MODEL=gemini` is set; runner log records `backend=gemini`
- [ ] `composer run --research --milestone <m>` completes using the GPT-4.1 backend when
  `COMPOSER_MODEL=gpt-4.1` is set; runner log records `backend=gpt-4.1`
- [ ] With no `COMPOSER_MODEL` set, a security-domain research issue dispatches to a
  high-tier backend and a routine data-format research issue dispatches to a cheaper
  backend; runner log records different backends for each
- [ ] When the estimator returns confidence below threshold, the orchestrator upgrades
  to the next tier; the upgrade event and its reasoning appear in the runner log
- [ ] When `COMPOSER_MODEL` is set, no estimator call is made; all tasks use the
  specified backend regardless of estimated complexity
- [ ] Estimator failure (timeout, malformed response) falls back to the session default
  backend; dispatch proceeds and a warning appears in the runner log
- [ ] The orchestrator sends a Telegram message at each stage transition
- [ ] The orchestrator sends a Telegram approval request before release and does not
  proceed until `/approve` is received
- [ ] `make test` passes clean

## Scope

**Included:**

- **Multi-model backends**: Gemini and GPT-4.1 support routed through `breadmin-llm`'s
  `ProviderRegistry`; orchestrator calls (complexity estimator, triage, design) use
  `registry.complete()`; backend selected by `COMPOSER_MODEL` or by the complexity estimator
- **Task complexity estimator**: pre-dispatch step that reads each issue and produces a
  tier and confidence score; tier determines the backend; confidence below threshold
  rounds up to the next tier; `COMPOSER_MODEL` overrides when set; estimator always uses
  the cheapest configured backend
- **Telegram integration**: stage-transition notifications and a human approval gate
  before the release stage

## Failure Modes

**Multi-model backends:**
- Provider API key missing → `brimstone health` reports fatal for that provider;
  `brimstone run` exits with a clear error before dispatch
- Provider returns error response → retry up to `max_retries` via breadmin-llm retry
  logic; on exhaustion, fall back to session default backend and log warning
- Provider times out → breadmin-llm timeout handling applies; orchestrator logs and
  continues with default backend

**Task complexity estimator:**
- Estimator call times out → use session default backend, log warning, dispatch proceeds
- Estimator returns malformed or missing tier → use session default backend, log warning
- Estimator confidence below threshold → upgrade to next tier; log upgrade with reasoning
- No backend available for the selected tier → fall through to the highest available tier

**Telegram:**
- Telegram API unreachable → log warning, continue without notification; pipeline does
  not block on notification delivery
- Approval gate receives no response within 30 minutes → send reminder; after 24 hours
  without approval, send final warning and halt; do not auto-release

## Interface Contracts

**Environment variables:**

| Variable | Required | Purpose |
|----------|----------|---------|
| `COMPOSER_MODEL` | No | Backend override: `claude` (default), `gemini`, `gpt-4.1`; disables estimator when set |
| `COMPOSER_ESTIMATOR_THRESHOLD` | No | Confidence below which estimator upgrades to next tier (default: `0.6`) |
| `COMPOSER_TELEGRAM_TOKEN` | No | Telegram bot token; Telegram features disabled if absent |
| `COMPOSER_TELEGRAM_CHAT_ID` | No | Telegram chat ID for notifications |

**Telegram message format:**

```
[composer] <stage> complete: <version>
Repo: <owner>/<repo>
Issues closed: <N>
Duration: <HH:MM>
Next: <next stage>
```

Approval gate message:
```
[composer] Ready to release <version>. Reply /approve to proceed or /hold to pause.
```

## Constraints

- Depends on v0.2.0: credential proxy must be running before any non-Claude backend
  is dispatched; crash recovery must be in place before parallel backend sessions run
- Orchestrator LLM calls (complexity estimator, triage, design generation) route through
  `breadmin-llm`'s `ProviderRegistry`; adding a new backend requires only registering a
  new provider in breadmin-llm — no changes to the orchestrator or estimator
- Agent dispatch (`claude -p` subprocess) remains subprocess-based; breadmin-llm is not
  used for agent invocations, only for orchestrator-level completions
- Costs from `claude -p` agent invocations are reported into breadmin-llm after each
  invocation by parsing stream-json output; orchestrator-level call costs flow through
  breadmin-llm automatically
- The estimator must complete in under 10 seconds per issue and must not block the
  dispatch queue
- The estimator must use the cheapest configured backend; it must not use a high-tier
  model for classification
- A single estimator prompt must generalise across project domains without per-project
  configuration
- Telegram integration must degrade gracefully: no Telegram configured means the
  pipeline runs without notifications
- Multi-model support must not change the research doc format, triage rubric, or any
  v0.1.x pipeline stage behaviour

## Key Unknowns

1. ~~**[P1]** What backend interface abstraction allows the orchestrator and complexity
   estimator to remain unchanged as new backends are added beyond Gemini and GPT-4.1
   in v0.3.x?~~ **[RESOLVED]** Orchestrator completions route through `breadmin-llm`'s
   `ProviderRegistry`, which already provides a provider-agnostic interface with
   Anthropic, OpenAI, Gemini, and DeepSeek support. Adding a new backend requires only
   registering a new `breadmin-llm` provider — no orchestrator changes needed.

2. **[P1]** What signals in an issue body reliably distinguish high-stakes from routine
   tasks across different project domains — not just for brimstone's own issues but for
   downstream projects like wow-bot, which have different risk profiles? The answer
   determines whether the estimator prompt is universal or requires per-project tuning,
   and must hold as the pipeline is applied to projects with no shared vocabulary with
   the training examples.

3. **[P2]** What Gemini CLI tool restrictions are required for research and impl agents
   to function correctly in headless mode? The answer determines the dispatch command
   for Gemini-backend agents and whether tool configuration must be per-stage.

## Non-Goals

- Cross-model consensus (v0.2.2)
- Multi-repo orchestration (v0.2.2)
- Per-project estimator configuration or routing tables
- Learning or updating the estimator from past dispatch outcomes
- A persistent background daemon
