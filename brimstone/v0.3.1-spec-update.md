# breadmin-composer v0.3.1 — Spec Update

**Theme:** Research answers flow forward.

## Overview

Adds `composer run --spec-update` as a post-release pipeline stage. After a version
releases, research findings and implementation discoveries should propagate forward
to future specs — resolving Key Unknowns that are now answered, tightening Constraints
based on what was learned, and flagging any assumption that was invalidated. Without
this stage, future specs accumulate stale unknowns and uncorrected assumptions as
the project advances. With it, each release automatically elaborates the specs for
every version that follows.

---

## Success Criteria

- [ ] `composer run --spec-update` after a release reads all merged research docs
  and closed PRs from the completed version; for each future-version spec, marks
  each now-answered Key Unknown with its answer and source; commits a summary
  of changes; exits 0
- [ ] A Key Unknown resolved by a merged research doc appears in the spec's
  `## Elaboration Log` section with the answer, doc reference, and resolution date;
  the original Key Unknown text is preserved with a `[RESOLVED]` prefix
- [ ] When a merged finding contradicts a future spec's core premise,
  `composer run --spec-update` writes an `[INVALIDATED]` marker to that spec,
  sends a Telegram alert identifying the spec and the conflicting finding, and
  exits 1
- [ ] `composer run --spec-update` makes no changes to the spec for the version
  that just released or any prior version; only future-version specs are touched
- [ ] `make test` passes clean

---

## Scope

**Included:**

- **`composer run --spec-update`**: post-release elaboration pass
  - Reads all merged research docs (`docs/research/*.md`) and closed PRs from the
    completed version
  - For each future-version spec in the campaign: identifies Key Unknowns that
    are now answerable from the research, tightens Constraints where findings
    constrain design choices, and flags any assumption that the research invalidated
  - Does not create new spec files; only modifies existing ones
  - Appends a `## Elaboration Log` section to each updated spec recording what
    changed and why; if the section already exists, appends to it
  - Commits all changes to the default branch with a structured commit message
    referencing the completed version: `spec-update: elaborate future specs from
    <version> research`
  - Sends a Telegram alert and exits 1 if any finding invalidates a future spec's
    core premise

---

## Failure Modes

**No research material:**
- No merged research docs found for the completed version → exit 0 with
  `nothing to elaborate: no merged research docs found for <version>`
- Research docs found but all Key Unknowns in future specs are already resolved →
  exit 0 with `all Key Unknowns already resolved; no changes made`

**Context limits:**
- Full research doc text exceeds model context → fall back to summaries; log warning
  with token count; note in Elaboration Log that summaries were used

**Invalidation:**
- Finding contradicts a future spec's core premise → write `[INVALIDATED]` marker to
  that spec, send Telegram alert listing the spec and conflicting finding, exit 1;
  other specs that were updated before the invalidation are committed before exit

**Spec file errors:**
- Spec file not writable → log error, skip that spec, continue with others; exit
  non-zero after all specs have been attempted

## Interface Contracts

**`composer run --spec-update` output (stdout):**

```
spec-update: v0.2.2 research → future specs

  v0.3.0-rolling-loop.md
    [RESOLVED] Key Unknown 2: crash recovery source of truth
    → GitHub Issues + PRs are sufficient; no local state store needed
    → Source: docs/research/42-crash-recovery-prototype.md

  v0.3.1-spec-update.md
    [RESOLVED] Key Unknown 1: minimum context for elaboration
    → Full doc text fits within context for up to 20 docs; summaries needed beyond that
    → Source: docs/research/43-spec-update-context-scaling.md

2 specs updated, 0 invalidated.
```

**Elaboration Log section appended to each updated spec:**

```markdown
## Elaboration Log

### v0.2.2 (2026-04-01)

**Key Unknown 2 resolved:** GitHub Issues + PRs are sufficient as the crash recovery
source of truth; no crash-safe local state file is needed.
Source: docs/research/42-crash-recovery-prototype.md
```

## Constraints

- Depends on v0.3.0 for `composer run --release`; spec-update is designed to run
  immediately after a release, so the release stage must exist before this stage
  is meaningful
- Must not create new spec files; only modifies existing ones
- Must not alter Success Criteria or Scope sections of any spec; only Key Unknowns,
  Constraints, and the Elaboration Log are modified
- Changes are committed directly to the default branch with a structured commit
  message; no PR required for spec elaboration commits
- Single-version propagation only: spec-update reads the most recently released
  version's research and propagates to all future specs; multi-milestone batch
  propagation is v0.4.0

---

## Key Unknowns

1. **[P1]** What is the minimum research context that produces reliable Key Unknown
   resolutions — full text of merged research docs, summaries only, or structured
   claim extracts — and does this scale as the number of research docs grows across
   many releases without exceeding model context limits? The answer determines whether
   a summarisation step is required before the elaboration pass and whether the
   spec-update agent degrades gracefully as the corpus grows.

---

## Non-Goals

- Multi-milestone spec-update propagation (v0.4.0 extends this to all downstream
  specs in the campaign simultaneously)
- Automatic resolution of `[INVALIDATED]` findings without human review
- Creating new spec files from research findings
- Modifying research docs or design docs (only future version specs are touched)
