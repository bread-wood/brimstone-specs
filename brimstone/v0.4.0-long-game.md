# breadmin-composer v0.4.0 — Long Game

**Theme:** Campaign mode. Plan many versions at once, execute them in sequence.

## Overview

Introduces the `plan` stage as a first-class pipeline step, splitting it out of
`init`. Today `init` both creates the GitHub milestone and seeds research issues
from the spec's Key Unknowns — two concerns bundled together. Long Game separates
them: `init` creates a milestone and nothing else; `plan` reads one or more spec
files, seeds research issues for each, and records the ordering relationship
between milestones so the orchestrator can gate implementation sequencing and
propagate spec-update findings across all downstream versions. A project with
ten planned versions can be fully seeded in one command and then left to execute
over time.

## Success Criteria

- [ ] `composer init --milestone v0.1.x` creates a GitHub milestone named `v0.1.x`
  and exits 0; no issues are created
- [ ] `composer init --milestone v0.1.x` when the milestone already exists exits 0
  with a notice and makes no changes
- [ ] `composer plan v0.1.x.md v0.2.x.md v0.3.x.md` creates `stage/research` seed
  issues for all three milestones (finding or creating each milestone as needed)
  and exits 0 with a summary: milestones created, issues filed per milestone
- [ ] `composer plan v0.1.x.md v0.2.x.md v0.3.x.md` records the ordering
  `v0.1.x → v0.2.x → v0.3.x` in the campaign checkpoint; subsequent
  `composer status` shows the ordering
- [ ] Research agents for all planned milestones can be dispatched concurrently
  without waiting for any other milestone to advance
- [ ] `composer run --impl --milestone v0.2.x` does not dispatch any impl agents
  until `v0.1.x` has a published GitHub release; prints
  `impl blocked: waiting for v0.1.x release` in `composer status`
- [ ] `composer run --impl --milestone v0.2.x` begins dispatching impl agents
  within one watchdog cycle after the `v0.1.x` release appears on GitHub
- [ ] `composer run --spec-update` after a release propagates findings to ALL
  downstream specs in the campaign, not only the immediately following one
- [ ] `composer status` prints a campaign-level view: each milestone, its current
  stage, agent counts, queue depth, and any blocking condition
- [ ] `composer plan v0.1.x.md` where `v0.1.x.md` does not exist prints
  `error: spec file not found: v0.1.x.md` to stderr and exits 1
- [ ] `composer plan v0.2.x.md v0.1.x.md` (later version listed first) exits 1
  with `error: v0.2.x listed before v0.1.x; ordering must be oldest-first` —
  ordering validation based on semver of milestone names extracted from spec filenames
- [ ] `make test` passes clean

## Scope

**Included:**

- **`composer init` reduced**: creates a GitHub milestone only; no issue seeding
- **`composer plan <spec> [<spec>...]`**: new command; for each spec in order:
  1. Find or create the GitHub milestone named in the spec header
  2. Seed `stage/research` issues from the spec's Key Unknowns
  3. Apply existing research triage rubric to seed issues
  4. Append the milestone to the campaign ordering in the checkpoint
- **Campaign checkpoint**: local state file recording the ordered list of milestones
  in the current campaign; used by impl sequencing gate and spec-update
- **Impl sequencing gate**: before dispatching any `stage/impl` issue for milestone N,
  verify a GitHub release exists for milestone N-1; block and poll if not
- **Multi-milestone `composer status`**: campaign-level view showing all milestones,
  their current pipeline stage, active agent counts, queue depth, and blocking conditions
- **Multi-milestone `spec-update`**: after a release, propagates findings to all
  downstream milestone specs in the campaign, not just the next one
- **Idempotent `plan`**: re-running `composer plan` for an already-planned milestone
  skips issue creation and exits 0 with a notice; does not duplicate issues
- **Glob support**: `composer plan specs/calculator/*.md` expands the glob before
  processing; specs are sorted by filename before ordering validation

## Failure Modes

**`composer plan`:**
- Spec file not found → `error: spec file not found: <path>` to stderr, exit 1
- Spec file has no Key Unknowns section → milestone created, zero research issues
  filed, warning emitted: `no Key Unknowns found in <spec>; no issues seeded`
- GitHub milestone creation fails → report which milestone failed; do not roll back
  milestones already created; exit 1
- Specs listed out of semver order → `error: <later> listed before <earlier>; ordering must be oldest-first` to stderr, exit 1
- Campaign checkpoint not writable → `error: cannot write campaign checkpoint: <reason>` to stderr, exit 1

**Impl sequencing gate:**
- No release found after `COMPOSER_RELEASE_WAIT_MINUTES` → emit Telegram alert;
  continue polling; do not halt dispatch for other milestones
- Prerequisite milestone has open P0 issues → block impl of dependent milestone
  and surface in `composer status` as `impl blocked: v0.1.x has open P0 issues`

**`spec-update` multi-milestone:**
- A finding invalidates a spec two or more versions downstream → each invalidated
  spec gets an `[INVALIDATED]` marker; a single Telegram alert lists all affected specs

**Campaign checkpoint:**
- Checkpoint missing on startup → orchestrator has no campaign context; warn and
  operate in single-milestone mode
- Checkpoint corrupt → `error: campaign checkpoint corrupt; re-run composer plan` to
  stderr; halt campaign-aware features; single-milestone operations still work

## Interface Contracts

**CLI:**

```
composer init --milestone <name>
composer plan <spec-file> [<spec-file> ...] [--dry-run]
```

| Flag | Purpose |
|------|---------|
| `--milestone <name>` | (`init` only) Name of the GitHub milestone to create |
| `--dry-run` | (`plan` only) Print what would be created without writing to GitHub |

**`composer status` campaign view:**

```
Campaign: v0.1.x → v0.2.x → v0.3.x

v0.1.x  [implementation]  active 2/5  queue 3  cost $1.24
v0.2.x  [research]        active 3/5  queue 2  cost $0.41
v0.3.x  [research]        active 1/5  queue 4  cost $0.18
v0.4.x  [planned]         blocked: waiting for v0.3.x release
```

**Campaign checkpoint schema:**

```json
{
  "version": 1,
  "milestones": ["v0.1.x", "v0.2.x", "v0.3.x", "v0.4.x"],
  "planned_at": "<iso8601>"
}
```

**Environment variables (new in v0.4.x):**

| Variable | Required | Default | Purpose |
|----------|----------|---------|---------|
| `COMPOSER_RELEASE_WAIT_MINUTES` | No | `60` | How long to wait for a prerequisite release before alerting |

## Constraints

- Preserves all v0.3.x constraints: rolling dispatch, watchdog, crash recovery,
  credential proxy, rate-limit caps
- Milestone ordering is determined by the positional order of spec arguments;
  ordering is validated against semver extracted from spec filenames and rejected
  if out of order
- Research stages for all milestones run concurrently with no sequencing gate;
  only `stage/impl` is gated
- The campaign checkpoint is append-only during a session; milestones are not
  removed from the campaign by orchestrator actions (only by human intervention)
- `composer plan` with a glob expands and sorts by filename before any other
  processing; the sort order must match the ordering validation rule
- `spec-update` propagation to downstream specs must not alter Success Criteria
  or Scope sections of any spec (constraint inherited from v0.3.x)

## Key Unknowns

1. **[P1]** Is the GitHub release tag sufficient as the "milestone complete" signal
   for the impl sequencing gate, or does the gate need to inspect the milestone's
   open issue count (ensuring no P0 issues remain) to avoid gating on a partial
   release? The answer determines whether the sequencing gate is stateless (GitHub
   API only) or requires local state to track release completeness.

2. **[P1]** When `spec-update` propagates findings to multiple downstream specs
   simultaneously, does each spec update need to be committed as a separate
   atomic commit (for auditability) or can all downstream updates be batched into
   a single commit? The answer determines the commit strategy for multi-milestone
   spec-update and whether the `Elaboration Log` in each spec can cross-reference
   updates made in the same batch.

---

## Non-Goals

- Campaign YAML file with explicit dependency graph (non-linear milestone ordering)
- Parallel implementation across milestones (impl sequences; research parallelizes)
- Automatic ordering inference from version numbers without filename-based validation
- Any changes to the research, design, scoping, qa, or release stages
