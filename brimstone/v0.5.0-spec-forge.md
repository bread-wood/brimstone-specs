# brimstone v0.5.0 — Spec Forge

> **Who reads this file and why**
>
> This spec is the authoritative statement of *what* this milestone delivers and *why*.
> It is consumed by:
> - **research agents** — investigate Key Unknowns; findings constrain design choices
> - **design agents** — use Scope, Constraints, and Failure Modes to bound design docs
> - **impl agents** — use Success Criteria as acceptance criteria for each issue
> - **qa** — runs every Success Criterion and every Failure Mode as a test case
> - **human reviewers** — use Non-Goals to prevent scope creep
>
> Write this file before seeding issues. Keep it product-focused (what and why),
> not technical (not how). The how belongs in design docs produced during the design stage.
> Do not prescribe issue decomposition — that is the planning agent's job.

---

## Overview

Brimstone v0.5.0 adds a `spec-forge` mode: an interactive, human-in-the-loop stage that takes a feature description and produces complete milestone specs following TEMPLATE.md. It scans all repos' CLAUDE.md files and existing specs, determines which repos are affected (or whether a new one is needed), conducts a structured interview with the user to resolve architectural decisions, drafts spec files iteratively, flags potential violations against platform architecture principles, and updates the platform campaign file. This automates the kind of collaborative planning work done manually today — exploring repos, asking clarifying questions, drafting specs, flagging architecture violations, and sequencing the result into the campaign.

**Prerequisite:** v0.4.0-long-game must complete before this milestone — campaign mode must exist before spec-forge can update campaigns. v0.2.2-scale-out is also required: spec write routing and the platform repo registry must be in place before spec-forge can resolve where to write specs for each repo.

## Success Criteria

- [ ] `brimstone spec "description"` opens an interactive session and produces at least one spec file in the correct location following TEMPLATE.md
- [ ] `brimstone spec --file feature.md` runs the same flow non-interactively from a file
- [ ] Produced spec files pass a structural validation check (all required sections present, Key Unknowns have priority labels, Non-Goals are explicit)
- [ ] After producing specs, spec-forge updates the platform CAMPAIGN.md with the new milestones at the correct wave position
- [ ] Architecture violations (as defined in ARCHITECTURE_REVIEW.md patterns) are flagged in a notes section appended to the spec, not silently ignored
- [ ] After producing specs and updating CAMPAIGN.md, spec-forge produces or updates a visual dependency graph reflecting the full platform milestone ordering and cross-repo blocking relationships
- [ ] `make test && make lint` pass on mainline

## Scope

**Included:**

- `brimstone spec [description | --file path]` command
- Platform repo scan: reads CLAUDE.md + existing specs from all known repos
- Structured interview: configurable question set covering repo home, delivery interface, cross-repo coordination, and P1 Key Unknowns
- Spec draft loop: draft → show to user → revise → confirm
- TEMPLATE.md validation: checks produced spec has all required sections before writing
- Architecture guard: checks new scope against known violation patterns (dual schedulers, direct cross-repo imports, parallel cost tracking)
- Campaign update: adds new milestones to CAMPAIGN.md at the correct wave position per dependency graph rules
- Dependency graph output: after updating CAMPAIGN.md, produces or updates a visual graph of the full platform milestone ordering and cross-repo blocking relationships; saved alongside CAMPAIGN.md in `brimstone-specs` (format: P1 Key Unknown)

**Explicitly excluded:**

- Fully autonomous spec design with no user interview (always human-in-the-loop)
- GitHub issue seeding (that's `brimstone init` scope)

---

## Failure Modes

**Repo scan:**
- Repo not cloned or unavailable → log warning; proceed with repos that are available; note missing repos in spec preamble

**Structured interview:**
- User gives ambiguous answer to interview question → re-ask with more specific options; do not silently assume an answer

**Spec validation:**
- Produced spec fails structural validation → show validation errors; do not write the file; ask user to resolve before saving

**Campaign update:**
- Campaign update produces a cycle in the dependency graph → flag the cycle; do not write the campaign update; ask user to resolve the dependency

**Dependency graph:**
- Graph tool unavailable (draw.io MCP not connected, or Mermaid renderer absent) → write graph source as plain text; log warning; do not block the spec or campaign write

## Interface Contracts

**CLI:**

```
brimstone spec [description]
brimstone spec --file <path>
```

**Spec output location:**

```
<repo.spec_dir>/<version>-<slug>.md   (resolved via repo registry)
```

**Dependency graph output location:**

```
<brimstone-specs root>/DEPENDENCY_GRAPH.<ext>   (ext determined by format — P1 Key Unknown)
```

**Architecture violation note (appended to spec when violations found):**

```markdown
## Architecture Notes

> The following potential violations were detected by spec-forge. Review before impl begins.

- [violation description]
```

---

## Constraints

- spec-forge resolves the destination for all spec writes via the repo registry's `spec_dir` for the target repo (v0.2.2 spec write routing); no hardcoded filesystem paths
- TEMPLATE.md location is resolved via the registry or a configurable path; spec-forge must not hardcode a path to TEMPLATE.md
- Interview must include at minimum: repo home, delivery/user surface, cross-repo coordination points, and P1 Key Unknowns
- Spec files are never overwritten without explicit user confirmation
- spec-forge always requires human confirmation before writing any file

---

## Key Unknowns

1. **[P1]** What is the right intake format — free-form natural language, a structured prompt template, or a multi-turn interactive dialogue? The answer determines whether spec-forge can run non-interactively from a file or always requires a live session.
2. **[P1]** What format should the visual dependency graph use — a `.drawio` file (requires draw.io MCP), an embedded Mermaid diagram in CAMPAIGN.md, or a standalone SVG? The answer determines the toolchain required at spec-forge runtime and whether the graph is human-editable or tool-generated only.
3. **[P2]** How should spec-forge determine wave position for new milestones when the dependency graph has ambiguous ordering? The answer determines how the campaign update logic resolves insertion position.

---

## Non-Goals

- Fully autonomous spec design (always requires user interview)
- GitHub issue creation or milestone seeding (brimstone init scope)
- Architecture violation auto-remediation (flags only, does not fix)
