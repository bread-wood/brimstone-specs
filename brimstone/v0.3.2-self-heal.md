# brimstone v0.3.2 — Self Heal

**Theme:** Close the monitor loop — classify anomalies by critical-path impact,
bead-ify them in their source repo, auto-repair what can be fixed deterministically,
and defer to human investigation for anything that requires judgment.

---

## Overview

The monitor detects and files anomalies. It cannot yet answer what matters: *does
this block the build, and can we fix it without human help?*

This milestone adds three capabilities:

**1. Critical-path classification.** Every anomaly is tagged `is_blocking: bool`
based on whether it affects a non-terminal bead in the active milestone. Blocking
anomalies interrupt the dispatch loop immediately.

**2. AnomalyBead — pinned to the source repo.** Anomalies become first-class beads
stored in the repo where the anomaly was detected (`~/.brimstone/beads/<owner>/<repo>/
anomalies/<id>.json`). If calculator has a bad bead state, the AnomalyBead lives in
calculator's bead store — not brimstone's. This makes the anomaly ledger self-contained
per repo. Brimstone watches remote repos' anomaly directories in `--watch` mode and
picks up AnomalyBeads that need repair work dispatched against them.

**3. Two-tier response.** Not all anomalies need a repair agent:

- **Auto-repair:** Anomalies where the correct fix is deterministic and safe to apply
  immediately. The monitor applies the fix inline (a bead write or a `gh` call),
  verifies on the next scan, and marks the AnomalyBead `repaired`. No issue filed,
  no agent dispatched. Examples: stale `in-progress` label, missing MergeQueue entry.

- **Deferred (human required):** Anomalies where the correct fix requires judgment
  about intent — which dependency is stale, whether a state corruption is legitimate,
  whether a zombie branch has useful uncommitted work. The monitor files a GitHub
  issue in the **source repo's** `repairs` milestone (not brimstone's), writes the
  AnomalyBead as `open`, and stops. A human triages the repair issue.

Brimstone's orchestrator does not dispatch repair agents for deferred anomalies in
this milestone — that extension (automated research + repair for ambiguous cases)
is a future milestone once the classification taxonomy matures.

**The cross-repo vision:** `brimstone monitor --watch=bread-wood/calculator` runs
alongside a calculator build. Calculator's monitor writes AnomalyBeads to
`~/.brimstone/beads/bread-wood/calculator/anomalies/`. Brimstone reads those,
auto-repairs what it can inline, and files deferred issues in calculator's own
`repairs` milestone. The build loop pauses for blocking anomalies, auto-repair
runs, and the loop resumes — without brimstone needing to dispatch a repair agent.

**Prerequisite:** v0.3.0-rolling-loop must complete — the pause/resume behavior
requires the rolling dispatch loop's concurrency slot model. Classification and
AnomalyBead work (tiers 1 and 2) can be implemented and tested independently.

## Success Criteria

- [ ] Every `Anomaly` produced by `run_all_detectors()` has an `is_blocking: bool`
  field set by `classify_blocking(anomaly, store, active_milestone)` per the
  Blocking Classification table
- [ ] AnomalyBeads are written to the source repo's bead store (`anomalies/` subdir)
  — never to brimstone's bead store when running in `--watch` mode against a remote
  repo
- [ ] Dedup is by `anomaly_id`; if an AnomalyBead already exists for a fingerprint,
  the monitor skips it; `monitor-filed.json` is a read-only fallback for beads
  created before this milestone
- [ ] **Auto-repair:** for each auto-repairable anomaly kind, the monitor applies the
  fix, logs it, and marks the AnomalyBead `repaired` on the next scan if the anomaly
  is gone — without filing a GH issue or dispatching an agent
- [ ] **Deferred:** for each deferred anomaly kind, the monitor writes the AnomalyBead
  (`state=open`) and files a GH issue in the source repo's `repairs` milestone with
  the kind-specific investigation checklist; no agent is dispatched
- [ ] When a blocking auto-repairable anomaly is detected while the rolling loop is
  running: the loop pauses new claims, auto-repair runs synchronously, and the loop
  resumes after the next clean scan; total pause is at most one scan interval
- [ ] When a blocking deferred anomaly is detected: the loop pauses, files the issue,
  emits a clear `brimstone status` message indicating the build is paused waiting for
  human intervention, and waits; the loop does NOT resume until the AnomalyBead
  transitions to `wont_fix` or `repaired` (manual update)
- [ ] `brimstone monitor --watch=<owner/repo>[,...]` reads each repo's
  `anomalies/` bead dir and source repo bead state; applies the same detect →
  classify → auto-repair or defer pipeline
- [ ] A `repairs` milestone exists in each monitored repo (created by
  `brimstone monitor --init --repo=<owner/repo>`); deferred issues are filed there
- [ ] `brimstone status` shows per-repo anomaly counts: `blocking`, `auto_repaired`,
  `deferred_awaiting_human`, `resolved`
- [ ] `brimstone monitor --once --dry-run` prints classification and proposed action
  (auto-repair or defer) without writing beads or filing issues
- [ ] `make test && make lint` pass on mainline

## Scope

**Included:**

- `Anomaly.is_blocking: bool` + `repair_tier: "auto" | "deferred"` fields
- `classify_blocking(anomaly, store, active_milestone) -> bool` in `monitor.py`
- `classify_repair_tier(anomaly) -> Literal["auto", "deferred"]` in `monitor.py`
  (pure function of `anomaly.kind` and `anomaly.details` — no I/O)
- `AnomalyBead` dataclass in `beads.py`; stored in source repo's bead store at
  `anomalies/<anomaly_id>.json`; `BeadStore` read/write/list methods
- `process_anomalies()` refactored:
  1. Write AnomalyBead to source repo store
  2. If `repair_tier == "auto"`: apply fix inline, log, no GH issue filed
  3. If `repair_tier == "deferred"`: file GH issue in source repo's `repairs`
     milestone; AnomalyBead records `gh_issue_number`
- Auto-repair implementations (see Auto-Repair table)
- Blocking pause in rolling loop: `_repair_pending` flag; slot-refill halts while set;
  resumes after next clean scan for auto-repairs; waits for human for deferred
- `--watch=<repos>` mode: reads remote bead stores via local filesystem paths;
  applies full pipeline per repo; writes AnomalyBeads to respective repo bead dirs
- `brimstone monitor --init --repo=<owner/repo>`: creates `repairs` milestone if absent
- `monitor-filed.json`: read-only backward-compat fallback; never written

**Repair tier classification:**

| Anomaly kind | Tier | Rationale |
|---|---|---|
| `label_drift` (any) | **auto** | Correct fix is deterministic: add or remove the label |
| `orphaned_merge` | **auto** | PR is open → add to MergeQueue; safe, reversible |
| `pre_pr_zombie` (branch exists with commits) | **auto** | Create the missing PR |
| `pre_pr_zombie` (no branch or empty branch) | **deferred** | Unknown if agent did useful work |
| `dep_cycle` | **deferred** | Which edge is stale requires human judgment |
| `phantom_dep` | **deferred** | Was the dep intentionally removed or accidentally lost? |
| `state_regression` | **deferred** | Cause of corruption must be understood before repair |
| `detector_error` | **deferred** | Monitor itself failed; human must investigate |

**Auto-repair actions (applied inline by the monitor, no agent dispatched):**

| Anomaly kind | Auto-repair action |
|---|---|
| `label_drift` claimed + missing label | `gh issue edit #N --add-label in-progress` |
| `label_drift` closed/abandoned + active label | `gh issue edit #N --remove-label in-progress` |
| `orphaned_merge` | Add to `MergeQueue`; write updated queue bead |
| `pre_pr_zombie` branch+commits | `gh pr create` from existing branch; write PRBead |

**Deferred issue bodies include:**
- Anomaly ID, kind, is_blocking, detected_at, affected issue numbers
- Bead state JSON dump (snapshot at time of detection)
- Kind-specific investigation checklist:
  - `dep_cycle`: list cycle members; determine which `blocked_by` entry is stale
  - `phantom_dep`: check if dep issue was deleted or never created; fix `blocked_by`
  - `state_regression`: inspect event log; identify root cause; restore from last valid state
  - `pre_pr_zombie` (no branch): determine if agent did work; re-dispatch or close
  - `detector_error`: identify which detector failed; fix or disable it

**Explicitly excluded:**

- Automated repair agent dispatch for deferred anomalies — this is the next milestone
  after classification matures
- Application-level detectors (test failures, runtime errors in target repos)
- Cross-repo anomaly correlation
- Multi-repo blocking pause coordination (each repo's pause is independent)
- Anomaly dashboard or historical analytics
- Telegram/Slack notifications

## Failure Modes

**Classification:**
- No campaign bead or no active milestone → all anomalies treated as non-blocking;
  log warning; do not abort monitor

**AnomalyBead write (source repo bead store):**
- Write fails → log error; skip this anomaly for this scan; retry next interval;
  do not file GH issue without bead

**Auto-repair:**
- `gh` call fails (API error, rate limit) → log error; leave AnomalyBead `open`;
  retry next scan; do not escalate to deferred without 3 consecutive failures
- Auto-repair applied but anomaly still present on next scan → log warning; leave
  AnomalyBead `open`; retry up to 3 times; after 3 failures escalate: transition
  to deferred, file GH issue

**Deferred issue filing:**
- `repairs` milestone missing → log error; skip issue filing; emit warning in
  `brimstone status`; operator must run `brimstone monitor --init`
- GH issue create fails → AnomalyBead stays `open` with `gh_issue_number=null`;
  retry next scan up to 3 times; after 3 failures log and stop retrying

**Blocking pause (deferred):**
- Build loop paused and operator never resolves the AnomalyBead → loop waits
  indefinitely; `brimstone status` clearly indicates "build paused, awaiting human
  intervention on anomaly <id>"; no auto-timeout or auto-resume

**`--watch` mode:**
- Source repo bead store unreadable → log error per repo; skip; continue watching others
- Source repo has no `anomalies/` dir → not an error; skip silently

## Interface Contracts

**AnomalyBead (in source repo bead store: `anomalies/<anomaly_id>.json`):**

```json
{
  "v": 1,
  "anomaly_id": "<first-16-hex-of-sha256-fingerprint>",
  "source_repo": "bread-wood/calculator",
  "kind": "orphaned_merge",
  "severity": "critical",
  "is_blocking": true,
  "repair_tier": "auto",
  "description": "<one-line summary>",
  "details": {},
  "state": "open",
  "auto_repair_attempts": 0,
  "gh_issue_number": null,
  "gh_issue_url": null,
  "detected_at": "<iso8601>",
  "resolved_at": null
}
```

**`brimstone status` anomaly block:**

```
anomalies:
  bread-wood/brimstone   blocking=0  auto_repaired=2  deferred=0  resolved=5
  bread-wood/calculator  blocking=1  auto_repaired=0  deferred=1  resolved=0
  [BUILD PAUSED — deferred anomaly bread-wood/calculator#42 awaiting human]
```

**`repairs` milestone GH issue labels:**

```
bug  P0  stage/impl   ← blocking deferred
bug  P2  stage/impl   ← non-blocking deferred
```

## Constraints

- AnomalyBeads always live in the source repo's bead store — never in brimstone's
  bead store when operating in `--watch` mode
- `_seed_work_beads` must skip the `repairs` milestone by name; no repair issues
  enter the normal impl queue
- Auto-repair actions must be idempotent — applying the same fix twice is safe
- Blocking pause must not terminate active agents — only halts slot-refill
- `anomaly_id` = first 16 hex chars of SHA-256(fingerprint string) — stable, short
- `repair_tier` is a pure function of `kind` + `details`; no network I/O required
  to classify; classification must complete in < 1ms

## Key Unknowns

1. **[P1]** How does `--watch` mode access the bead store of a remote repo? If
   brimstone and the target repo run on the same machine (dev), direct filesystem path
   works. In CI/production they may run on different hosts. Research stage must
   determine the target deployment topology and whether a shared path, object store,
   or lightweight read API is needed. This directly affects `BeadStore`'s interface
   contract and whether `--watch` is feasible in non-local deployments.

2. **[P2]** After 3 consecutive auto-repair failures, should the monitor escalate
   to deferred (file issue, stop trying) or should there be a backoff-and-retry loop?
   Escalation is safer (prevents infinite retry thrash) but may surprise the operator.
   Research stage should check whether any auto-repair action can fail transiently for
   legitimate reasons (e.g. transient GitHub API rate limit) vs. being genuinely broken.
