# breadmin-composer v0.2.0 — Hardened Core

**Theme:** Production-safe unattended execution.

## Overview

Makes the pipeline safe and reliable to run without a local Claude session. Hardens
credential handling so API keys never appear in subprocess environments, adds CI and
server deployment, and makes the orchestrator resilient to mid-run crashes. These three
capabilities are the prerequisite for everything in v0.2.1 and beyond: you cannot safely
deploy multi-model backends or multi-repo orchestration without a credential proxy and
a crash-safe orchestrator.

---

## Success Criteria

- [ ] A research agent subprocess receives credentials via a loopback proxy; intercepting
  the subprocess environment reveals no raw API key
- [ ] Key rotation occurs during a long session without interrupting in-flight agents
- [ ] `composer run --all --milestone <m>` triggered by a GitHub Actions `issues` event
  completes successfully without a personal Claude Code session on the host
- [ ] An orchestrator session killed mid-pipeline restarts and resumes from its last
  checkpoint; no issues are double-dispatched and no `in-progress` labels are orphaned
- [ ] A checkpoint file corrupted or deleted on restart produces a warning and a clean
  start rather than a crash
- [ ] `composer health` reports all v0.2.0 components as healthy before any dispatch begins
- [ ] `brimstone beads --repo <owner/repo>` renders a table of all WorkBead and PRBead
  states for the active campaign without requiring a running orchestrator session
- [ ] `make test` passes clean

---

## Scope

**Included:**

- **Credential proxy**: loopback proxy that intercepts the API base URL; subprocesses
  receive a session token rather than a raw API key; key rotation during long sessions
- **CI/server deployment**: GHA workflow triggered on `issues` events; deployment guide
  for running composer on a server without a personal session
- **Orchestrator crash recovery**: checkpoint written after every state transition;
  on restart, reconcile `in-progress` issues against open PRs; reclaim orphaned issues
  automatically
- **Bead visualizer**: `brimstone beads` renders a live terminal view of WorkBead and
  PRBead state across all issues in the active campaign — campaign progress bar,
  per-issue state (open/claimed/pr_open/merge_ready/closed), and PR CI status

## Failure Modes

**Credential proxy:**
- Proxy process crashes mid-session → orchestrator detects missing proxy, restarts it
  before next dispatch; no agent dispatched without a live proxy
- Key rotation fails → orchestrator logs error, pauses dispatch, does not dispatch with
  a stale key
- Subprocess ignores the overridden base URL → proxy receives no requests; orchestrator
  detects no output events and marks the issue failed

**CI/server deployment:**
- GHA runner lacks required secrets → workflow fails at pre-flight health check with a
  clear error before any dispatch
- Server deployment loses network mid-pipeline → crash recovery handles resumption on
  next run; no special server-specific path needed

**Crash recovery:**
- `in-progress` issue with open PR → resume monitoring; do not re-dispatch
- `in-progress` issue with no PR → unclaim and re-queue
- `in-progress` issue where PR is already merged → close issue, continue
- Checkpoint file corrupt or missing → start fresh; log warning; do not inherit stale state

## Interface Contracts

**Environment variables:**

| Variable | Required | Purpose |
|----------|----------|---------|
| `COMPOSER_GH_TOKEN` | Yes | GitHub token for issue/PR operations |
| `ANTHROPIC_API_KEY` | Yes | Raw API key; held only by the proxy after startup |

**CI trigger:**

The GHA workflow listens on the `issues` event. It does not require a persistent daemon;
the runner is ephemeral and relies entirely on checkpoint + GitHub state for resumption.

---

## Constraints

- Raw API keys must never appear in subprocess environment after the proxy is active
- Crash recovery must not require a state store beyond GitHub Issues and the checkpoint
  JSON; no database or new IPC mechanism in this version
- The proxy architecture must be backend-agnostic: it must not assume Claude Code is
  the only subprocess that will use it, since v0.2.1 adds Gemini and GPT-4.1 backends
- The checkpoint schema must be repo-scoped from the start; retrofitting multi-repo
  support into a single-repo schema in v0.2.2 must not require a migration
- CI deployment must not require a persistent daemon; the GHA runner is ephemeral
- All v0.1.x pipeline stages and CLI surface remain intact; no breaking changes
- **Credential proxy scope is brimstone subprocesses only.** Product repos (moot,
  breadwinner, jobman) read credentials from their own `.env` files — they do not use
  the brimstone credential proxy. When brimstone dispatches an impl agent to work in a
  product repo, that agent uses the repo's local `.env` for `make test` and `make lint`
  runs. The proxy scope must not silently expand to serve product repo processes.

## Key Unknowns

1. **[P1]** Can the API base URL be overridden so that all current and future subprocess
   backends route through a local credential proxy without per-backend configuration?
   The answer determines whether the proxy is a single universal intercept point or
   whether each new backend (Gemini, GPT-4.1) added in v0.2.1 requires its own
   credential-routing mechanism.

2. **[P1]** What checkpoint schema supports crash recovery for the current single-repo
   pipeline while remaining directly extensible to multi-repo orchestration in v0.2.2
   — specifically, can a per-repo scope be added to the existing schema without
   breaking existing checkpoints or requiring a migration step?

---

## Non-Goals

- Multiple model backends (v0.2.1)
- Telegram notifications (v0.2.1)
- Task complexity estimator (v0.2.1)
- Cross-model consensus (v0.2.2)
- Multi-repo orchestration (v0.2.2)
- A persistent background daemon (launchd, systemd)
- Agent memory or context persistence across retries
