# breadmin-youtube v0.1.0 — Prospector

**Theme:** Automated discovery and relevance scoring of YouTube financial content —
identifies videos worth mining for investment signals and writes them to a shared
queue consumable by breadwinner's ingestion pipeline.

---

## Overview

Financial YouTube channels publish continuously. Manually identifying which videos
contain actionable investment themes is high-effort and slow. This package automates
that discovery: given a configured set of channels (and optional keyword filters),
`breadmin-youtube` polls for new videos, scores them for investment relevance, and
writes qualifying video URLs to a queue that breadwinner's `bw ingest` command can
drain.

`breadmin-youtube` is a standalone signal-source package in the same role as
`breadmin-reddit`: it owns channel configuration, discovery cadence, dedup tracking,
and relevance scoring. It does not execute trades, access Alpaca, or call the LLM at
discovery time — the expensive LLM work happens downstream in `bw ingest`.
Scheduling is handled by `breadmin-jobman` via a `youtube-scan` job (see
breadmin-jobman v0.4.0-signal-grid).

The queue store is a lightweight SQLite file at a configurable path (default:
`~/.breadmin/youtube/queue.db`). `bw ingest --from-queue` drains it
(breadwinner v0.5.0-ticker-tape extends the CLI to support queue mode).

---

## Success Criteria

- [ ] `breadmin-youtube scan` discovers new videos published since the last run across
  all configured channels and writes qualifying video records to the queue store
- [ ] Videos already present in the queue store (by video ID) are never written again;
  dedup is enforced at scan time
- [ ] Each queued video record includes: video ID, channel ID, channel name, title,
  description snippet, published timestamp, relevance score, and status (`pending`)
- [ ] Relevance scoring uses title + description heuristics (keyword matching against a
  configurable investment keyword list); videos scoring below the configured threshold
  are discarded at scan time, not written to the queue
- [ ] `breadmin-youtube channels add <url-or-id>` adds a channel to the registry;
  `breadmin-youtube channels list` prints all configured channels
- [ ] `breadmin-youtube queue list` prints all pending queue entries with relevance
  scores
- [ ] `breadmin-youtube queue drain --exec="bw ingest --source=youtube --url={url} --watchlist={channel_slug}"` 
  iterates pending entries and runs the given command template for each, updating
  status to `processing` → `done` or `error`
- [ ] When a channel becomes unavailable (deleted, private) it is flagged in the
  registry with `status: unavailable` and skipped on subsequent scans without error
- [ ] `make test && make lint` pass on mainline

## Scope

**Included:**

- Channel registry: YAML config at `~/.breadmin/youtube/channels.yaml` (or path from
  env `BREADMIN_YOUTUBE_CONFIG`); channels identified by channel ID or handle; each
  entry supports optional keyword override list and per-channel relevance threshold
- Video discovery: poll each configured channel for videos newer than last scan
  timestamp; discovery method determined by research stage (YouTube Data API v3 or
  yt-dlp channel scanning — see Key Unknowns)
- Dedup tracking: `last_scanned_at` per channel in the queue store; video IDs stored
  permanently to prevent re-queuing
- Relevance scoring: configurable keyword list (investment-domain defaults provided)
  matched against title + description; score = weighted keyword hit count normalized to
  0.0–1.0; configurable threshold (default: 0.3)
- Queue store: SQLite at `~/.breadmin/youtube/queue.db` with schema below
- CLI commands: `scan`, `channels add/list/remove`, `queue list/drain/clear`
- `--dry-run` flag on `scan`: prints discovered + scored videos without writing to queue

**Explicitly excluded:**

- LLM-based relevance scoring at discovery time (too expensive per-video; reserved for
  `bw ingest` after manual queue review or queue drain)
- Transcript fetching (breadwinner's `bw ingest` handles this)
- breadmin-jobman scheduling (breadmin-jobman v0.4.0-signal-grid scope)
- TradingView or Alpaca integration (breadwinner scope)
- Video comments, community posts, or YouTube Shorts
- OAuth or YouTube account-level API access (public channel data only)

## Failure Modes

**Discovery:**
- YouTube Data API quota exceeded → log warning; skip remaining channels this cycle;
  write `quota_exceeded` flag to queue store so jobman can back off next schedule
- yt-dlp scrape blocked or returns no results → log warning; mark channel
  `last_scan_status: failed`; do not mark channel unavailable until 3 consecutive failures
- Channel has no new videos since last scan → skip silently; update `last_scanned_at`

**Relevance scoring:**
- Keyword config file missing or malformed → fail fast with clear error; do not scan
- Video title/description is empty (unlisted or age-restricted) → assign score 0.0;
  do not queue

**Queue store:**
- SQLite write fails (disk full, permissions) → log error; abort current scan; do not
  partially write
- Queue store missing on first run → auto-create with schema; not an error

**Queue drain:**
- Drain command template fails for a video → update status to `error`; log error;
  continue to next entry; do not abort the drain

## Interface Contracts

**Queue store schema (SQLite):**

```sql
CREATE TABLE videos (
  video_id        TEXT PRIMARY KEY,
  channel_id      TEXT NOT NULL,
  channel_name    TEXT NOT NULL,
  channel_slug    TEXT NOT NULL,
  title           TEXT NOT NULL,
  description     TEXT,
  published_at    TEXT NOT NULL,   -- ISO 8601
  discovered_at   TEXT NOT NULL,   -- ISO 8601
  relevance_score REAL NOT NULL,
  status          TEXT NOT NULL DEFAULT 'pending',  -- pending | processing | done | error
  url             TEXT NOT NULL,
  error_detail    TEXT
);

CREATE TABLE channels (
  channel_id      TEXT PRIMARY KEY,
  handle          TEXT,
  name            TEXT,
  added_at        TEXT NOT NULL,
  last_scanned_at TEXT,
  last_scan_status TEXT,           -- ok | failed | quota_exceeded
  status          TEXT NOT NULL DEFAULT 'active'  -- active | unavailable
);
```

**Channel config (`channels.yaml`):**

```yaml
channels:
  - id: UCxxxxxx
    name: "Example Finance Channel"
    slug: example-finance
    keywords: []          # empty = use global keyword list
    threshold: 0.3        # 0.0-1.0; overrides global if set

keyword_list:
  - "options"
  - "earnings"
  - "breakout"
  - "sector rotation"
  # ... investment domain defaults

default_threshold: 0.3
lookback_days: 7          # how far back to scan on first run per channel
```

**Queue record JSON (emitted by `queue list --json`):**

```json
{
  "video_id": "<str>",
  "url": "https://www.youtube.com/watch?v=<video_id>",
  "channel_slug": "<str>",
  "title": "<str>",
  "published_at": "<iso8601>",
  "relevance_score": 0.0,
  "status": "pending"
}
```

## Constraints

- No LLM calls at discovery/scoring time — scoring is keyword heuristics only
- No Alpaca, moot, or breadwinner imports — this package is a standalone signal source
- Queue store path is configurable via `BREADMIN_YOUTUBE_QUEUE_DB` env var; default
  `~/.breadmin/youtube/queue.db`
- Discovery method (YouTube Data API vs yt-dlp) is determined by research stage and
  hard-configured at install time — the package does not try both
- If YouTube Data API is used: API key is provided via `YOUTUBE_DATA_API_KEY` env var;
  never hardcoded
- All CLI output is machine-readable with `--json` flag
- Package is type-annotated; passes `ruff check` and `mypy --strict`

## Key Unknowns

1. **[P1]** YouTube Data API v3 vs yt-dlp for channel polling: the Data API is reliable
   and structured but has a daily quota (10,000 units/day; a channels.list + search.list
   per channel costs ~102 units, so ~98 channels/day on the free tier). yt-dlp requires
   no API key but is a scraper and can break. The choice determines the package's
   dependency surface, credential requirements, and failure characteristics. Research
   stage must evaluate quota headroom for the expected channel count and assess yt-dlp
   stability risk.

2. **[P2]** Is title + description keyword heuristics sufficient to filter out
   non-investment-relevant videos (vlogs, reaction content, shorts compilations), or
   does a lightweight classifier (e.g. zero-shot classification via a small local model)
   materially improve queue quality? Research stage should sample 50 videos from target
   channels and measure precision/recall of keyword scoring at threshold 0.3.

3. **[P2]** Should `breadmin-youtube` pre-screen video content quality using Gemini
   multimodal analysis before writing to the queue — rather than relying purely on title
   keywords? Gemini can process a video URL and return a relevance verdict in a single
   call, potentially catching misleading titles and filtering out low-signal content
   before `bw ingest` is invoked. This must be weighed against per-video Gemini API
   cost at discovery volume. Research stage coordinates with breadwinner v0.5.0 Key
   Unknown #1 — if Gemini is adopted as the primary decomposition method there, the
   same API is already available here at marginal cost.

---

## Non-Goals

- LLM-based video relevance scoring (reserved for future milestone if keyword scoring
  proves insufficient)
- Comment section analysis
- YouTube Shorts or live stream discovery
- Multi-platform support (Twitter/X Spaces, podcasts) — separate package if needed
- Automated ingestion without human queue review (queue drain is an explicit manual step
  unless scheduled by jobman)
- YouTube account authentication (public channel data only)
