## Overview

Takes a confirmed meal plan from v0.1.0 and produces a ready-to-order grocery cart. Ingredients across all meals are aggregated (duplicates combined, units normalised), mapped to purchasable products via a grocery delivery API, and a checkout link is generated with the cart pre-filled. The link is emailed and displayed in the web UI's shopping cart view. The user always completes the purchase themselves — no automatic ordering.

**Prerequisite:** v0.1.0-meal-planner must complete before this milestone — a confirmed plan with ingredient data is required as input to the cart pipeline.

## Success Criteria

- [ ] Web UI: confirming a meal plan shows a "Build cart" button that produces an aggregated shopping list
- [ ] `pantry cart --plan <plan_id>` outputs aggregated ingredient list as JSON
- [ ] Checkout link pre-fills a cart at the delivery service with all required ingredients
- [ ] "2 cups flour" + "1 cup flour" across two recipes → "3 cups flour" in the cart
- [ ] When a checkout link cannot be generated, the plain ingredient list is emailed as fallback
- [ ] Checkout link is displayed in the web UI and emailed
- [ ] `make test && make lint` pass on mainline

## Scope

**Included:**

- Ingredient aggregation: combine quantities across meals, normalise units
- Product matching: map aggregated ingredients to orderable products via delivery API
- Checkout link generation (Instacart Connect API or best available alternative — P1 Key Unknown)
- Web UI: shopping cart view with editable items before checkout; "regenerate link" button
- Email: shopping list + checkout link
- CLI: `pantry cart [--plan <plan_id>]`

**Explicitly excluded:**

- Price comparison across stores
- Pantry inventory tracking (deducting what you already have)

---

## Failure Modes

**Delivery API:**
- Delivery API unavailable → email plain ingredient list; web UI shows list with "checkout unavailable" notice
- Product not found for an ingredient → include in plain list with note; do not silently omit
- Checkout link expires → web UI "regenerate" button calls API again for a fresh link

**Ingredient aggregation:**
- Unit normalisation ambiguity (e.g. "2 cloves garlic" + "1 tbsp minced garlic") → flag for manual review in web UI; do not silently merge incompatible units

---

## Interface Contracts

**Shopping cart JSON (CLI output, web UI backend, email payload):**

```json
{
  "plan_id": "<uuid>",
  "items": [
    { "name": "<string>", "quantity": "<number>", "unit": "<string>", "notes": "<string|null>" }
  ],
  "checkout_url": "<string|null>",
  "generated_at": "<iso8601>"
}
```

**CLI:**

```
pantry cart [--plan <plan_id>]
```

---

## Constraints

- User always completes purchase; no automatic ordering
- Delivery API credentials in `.env`, gitignored
- Ingredient list is always complete even when checkout link fails
- No price comparison or store selection in this version
- No direct imports from breadmin-jobman; pantry remains standalone

---

## Key Unknowns

1. **[P1]** Is the Instacart Connect API accessible without formal partner approval? If not, what is the best available alternative for generating a pre-filled grocery cart checkout link (Kroger API, Walmart Grocery API, AnyList deep-link, or other)?
2. **[P2]** What is the correct strategy for normalising ingredient units across recipes from different sources? The answer determines whether a measurement conversion library is needed and how the aggregation pipeline handles ambiguous units.

---

## Non-Goals

- Automatic order placement (user clicks through to complete)
- Price comparison across stores
- Pantry inventory / "what do I already have" tracking
- Subscription or recurring order setup
- Delivery time slot selection
