## Overview

Pantry generates a personalised weekly meal plan and delivers it via email and a local web UI. A scheduled job fires on a configurable day (Friday by default) and sends an email with N meal suggestions (user-specified, default 7) plus a link to the web UI for review and refinement. The user can accept meals, reject them, or request alternatives with free-text specifications ("more fish, no gluten"). On-demand requests work for a single day or a date range. Recipes combine LLM generation with API-validated data, scaled to 1 person. No shopping or ordering in this milestone.

**Prerequisite:** none — this is v0.1.0.

## Success Criteria

- [ ] `pantry survey` sends a meal plan email and serves the web UI with N=7 suggestions
- [ ] `pantry survey --days 3` returns a 3-meal plan
- [ ] `pantry survey --date 2026-03-10` anchors the plan to that date
- [ ] `pantry survey --meals 5` returns exactly 5 suggestions
- [ ] Weekly job fires on the configured day; email arrives with a link to the web UI
- [ ] Web UI: requesting "more suggestions" with a free-text spec returns alternatives matching the spec
- [ ] Each confirmed meal has ingredient names and quantities scaled to 1 person
- [ ] Recipes are sourced from the external recipe API where a match exists; LLM-generated where not
- [ ] Email includes the full meal plan as text (not just a link) so it's readable without the web UI
- [ ] `make test && make lint` pass on mainline

## Scope

**Included:**

- CLI: `pantry survey [--days N] [--date YYYY-MM-DD] [--meals N]`
- LLM meal suggestion via breadmin-llm
- Recipe lookup via external recipe API (Spoonacular or Edamam — P2 Key Unknown)
- LLM-generated recipe as fallback when API has no match
- Recipe scaling to 1 person
- FastAPI web UI: browse suggestions, accept/reject, free-text refinement, confirm plan
- Email delivery: survey with web UI link + full plan text inline; confirmation with final plan
- Weekly scheduler config: launchd plist installed via `pantry schedule --install`
- SQLite: current plan, meal history

**Explicitly excluded:**

- Instacart / any shopping integration (v0.2.0)
- Multi-person scaling or household profiles

---

## Failure Modes

**Recipe API:**
- Recipe API unavailable → fall back to LLM-generated recipe; log warning; do not block the plan

**LLM:**
- LLM call fails → retry once; surface error in web UI and email; do not deliver partial plan silently

**Email delivery:**
- Email delivery fails → log error; web UI still serves the plan; retry on next run
- Weekly job fires but email credentials not configured → log error at job time; warn on next CLI run
- Email link unreachable → email includes full plan as plain text fallback

## Interface Contracts

**Meal plan JSON (CLI output, web UI backend, SQLite record):**

```json
{
  "plan_id": "<uuid>",
  "generated_at": "<iso8601>",
  "days": 7,
  "meals": [
    {
      "name": "<string>",
      "recipe_source": "api | llm",
      "ingredients": [
        { "name": "<string>", "quantity": "<number>", "unit": "<string>" }
      ],
      "instructions": "<string>",
      "servings": 1
    }
  ]
}
```

**CLI:**

```
pantry survey [--days N] [--date YYYY-MM-DD] [--meals N]
pantry schedule --install
```

## Constraints

- Web UI served on localhost only
- Recipe API key in `.env`, gitignored
- Gmail credentials in `.env` or credentials JSON path, gitignored
- All recipes stored scaled to exactly 1 serving
- No direct imports from breadmin-jobman; pantry is standalone
- Email transport follows `breadmin-core/transports/email.py` pattern; no jobman import

---

## Key Unknowns

1. **[P2]** Which recipe API (Spoonacular, Edamam, or other) provides the best ingredient data quality and free-tier coverage for single-person meal planning at scale?
2. **[P2]** When the LLM suggests a meal with no API match, what is the right fallback strategy — LLM-only recipe, re-suggest a different meal, or present both? The answer determines the recipe resolution pipeline.

## Non-Goals

- Instacart / shopping integration (v0.2.0)
- Multi-person scaling or household profiles
- Nutritional tracking or dietary scoring
- Budget-constrained planning
- Mobile app or native push notifications
