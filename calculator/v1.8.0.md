# Calculator CLI v1.8.x — Vector Fields

**Theme:** Arrows across the plane. Direction and flow made visible.

## Overview

Adds vector field plotting and streamline rendering. A vector field assigns
a 2D vector to every point in the plane; plotting it as arrows reveals the
flow of a physical system, the gradient of a scalar field, or the direction
field of a differential equation. `calc plotvf '[y, -x]'` shows the
counterclockwise rotation field; `calc plotvf '[grad f]'` shows where a
surface climbs fastest. This version also integrates vector fields with the
existing curve renderer so that solution curves and field arrows appear on
the same axes.

---

## Success Criteria

- [ ] `calc plotvf '[y, -x]'` opens a window displaying arrows arranged in
  a counterclockwise pattern and exits 0 when closed
- [ ] `calc plotvf '[1, 0]'` displays uniform horizontal arrows
- [ ] `calc plotvf '[y, -x]' --screenshot out.png` renders to file and exits 0
- [ ] `calc plotvf '[y, -x]' --streamlines` renders smooth flow lines instead
  of discrete arrows
- [ ] `calc plotvf '[y, -x]' --normalize` displays arrows of uniform length
  (direction only, not magnitude)
- [ ] `calc plot 'sin(x)' --vf '[cos(x), 0]'` overlays a vector field on
  a standard 2D curve plot in the same window
- [ ] Arrow color encodes magnitude by default; `--colormap` selects palette
- [ ] `calc plotvf '[y]'` (not a 2-element vector) prints
  `error: vector field must return a 2-element vector` to stderr and exits 1
- [ ] `make test` passes clean using `--screenshot` mode

---

## Scope

**Included:**

- `calc plotvf '[Fx(x,y), Fy(x,y)]'` subcommand; free variables are `x` and `y`
- Arrow rendering: sampled on a regular grid; arrow length proportional to magnitude
- Streamline rendering: `--streamlines` flag; smooth curves following the field
- Normalized arrows: `--normalize` flag; uniform arrow length
- Color by magnitude: default; configurable with `--colormap`
- Overlay on curve plots: `--vf '[Fx, Fy]'` flag on `calc plot`
- Configurable domain and grid density: all `plot3d` domain flags apply
- `--screenshot` for headless rendering

---

## Failure Modes

**Expression:**
- Field expression does not return a 2-element list → `error: vector field must return a 2-element vector` to stderr, exit 1
- Field undefined at one or more grid points → skip those arrows; not an error

---

## Interface Contracts

**CLI:**

```
calc plotvf '<[Fx, Fy]>' [--streamlines] [--normalize] [--colormap NAME]
    [--xmin N] [--xmax N] [--ymin N] [--ymax N] [--resolution N]
    [--screenshot PATH] [--output PATH] [--width N] [--height N]

calc plot '<expr>' --vf '<[Fx, Fy]>' --interactive [all plot flags]
```

| Stream | Content |
|--------|---------|
| stdout | Empty |
| stderr | `error: <description>` on failure |
| exit 0 | Window closed or `--screenshot` completed |
| exit 1 | Fatal error |

---

## Constraints

- All v0.7.x constraints remain in force
- Vector field expressions must use the same expression language as all other
  modes; no new syntax is introduced for field components
- The `[Fx, Fy]` expression is a list literal from v1.5.x; `plotvf` requires
  lists to be available
- Streamline integration uses a fixed-step ODE solver; accuracy and termination
  conditions are configurable via environment variables

---

## Key Unknowns

1. **[P2]** Does rendering arrows and streamlines in the same scene as 2D curves
   compose naturally with the existing curve rendering pipeline — sharing axes,
   labels, and color palette management — or does the fundamentally different
   nature of field data (defined over a 2D grid rather than a 1D sample sequence)
   require a separate rendering layer that cannot reuse the curve abstraction?
   The answer determines whether vector fields are an extension of the curve
   renderer or a new renderer composited at the scene level.

---

## Non-Goals

- 3D vector fields
- Curl and divergence as first-class functions (computable from `diff` in v2.0.x)
- Animated fields (time-varying vector fields)
- Windows support
