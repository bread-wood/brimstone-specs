# Calculator CLI v1.4.x — Bitwise

**Theme:** The other half of a developer calculator. Bits, not just numbers.

## Overview

Adds hexadecimal, binary, and octal integer literals, bitwise operators, and
output base selection. The target audience — developers working in the terminal
— reaches for `0xFF & 0x0F` and `1 << 4` as often as they reach for `sin`.
Every other developer calculator supports this. This version adds it cleanly
to a language that was designed from the start to extend without parser rewrites.

- [ ] `calc '0xFF'` prints `255` to stdout and exits 0
- [ ] `calc '0b1010'` prints `10` to stdout and exits 0
- [ ] `calc '0o17'` prints `15` to stdout and exits 0
- [ ] `calc '0xFF & 0x0F'` prints `15` to stdout and exits 0
- [ ] `calc '0xFF | 0x100'` prints `511` to stdout and exits 0
- [ ] `calc '0xFF ^ 0x0F'` prints `240` to stdout and exits 0
- [ ] `calc '~0'` prints `-1` to stdout and exits 0
- [ ] `calc '1 << 4'` prints `16` to stdout and exits 0
- [ ] `calc '256 >> 4'` prints `16` to stdout and exits 0
- [ ] `calc --hex '255'` prints `0xff` to stdout and exits 0
- [ ] `calc --binary '10'` prints `0b1010` to stdout and exits 0
- [ ] `calc --octal '8'` prints `0o10` to stdout and exits 0
- [ ] `calc '0xFF & 1.5'` prints `error: bitwise operand must be integer` to
  stderr and exits 1
- [ ] All prior success criteria still pass
- [ ] `make test` passes clean on macOS and Linux

## Scope

**Included:**

- Integer literals: `0x<hex>`, `0b<binary>`, `0o<octal>`
- Bitwise operators: `&` (AND), `|` (OR), `^` (XOR), `~` (NOT), `<<` (left shift),
  `>>` (right shift)
- Output base flags: `--hex`, `--binary`, `--octal` (apply to the final result)
- Operator precedence: bitwise operators below arithmetic, above comparisons
- Bitwise operators require integer operands; applying them to non-integer values
  is an error
- Integer literals may be used anywhere a number is valid

---

## Failure Modes

**Type errors:**
- Bitwise operator applied to non-integer → `error: bitwise operand must be integer` to stderr, exit 1
- Shift amount is negative → `error: shift amount must be non-negative` to stderr, exit 1

**Overflow:**
- Shift produces a value exceeding the integer representation range →
  `error: overflow` to stderr, exit 1

---

## Interface Contracts

**CLI:**

```
calc [--hex | --binary | --octal] '<expression>'
```

| Flag | Purpose |
|------|---------|
| `--hex` | Print result in hexadecimal (`0x...`) |
| `--binary` | Print result in binary (`0b...`) |
| `--octal` | Print result in octal (`0o...`) |

**Operator precedence (updated):**

```
or / and / not
> < >= <= = !=
| (bitwise OR)
^ (bitwise XOR)
& (bitwise AND)
<< >>
+ -
* /
unary - / ~
exponentiation / function call
```

| Stream | Content |
|--------|---------|
| stdout | Result in selected base |
| stderr | `error: <description>` on failure |
| exit 0 | Success |
| exit 1 | Any error |

---

## Constraints

- All prior constraints remain in force
- Bitwise operations work on 64-bit signed integers; results outside this range
  produce an overflow error
- The `^` operator is XOR in bitwise context and exponentiation in arithmetic
  context; the parser must disambiguate by operand type at evaluation time
  (both operands integer → XOR; otherwise → exponentiation)
- Output base flags apply only to the top-level result; intermediate values
  are always computed in decimal

## Key Unknowns

1. **[P1]** Does the `^` token's dual role — XOR when both operands are integers,
   exponentiation otherwise — require type-aware parsing or can it be resolved
   cleanly at evaluation time without ambiguity in all expression contexts
   (including complex numbers from v1.3.x, where `i^2` must remain exponentiation)?
   The answer determines whether `^` retains both meanings or whether XOR needs
   a dedicated operator token (e.g. `xor` or `^^`).

---

## Non-Goals

- Arbitrary-precision integers
- Unsigned integer types
- Bit-field manipulation
- Windows support
