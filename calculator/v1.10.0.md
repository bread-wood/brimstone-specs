# Calculator CLI v1.10.x — Pipeline

**Theme:** The calculator joins the shell. Composable with everything.

## Overview

Makes the calculator a proper shell citizen. Expressions can be piped in from
stdin, read from script files, and watched for file changes. A script file is
a sequence of statements that the evaluator runs top to bottom, printing the
value of each expression statement. Watch mode re-evaluates the file every time
it changes on disk — the developer edits a formula file in one pane and sees
results update in the other. This is the version that turns `calc` from a
one-shot command into a tool that lives in workflows.

---

## Success Criteria

- [ ] `echo '2 + 2' | calc` prints `4` to stdout and exits 0
- [ ] `echo 'sin(pi/6)' | calc` prints `0.5` and exits 0
- [ ] `calc run script.calc` evaluates the file top to bottom, prints the value
  of each expression statement, and exits 0
- [ ] A script file with a syntax error on line 3 prints
  `error: line 3: <description>` to stderr and exits 1
- [ ] `calc watch script.calc` evaluates the file, prints results, then
  re-evaluates and reprints whenever the file changes; Ctrl-C exits 0
- [ ] `calc watch script.calc` detects a file change within 500 ms of the
  write completing
- [ ] `calc run script.calc` where `script.calc` does not exist prints
  `error: cannot read file: script.calc` to stderr and exits 1
- [ ] `calc run script.calc --output results.json` writes all expression values
  as a JSON array to `results.json` and exits 0
- [ ] `make test` passes clean

## Scope

**Included:**

- Stdin mode: when no expression argument is given and stdin is a pipe,
  read the full stdin as the expression
- `calc run <file>` subcommand: evaluate a `.calc` script file
- Script file format: one statement per line; `#` begins a line comment;
  blank lines ignored; same statement grammar as the interactive REPL
- Per-statement output: value of each expression statement printed; assignment
  statements print nothing unless they are the last statement
- `calc watch <file>` subcommand: run then re-run on file change
- `--output PATH` on `calc run`: write all expression results as a JSON array
- Error reporting with line numbers for script files

---

## Failure Modes

**Stdin:**
- Stdin is a TTY (not a pipe) and no expression argument given → print usage
  to stderr and exit 1

**Script:**
- File not found → `error: cannot read file: <path>` to stderr, exit 1
- File contains a syntax error → `error: line <N>: <description>` to stderr, exit 1;
  evaluation halts at the first error

**Watch:**
- File deleted while watching → print `error: file deleted: <path>` to stderr;
  continue watching; re-evaluate if the file reappears

## Interface Contracts

**CLI:**

```
calc                                   # stdin mode (only when stdin is a pipe)
calc run <file> [--output <path>]
calc watch <file>
```

**Script file format:**

```
# This is a comment
x = 5
y = x * 2
y + 1        # prints 11
```

**`--output` JSON schema:**

```json
[
  { "line": 3, "expression": "y + 1", "value": 11 }
]
```

| Stream | Content |
|--------|---------|
| stdout | Expression results, one per line |
| stderr | `error: <description>` on failure |
| exit 0 | All statements evaluated successfully |
| exit 1 | Any error |

---

## Constraints

- All prior constraints remain in force
- Script files use the same parser and evaluator as the interactive modes;
  no new language features are introduced
- Watch mode uses filesystem events, not polling, where the OS supports them;
  falls back to 250 ms polling if not
- The script format must extend to a notebook format (interleaved markdown and
  expressions) in a future version; the `.calc` extension is reserved for
  pure-expression scripts; `.calcbook` will be the notebook extension

---

## Key Unknowns

1. **[P2]** Does the per-statement output model — print each expression value
   as it is evaluated — extend cleanly to a notebook format where expression
   output is embedded inline within a markdown document, or does the notebook
   format require a fundamentally different output model (collect all results,
   then render the document)? The answer determines whether the script runner's
   output pipeline is the foundation for notebooks or must be redesigned.

---

## Non-Goals

- Notebook format (`.calcbook`) — future version
- Shell integration (completion, `calc` as a REPL shell)
- Parallel script evaluation
- Import/include between script files
- Windows support
