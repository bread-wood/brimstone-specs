# Calculator CLI v1.3.x ŌĆö Complex Numbers

**Theme:** Extend the number line into the plane. Imaginary is real.

## Overview

Extends the number system to complex numbers. `sqrt(-1)` produces `i` rather
than a domain error; `exp(i*pi)` produces `-1`; `(1 + 2i) * (3 - i)` produces
`5 + 5i`. All existing functions are extended to accept complex inputs where
mathematically defined. This version validates that the type system introduced
in v0.1.x extends to a second numeric type without replacing the evaluator ŌĆö
and it unlocks domain coloring in the renderer for a future version.

## Success Criteria

- [ ] `calc 'i'` prints `0+1i` to stdout and exits 0
- [ ] `calc 'i^2'` prints `-1` to stdout and exits 0
- [ ] `calc 'sqrt(-1)'` prints `0+1i` to stdout and exits 0
- [ ] `calc '(1 + 2i) * (3 - i)'` prints `5+5i` to stdout and exits 0
- [ ] `calc 'exp(i * pi)'` prints `-1+0i` ŌĆö displayed as `-1` (zero imaginary
  part is suppressed when exactly zero, or within floating-point tolerance)
- [ ] `calc 'abs(3 + 4i)'` prints `5` to stdout and exits 0
- [ ] `calc 'real(3 + 4i)'` prints `3` to stdout and exits 0

- [ ] `calc 'conj(3 + 4i)'` prints `3-4i` to stdout and exits 0
- [ ] `calc 'log(-1)'` prints `0+3.141592653589793i` to stdout and exits 0
  (principal value)
- [ ] `calc '1 / 0i'` prints `error: division by zero` to stderr and exits 1
- [ ] All prior success criteria still pass with real inputs producing real outputs
- [ ] `make test` passes clean on macOS and Linux

## Scope

**Included:**

- `i` as a built-in constant: `0 + 1i`
- Complex literal syntax: `a + bi` where `a` and `b` are real literals
- Arithmetic over complex numbers: `+`, `-`, `*`, `/`, `^`
- Extended built-in functions accepting complex input: `sqrt`, `exp`, `log`,
  `sin`, `cos`, `tan`, `abs`, `pow`
- New complex-specific functions: `real(z)`, `imag(z)`, `conj(z)`, `arg(z)`
- Output formatting: `a+bi`, `a-bi`, `bi` (zero real part), `a` (zero imaginary
  part); trailing `+0i` suppressed when imaginary part is within tolerance of zero
- All prior language features unchanged; real-valued expressions continue to
  produce real output

---

## Failure Modes

**Arithmetic:**
- Division by zero in complex arithmetic ŌåÆ `error: division by zero` to stderr, exit 1
- Overflow in real or imaginary component ŌåÆ `error: overflow` to stderr, exit 1

**Output:**
- Complex result where a purely real result was expected by the caller
  (e.g. in `sum` bounds) ŌåÆ `error: expected real value, got complex` to stderr, exit 1

## Interface Contracts

**CLI:** Unchanged.

**New functions:**

| Function | Description |
|----------|-------------|
| `real(z)` | Real part of z |
| `imag(z)` | Imaginary part of z |
| `conj(z)` | Complex conjugate |
| `arg(z)` | Argument (angle) in radians, principal value in (-ŽĆ, ŽĆ] |

**Output format:**

| Case | Example output |
|------|----------------|
| Pure real | `5` |
| Pure imaginary | `3i` |
| Complex | `3+4i` or `3-4i` |
| Near-zero imaginary | `ŌłÆ1` (not `ŌłÆ1+0i`) |

| Stream | Content |
|--------|---------|
| stdout | Result in output format above |
| stderr | `error: <description>` on failure |
| exit 0 | Success |
| exit 1 | Any error |

## Constraints

- All prior constraints remain in force
- Real inputs to functions that previously only accepted reals must continue
  to produce real outputs (no silent promotion to complex)
- The complex number type must extend to the graphing layer: plotting a
  real-valued expression that returns complex values at some points (e.g. `sqrt(x)`
  for negative `x`) should skip those points rather than error
- The type system extension must accommodate vectors and matrices in v1.5.x
  without requiring a third incompatible numeric type implementation

## Non-Goals

- Quaternions or higher-dimensional algebras
- Exact complex arithmetic (Gaussian integers)
- Windows support
