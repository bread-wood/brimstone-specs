# Calculator CLI v1.0.x — Session

**Theme:** Work persists. Close the window; come back where you left off.

## Overview

Adds session save and load so users can close the calculator and reopen their
work exactly as they left it: the same expressions, the same viewport, the same
variable definitions. Introduces REPL mode as a terminal-native complement to
the interactive window — a persistent prompt where expressions, variable
assignments, and function definitions accumulate across commands in a single
session. This is the version where the calculator becomes a tool users live in,
not one they invoke and discard.

---

## Success Criteria

- [ ] `calc plot --interactive --session my-work.json` saves the session to
  `my-work.json` on quit and exits 0
- [ ] Re-running `calc plot --interactive --session my-work.json` restores the
  expressions, viewport (xmin, xmax, ymin, ymax), and variable definitions
  exactly as saved
- [ ] `calc plot --interactive` without `--session` behaves exactly as v0.8.x
  (no auto-save, no session file written)
- [ ] `calc plot --interactive --session missing.json` where the file does not
  exist opens a fresh empty session and exits 0 on quit after writing the file
- [ ] `calc plot --interactive --session corrupt.json` where the file is not
  valid JSON prints `error: session file corrupt: <reason>` to stderr and exits 1
- [ ] `calc repl` opens an interactive prompt; typing `x = 5` and then `x * 2`
  on the next prompt prints `10`; typing `def f(n) = n^2` and then `f(4)` prints `16`
- [ ] `calc repl --session my-work.json` restores variable and function definitions
  from the session on startup; saves them on exit
- [ ] `calc repl` without `--session` starts fresh each time; no state written to disk
- [ ] `calc repl` typing `quit` or pressing Ctrl-D exits 0
- [ ] `make test` passes clean on macOS and Linux; REPL tests use scripted stdin

## Scope

**Included:**

- Session file format: JSON document capturing expression list, viewport bounds,
  and named variable/function definitions
- `--session PATH` flag on `calc plot --interactive`: save on quit, restore on open
- `--session PATH` flag on `calc repl`: save definitions on exit, restore on start
- `calc repl` subcommand: line-oriented REPL where each line is a statement;
  definitions accumulate; last expression value is printed after each line
- REPL supports all statement types from v0.4.x: arithmetic, variable assignment,
  function definition
- REPL built-in commands: `quit`, `clear` (reset all definitions), `list`
  (print all defined variables and functions)
- Session file auto-saved on clean exit; not written on crash

---

## Failure Modes

**Session file:**
- File path not writable → `error: cannot write session: <reason>` to stderr on exit;
  session not saved; exit 1
- File exists but is not valid JSON → `error: session file corrupt: <reason>` to
  stderr; exit 1
- File exists but was written by a newer version → `error: session file version
  unsupported` to stderr; exit 1
- Process killed mid-write → session file may be partially written; next open
  detects corruption and reports it; no silent data loss

**REPL:**
- Syntax error on a REPL line → print `error: <description>`; prompt for next line;
  no accumulated state is affected
- Domain or arithmetic error → print `error: <description>`; prompt continues
- Ctrl-C → cancel current line; prompt for new line (does not exit)
- Ctrl-D → exit 0; session saved if `--session` specified

## Interface Contracts

**CLI:**

```
calc plot ['<expr>' ...] --interactive [--session PATH] [all v0.9.x flags]
calc repl [--session PATH]
```

| Flag | Default | Purpose |
|------|---------|---------|
| `--session PATH` | — | Path to session file; save on exit, restore on open |

**REPL prompt:**

```
> x = 5
> x * 2
10
> def f(n) = n^2
> f(4)
16
> list
variables: x = 5
functions: f(n) = n^2
> quit
```

**Session file schema:**

```json
{
  "version": 1,
  "viewport": { "xmin": -10, "xmax": 10, "ymin": -6, "ymax": 6 },
  "expressions": [
    { "text": "sin(x)", "color": "#1f77b4" },
    { "text": "x^2 + y^2 = 25", "color": "#ff7f0e" }
  ],
  "variables": { "x": 5 },
  "functions": [
    { "name": "f", "params": ["n"], "body": "n^2" }
  ]
}
```

| Stream | Content |
|--------|---------|
| stdout | REPL output (expression results, `list` output) |
| stderr | `error: <description>` on fatal errors |
| exit 0 | Clean exit |
| exit 1 | Session file corrupt, version unsupported, or write failure on exit |

---

## Constraints

- All v0.9.x constraints remain in force
- Session file format is versioned; a `version` field must be present and checked
  on load; unrecognised versions are rejected with a clear error
- The session file must be written atomically (write to a temp file, then rename)
  to prevent corruption on crash mid-write
- REPL state and interactive window session state use the same session file schema;
  a session created in the REPL can be opened in the window and vice versa

---

## Key Unknowns

1. **[P1]** Is the session file format — a JSON snapshot of expressions, viewport,
   and definitions — a sufficient foundation for future sharing and collaboration
   features, or does it need to be a diff-based or CRDT-structured format from the
   start to avoid a breaking schema change when sharing is introduced? The answer
   determines whether the v1.0.x session schema is the permanent on-disk format
   or a first version that will require migration.

---

## Non-Goals

- Auto-save on every change (save on clean exit only)
- Session sharing or sync across machines
- Variable sliders
- Undo/redo across sessions
- Plugin or extension system
- Windows support
