# Calculator CLI v1.2.x — Summation

**Theme:** Math notation for repeated operations. Σ, Π, ∫ as first-class syntax.

## Overview

Adds summation, product, and numerical integration as first-class language
constructs with bound variable scoping. `sum(k, 1, 10, k^2)` is cleaner and
more readable than any equivalent user-defined function, and `integral(sin(x),
0, pi)` produces a numerical result without requiring the user to understand
quadrature. These are the operations mathematicians reach for constantly; their
absence from the language is conspicuous. The bound variable mechanism introduced
here also sets the pattern for list comprehensions and generators in v1.5.x.

## Scope

**Included:**

- `sum(var, from, to, expr)` — discrete sum; `var` takes integer values from `from` to `to` inclusive
- `product(var, from, to, expr)` — discrete product over the same range
- `integral(expr, from, to)` — numerical integration of `expr` over `x` from `from` to `to`; `x` is the free variable
- Bound variable scoping: `var` in `sum`/`product` is local to `expr`; it shadows outer variables of the same name
- Empty range: `sum` with `from > to` returns 0; `product` returns 1
- Nested `sum`/`product`/`integral` calls supported
- All prior language features unchanged

---

## Failure Modes

**Range errors:**
- Non-integer `from` or `to` in `sum`/`product` → truncated to integer; not an error
- `from` or `to` evaluates to a domain error → `error: domain error in range` to stderr, exit 1

**Integration:**
- Integrand evaluates to a domain error at one or more sample points within the
  domain → `error: domain error` to stderr, exit 1
- Integration result fails to converge within tolerance → `error: integral did not converge` to stderr, exit 1

## Interface Contracts

**CLI:** Unchanged.

**Construct signatures:**

| Construct | Signature | Notes |
|-----------|-----------|-------|
| `sum` | `sum(var, from, to, expr)` | `var` bound; integer steps |
| `product` | `product(var, from, to, expr)` | `var` bound; integer steps |
| `integral` | `integral(expr, from, to)` | `x` as implicit free variable; numerical |

| Stream | Content |
|--------|---------|
| stdout | Computed result |
| stderr | `error: <description>` on failure |
| exit 0 | Success |
| exit 1 | Any error |

---

## Constraints

- All prior constraints remain in force
- `sum` and `product` iterate with integer steps only; non-integer ranges are
  implicitly truncated — no fractional iteration
- `integral` uses `x` as the integration variable; this is consistent with the
  graphing free variable convention and avoids a fourth argument for the common case
- The bound variable mechanism in `sum`/`product` must extend to list
  comprehension syntax in v1.5.x without introducing a second variable-binding
  mechanism
- Numerical integration accuracy must be within 1e-6 of the true value for
  smooth integrands on finite domains

## Key Unknowns

1. **[P1]** Does the bound variable scoping mechanism introduced for `sum` and
   `product` — where a name is locally bound within a sub-expression — extend
   cleanly to list comprehensions (`[f(k) for k in 1..10]`) in v1.5.x without
   requiring a second, incompatible variable-binding implementation? The answer
   determines whether the scoping model from this version is the permanent
   mechanism for all bound variables in the language.

---

## Non-Goals

- Symbolic (exact) integration — numerical only
- Integration over infinite domains
- Multi-variable integration
- Step size other than 1 for `sum`/`product`
- Windows support
