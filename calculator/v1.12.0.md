# Calculator CLI v1.12.x — Units

**Theme:** Numbers with meaning. Dimensional arithmetic that catches mistakes.

## Overview

Adds physical units to the number system. `5 km + 3 mi` converts and adds;
`9.8 m/s^2 * 70 kg` produces `686 N`; `5 m + 3 s` is a type error caught at
evaluation time. Targets engineers, physicists, and students who work with
physical quantities and want the calculator to enforce dimensional consistency
automatically. A calculation that passes dimension checking is more trustworthy
than one that doesn't.

---

## Success Criteria

- [ ] `calc '5 km + 3 km'` prints `8 km` to stdout and exits 0
- [ ] `calc '5 km + 3 mi'` converts and prints the result in km and exits 0
- [ ] `calc '9.8 m/s^2 * 70 kg'` prints `686 N` and exits 0
- [ ] `calc '60 mi/hr --unit m/s'` converts and prints `26.8224 m/s` and exits 0
- [ ] `calc '5 m + 3 s'` prints `error: dimension mismatch: length + time` to stderr
  and exits 1
- [ ] `calc '5 m * 3 m'` prints `15 m^2` and exits 0
- [ ] `calc '10 m / 2 s'` prints `5 m/s` and exits 0
- [ ] `calc 'sqrt(9 m^2)'` prints `3 m` and exits 0
- [ ] `calc '5 km'` prints `5 km` — unit preserved in output
- [ ] `calc '5'` prints `5` — dimensionless values are unchanged
- [ ] `calc --unit ft '1 m'` prints `3.28084 ft` and exits 0
- [ ] `make test` passes clean

## Scope

**Included:**

- Unit literal syntax: `<number> <unit>` (e.g. `5 km`, `9.8 m/s^2`)
- SI base units: `m`, `kg`, `s`, `A`, `K`, `mol`, `cd`
- SI derived units: `N`, `J`, `W`, `Pa`, `Hz`, `V`, `Ω`, `C`, `F`, `T`
- Common non-SI: `km`, `cm`, `mm`, `μm`, `nm`; `mi`, `ft`, `in`, `yd`;
  `hr`, `min`, `ms`, `μs`, `ns`; `L`, `mL`; `°C`, `°F` (temperature conversion)
- SI prefixes: `k`, `M`, `G`, `T`; `c`, `m`, `μ`, `n`, `p`
- Arithmetic: unit-aware `+`, `-`, `*`, `/`, `^`; automatic conversion for `+`/`-`
- Dimension checking: `+` and `-` require matching dimensions; `*` and `/`
  compose dimensions; `^` raises dimensions to a power
- Output: result with simplified unit; `--unit <unit>` forces output in specified unit
- Dimensionless results (units cancel): printed as a plain number

---

## Failure Modes

**Dimension:**
- Adding or subtracting incompatible dimensions → `error: dimension mismatch: <dim1> + <dim2>` to stderr, exit 1
- Function argument requires dimensionless input (`sin`, `log`, etc.) but receives
  a dimensioned value → `error: expected dimensionless argument, got <unit>` to stderr, exit 1

**Unit:**
- Unknown unit name → `error: unknown unit: <name>` to stderr, exit 1
- `--unit` specifies a unit with incompatible dimension → `error: cannot convert <result-dim> to <target-unit>` to stderr, exit 1

---

## Interface Contracts

**CLI:**

```
calc [--unit <unit>] '<expression with units>'
```

| Flag | Default | Purpose |
|------|---------|---------|
| `--unit <unit>` | — | Force output in this unit; error if dimension incompatible |

**Unit syntax in expressions:**

```
5 km           # 5 kilometres
9.8 m/s^2      # acceleration
70 kg          # mass
5 m * 3 m      # area: 15 m^2
```

| Stream | Content |
|--------|---------|
| stdout | Result with unit (or plain number if dimensionless) |
| stderr | `error: <description>` on dimension mismatch or unknown unit |
| exit 0 | Success |
| exit 1 | Any error |

---

## Constraints

- All prior constraints remain in force
- Unit-carrying values and dimensionless values are interoperable: multiplying
  a dimensionless number by a unit value yields a unit value
- Trigonometric functions (`sin`, `cos`, `tan`) require dimensionless arguments;
  angles must be in radians (no degree unit in this version)
- The unit database is a data file, not hardcoded; adding a new unit requires
  adding one entry
- The unit system must extend to derived units defined by the user (`def knot = 0.514444 m/s`)
  in a future version without changing the unit evaluation mechanism

## Key Unknowns

1. **[P1]** Does adding a unit type to the value representation — so every value
   carries an optional dimension alongside its magnitude — require changes to all
   existing functions and operators, or can unit propagation be implemented as a
   wrapper layer that intercepts operations without modifying the core evaluator?
   The answer determines the scope of changes to the v0.1.x evaluator and whether
   unit support is an additive layer or a pervasive rewrite.

## Non-Goals

- Degree unit for trig functions
- User-defined units in this version
- Currency as a unit
- Uncertainty/error propagation
- Windows support
