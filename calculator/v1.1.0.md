# Calculator CLI v1.1.x — Conditionals

**Theme:** Expressions that branch. Piecewise functions become writable.

## Overview

Adds conditional expressions and comparison operators to the language. A user
can write `if x > 0 then x else -x` and the graphing window immediately shows
the V-shape of absolute value. Without conditionals, piecewise functions are
inexpressible and a significant class of interesting curves is unavailable.
This is the first version that makes the language Turing-relevant: the
evaluator must decide, not just compute.

---

## Success Criteria

- [ ] `calc 'if 3 > 2 then 1 else 0'` prints `1` to stdout and exits 0
- [ ] `calc 'if 3 < 2 then 1 else 0'` prints `0` to stdout and exits 0
- [ ] `calc 'x = -5; if x > 0 then x else -x'` prints `5` and exits 0
- [ ] `calc 'if x > 0 then x else if x < 0 then -x else 0'` entered in the
  graphing window with `x` as the free variable plots the absolute value function
- [ ] `calc '3 > 2'` prints `1` to stdout and exits 0
- [ ] `calc '3 = 3'` prints `1` to stdout and exits 0
- [ ] `calc '3 != 3'` prints `0` to stdout and exits 0
- [ ] `calc 'if 1 then 5 else 6'` prints `5` — any nonzero value is truthy
- [ ] `calc 'if 0 then 5 else 6'` prints `6` — zero is falsy
- [ ] `calc 'if x > 0 then'` prints `error: expected then-branch` to stderr
  and exits 1
- [ ] All prior success criteria still pass
- [ ] `make test` passes clean on macOS and Linux

---

## Scope

**Included:**

- Conditional expression: `if <cond> then <expr> else <expr>`
- Nested conditionals in either branch
- Comparison operators: `>`, `<`, `>=`, `<=`, `=` (equality test), `!=`
- Boolean semantics: nonzero = true, zero = false; comparisons return `1` or `0`
- Logical operators: `and`, `or`, `not`
- Conditionals usable in function bodies, variable assignments, and plot expressions
- All prior language features unchanged

---

## Failure Modes

**Parse errors:**
- `if` without `then` → `error: expected 'then' after condition` to stderr, exit 1
- `if ... then` without `else` → `error: expected 'else' branch` to stderr, exit 1

**Evaluation:**
- Condition evaluates to a domain error → `error: domain error in condition` to stderr, exit 1

---

## Interface Contracts

**CLI:** Unchanged. `calc '<expression>'` where expression now includes `if/then/else`.

**Grammar extension (informal):**

```
expression = 'if' expression 'then' expression 'else' expression
           | expression ('>' | '<' | '>=' | '<=' | '=' | '!=') expression
           | expression ('and' | 'or') expression
           | 'not' expression
           | <prior grammar>
```

**Operator precedence (lowest to highest):**

```
or
and
not
> < >= <= = !=
+ -
* /
unary -
function call, exponentiation
```

| Stream | Content |
|--------|---------|
| stdout | Result or `1`/`0` for comparisons |
| stderr | `error: <description>` on failure |
| exit 0 | Success |
| exit 1 | Any error |

---

## Constraints

- All prior constraints remain in force
- `=` in conditional context is equality test; `=` in statement context is
  assignment — the parser must disambiguate by position
- The conditional expression must be implementable without a new AST node
  type that conflicts with user-defined function bodies in v0.4.x
- The conditional must compose naturally with implicit curve syntax in v0.9.x:
  `if x > 0 then x^2 else -x^2 = y` must parse and render correctly

---

## Key Unknowns

1. **[P1]** Does the `=` token's dual role — assignment in statement position,
   equality test in expression position — require a context-sensitive parser
   that can distinguish the two without ambiguity across all expression forms
   (including function bodies and implicit curve syntax from v0.9.x), or does
   resolving this cleanly require a dedicated equality operator token (e.g. `==`)?
   The answer determines whether `=` remains the equality operator or is split.

---

## Non-Goals

- Loops or iteration — v1.2.x summation covers the common cases
- Short-circuit evaluation guarantees (implementation detail)
- Pattern matching
- Windows support
