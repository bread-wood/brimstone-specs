# Calculator CLI v1.11.x — LaTeX

**Theme:** One command to typeset. Math goes into documents.

## Overview

Adds a `latex` subcommand that converts a calculator expression to a
well-formatted LaTeX string. `calc latex 'x^2/4 + y^2/9 = 1'` produces
`\frac{x^2}{4} + \frac{y^2}{9} = 1`. `calc latex 'sum(k, 1, n, k^2)'`
produces `\sum_{k=1}^{n} k^{2}`. Targets academics and engineers who write
math in LaTeX and want to convert their working calculations into document-ready
notation without transcribing by hand. The output is copy-paste ready.

- [ ] `calc latex 'pi'` prints `\pi` and exits 0
- [ ] `calc latex 'e'` prints `e` and exits 0
- [ ] `calc latex 'sum(k, 1, n, k^2)'` prints `\sum_{k=1}^{n} k^{2}` and exits 0

## Scope

**Included:**

- `calc latex '<expression>'` subcommand
- LaTeX rendering for all language features through v1.5.x:
  arithmetic, functions, constants, variables, user-defined functions,
  conditionals (`\begin{cases}`), summation, products, integrals, complex numbers
- Named constants rendered as LaTeX symbols: `pi` → `\pi`, `e` → `e`, `i` → `i`
- Named functions rendered as LaTeX operators: `sin` → `\sin`, `log` → `\ln`, etc.
- Fractions: any division of non-trivial numerator or denominator → `\frac{}{}`
- Square roots: `sqrt(expr)` → `\sqrt{expr}`
- Exponentiation: `x^expr` → `x^{expr}`
- Absolute value: `abs(x)` → `\left|x\right|`
- Piecewise: `if/then/else` → `\begin{cases}...\end{cases}`

---

## Failure Modes

**Parse:**
- Malformed expression → `error: <description>` to stderr, exit 1 (same as `calc '<expr>'`)

**Rendering:**
- Expression uses a language feature with no LaTeX mapping → rendered using a
  text fallback: `\text{<feature>}(<args>)`; not an error; a warning is emitted
  to stderr

---

## Interface Contracts

**CLI:**

```
calc latex '<expression>' [--display]
```

| Flag | Default | Purpose |
|------|---------|---------|
| `--display` | off | Wrap output in `\[...\]` (display math) instead of inline |

**Symbol mappings (selected):**

| Expression | LaTeX output |
|------------|-------------|
| `pi` | `\pi` |
| `e` | `e` |
| `i` | `i` |
| `sin(x)` | `\sin(x)` |
| `log(x)` | `\ln(x)` |
| `abs(x)` | `\left\|x\right\|` |
| `conj(z)` | `\overline{z}` |
| `sqrt(x)` | `\sqrt{x}` |
| `a / b` | `\frac{a}{b}` |
| `integral(f, a, b)` | `\int_{a}^{b} f \, dx` |
| `sum(k, a, b, f)` | `\sum_{k=a}^{b} f` |
| `product(k, a, b, f)` | `\prod_{k=a}^{b} f` |

| Stream | Content |
|--------|---------|
| stdout | LaTeX string |
| stderr | `error: <description>` on parse failure; warnings for unmapped features |
| exit 0 | Success |
| exit 1 | Parse error |

---

## Constraints

- All prior constraints remain in force
- LaTeX output must be valid, compilable LaTeX when pasted into a standard
  math environment (`\[...\]` or `$...$`)
- No external LaTeX dependencies; output is a plain string
- The LaTeX renderer operates on the AST; it does not evaluate the expression

---

## Key Unknowns

1. **[P1]** Does the existing AST — designed for evaluation — contain sufficient
   structural information to produce well-formatted LaTeX (e.g., knowing whether
   a division should render as `\frac{}{}` or as `a/b` depending on complexity),
   or does LaTeX generation require a separate pretty-printer representation that
   carries formatting intent the evaluator AST discards? The answer determines
   whether the LaTeX renderer is a straightforward AST visitor or requires an
   intermediate representation.

---

## Non-Goals

- Rendering LaTeX back to an expression (reverse direction)
- MathML output
- Rendering to PDF
- User-configurable symbol mappings
- Windows support
