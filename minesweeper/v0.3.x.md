# Minesweeper v0.3.x — No-Guess

**Theme:** Every move is provable. The coin flip is abolished.

## Overview

Generates boards that the v0.2.x solver can always complete without guessing.
A no-guess board guarantees that at every point during play — after the first
reveal — at least one provably safe cell exists. Boards that don't satisfy
this property are rejected and regenerated. This version makes Minesweeper
a game of pure logic: a loss means a mistake, not bad luck.

---

## Success Criteria

- [ ] `minesweeper --no-guess` starts a game on a board the solver can complete
  without any guesses; the guarantee holds across 1000 independently generated
  boards for beginner difficulty
- [ ] `minesweeper --no-guess --intermediate` generates a no-guess intermediate
  board; median generation time under 5 seconds on commodity hardware
- [ ] A no-guess board displayed with `--auto-solve` always produces
  `"guesses": 0` in the result report
- [ ] `minesweeper --no-guess --expert` produces a no-guess expert board or
  prints `warning: no-guess expert boards are rare; generation may take longer`
  and continues trying
- [ ] `minesweeper --no-guess` accepts the same `--width`, `--height`, `--mines`
  flags as v0.1.x; the guarantee applies to any configuration
- [ ] `make test` passes clean

---

## Scope

**Included:**

- Board generation loop: generate a candidate board, run the solver from a
  chosen first reveal, check whether any guess is required; if so, regenerate
- First reveal position selection: the position is chosen to maximise the
  initial flood-fill area, maximising solver information from move one
- Generation timeout: if no valid board is found within `MINESWEEPER_NOGUE_TIMEOUT_SECONDS`,
  exit 1 with a clear message rather than looping indefinitely
- `--no-guess` flag usable with all existing board size flags
- `--seed` in `--no-guess` mode: the seed applies to the generation search, not
  a specific board layout; the same seed produces the same valid board

---

## Failure Modes

**Generation:**
- No valid board found within timeout → `error: could not generate a no-guess board within <N>s; try a lower mine count` to stderr, exit 1
- Configuration makes no-guess generation theoretically impossible (e.g. mines > 50% of cells on large boards) → emit warning before attempting; continue

**Solver dependency:**
- Solver marks a cell safe that is actually a mine (should never occur; indicates solver bug) → generation loop rejects the board and logs a critical warning rather than panicking

## Interface Contracts

**CLI additions:**

```
minesweeper --no-guess [all existing flags]
```

| Flag | Default | Purpose |
|------|---------|---------|
| `--no-guess` | off | Guarantee the board requires no guesses to complete |

**Environment variables:**

| Variable | Default | Purpose |
|----------|---------|---------|
| `MINESWEEPER_NOGUES_TIMEOUT_SECONDS` | `30` | Max generation time before giving up |

| Stream | Content |
|--------|---------|
| stdout | Normal game output |
| stderr | `error:` or `warning:` on generation issues |
| exit 0 | Game started and played normally |
| exit 1 | Generation timeout or invalid configuration |

---

## Constraints

- Depends on v0.2.x solver; no-guess generation is defined as "the solver can
  complete this board with zero guesses"
- Generation must not modify the board module; boards are generated as candidate
  layouts and validated externally
- The generation algorithm must not assume a specific first-reveal position;
  the first reveal is chosen by the generation algorithm, not the player, for
  validation purposes — the player still makes the first move in the actual game

---

## Key Unknowns

1. **[P1]** What is the practical reject rate for no-guess board generation at
   intermediate and expert difficulty — and is a simple generate-and-reject loop
   fast enough, or does the reject rate require a smarter generation algorithm
   (e.g. placing mines incrementally while checking solver feasibility) to
   achieve acceptable latency? The answer determines the generation architecture
   and whether a timeout warning is a rare edge case or a common occurrence at
   expert difficulty.

2. **[P2]** Does the no-guess guarantee depend on the first reveal position —
   and if so, should the generation algorithm fix the first reveal (chosen for
   maximum flood fill) or verify the guarantee holds for any first reveal
   position? The answer determines both the generation contract and whether the
   player's choice of first move can invalidate the no-guess guarantee.

---

## Non-Goals

- Generating the hardest possible no-guess board (difficulty rating — future version)
- Proving that a no-guess board is unique or minimal
- No-guess guarantee for boards where the player can choose any first reveal
  (fixed first reveal only in this version)
- Windows support
