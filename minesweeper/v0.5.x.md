# Minesweeper v0.5.x — Variants

**Theme:** The same rules, stranger boards.

## Overview

Extends the board module to support non-rectangular topologies. The game logic
— reveal, flag, flood fill, win/loss — is unchanged; only the neighbor
relationship changes. A hexagonal board gives each cell six neighbors instead
of eight; a toroidal board wraps edges so the leftmost and rightmost columns
are adjacent. This version validates that the v0.1.x board abstraction is
genuinely topology-agnostic and stress-tests the solver against new geometries.

---

## Success Criteria

- [ ] `minesweeper --topology hex` starts a playable hexagonal minesweeper game;
  each cell has at most 6 neighbors; flood fill, win, and loss all behave correctly
- [ ] `minesweeper --topology torus` starts a game where edges wrap: the top row
  is adjacent to the bottom row and the left column is adjacent to the right column
- [ ] `minesweeper --topology hex --auto-solve` runs the solver on a hex board;
  the solver correctly identifies safe cells given the six-neighbor constraint system
- [ ] `minesweeper --topology torus --no-guess` generates a no-guess toroidal board
- [ ] The hex board renders with offset rows (standard hex grid visual convention)
  that clearly distinguish it from a square grid
- [ ] All v0.1.x through v0.4.x flags (presets, stats, replay) work with both
  new topologies
- [ ] A replay saved from a hex game replays correctly and identifies the topology
  in its record
- [ ] `minesweeper --topology unknown` prints `error: unknown topology: unknown` to
  stderr and exits 1
- [ ] `make test` passes clean

---

## Scope

**Included:**

- Topology abstraction: a pluggable neighbor function; `--topology` selects the
  implementation; square is the existing default
- Hex topology: offset-row hexagonal grid; 6 neighbors per interior cell
- Torus topology: rectangular grid with wrap-around edges; 8 neighbors per cell
  (including wrap-around neighbors)
- Renderer extension: hex boards rendered with offset rows; torus boards rendered
  identically to square boards (wrap is invisible in rendering)
- Solver: works unchanged (operates on neighbor sets; topology-agnostic)
- No-guess generator: works unchanged (calls solver; topology-agnostic)
- Stats and replay: topology stored in game record; displayed in stats breakdown

---

## Failure Modes

**Configuration:**
- Unknown `--topology` value → `error: unknown topology: <value>; valid: square, hex, torus` to stderr, exit 1
- Hex board with odd dimensions that produce degenerate neighbor counts →
  handled gracefully; cells at corners/edges have fewer than 6 neighbors; not an error

**Rendering:**
- Terminal too narrow for offset hex rendering → `error: terminal too small for hex board at this width` to stderr, exit 1

## Constraints

- Depends on v0.1.x board module; topology is injected as a neighbor function,
  not a new board type — the board stores cells; the topology determines adjacency
- Depends on v0.2.x solver and v0.3.x no-guess generator; both operate through
  the neighbor abstraction and require no topology-specific changes
- Adding a new topology must require only implementing the neighbor function;
  no changes to game logic, solver, or generator

---

## Key Unknowns

1. **[P1]** Does the v0.1.x board module's data structure and interface support
   topology injection cleanly — specifically, can the neighbor relationship be
   swapped out at construction time without touching flood fill, solver, or
   renderer code — or does the square-grid assumption permeate enough of the
   implementation that topology abstraction requires a significant refactor?
   The answer determines whether v0.1.x's extensibility constraint was satisfied.

---

## Non-Goals

- 3D minesweeper (volumetric grid)
- Spherical or cylindrical topologies
- Irregular or procedurally generated board shapes
- Topology-specific scoring or difficulty adjustments
- Windows support
