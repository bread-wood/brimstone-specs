# breadmin-jobman v0.1.0 — Knowledge Wire

**Theme:** Register the knowledge graph as breadmin jobs and expose it for deliberation
context injection.

---

## Overview

The `breadmin-knowledge` package is built: `KnowledgeStore`, `ContextAssembler`,
`LibraryReader`, `BootstrapResult`, `ChangelogWatcher`, and `EmbeddingIndex` are all
implemented and tested. What doesn't exist yet is:

1. These components wired as scheduled breadmin jobs
2. A REST/CLI interface for querying the knowledge graph
3. Knowledge context delivered into moot deliberation briefings before agents run

This milestone wires the knowledge package into the breadmin scheduler and closes the
loop with moot. No new knowledge package code is needed; the work is job definitions,
schedule config, and the deliberation context injection protocol.

---

## Success Criteria

- [ ] A `knowledge-bootstrap` job runs on demand (or at startup) and populates the
  knowledge registry by reading from moot's `library_summaries` SQLite view (stable
  read contract); re-runs are idempotent
- [ ] A `knowledge-watch` job runs on a schedule (e.g., every 15 minutes) and picks up
  new library changelog entries via `ChangelogWatcher`
- [ ] A `knowledge-correlate` job runs on a schedule (e.g., daily) and runs
  `EmbeddingIndex.suggest_connections()` to discover and write high-similarity entity
  pairs as connection candidates
- [ ] The knowledge graph is queryable via REST API (`GET /knowledge/entities`,
  `GET /knowledge/context?seeds=...`) and the breadmin CLI (`breadmin knowledge query ...`)
- [ ] When a moot deliberation is triggered, a pre-deliberation job assembles a
  `ContextBrief` for the topic and injects it into the deliberation input
- [ ] `make test-knowledge && make lint` pass on mainline

---

## Scope

**Included:**

- `knowledge-bootstrap` job handler (wraps `KnowledgeStore` + `bootstrap.py`)
- `knowledge-watch` job handler (wraps `ChangelogWatcher`)
- `knowledge-correlate` job handler (wraps `EmbeddingIndex.suggest_connections()`)
- Schedule definitions for watch (interval) and correlate (daily cron) in a config file
- REST API endpoints under `/knowledge/` (entities, context assembly)
- CLI commands under `breadmin knowledge` (query, context, bootstrap)
- Deliberation context injection: a job or hook that runs `ContextAssembler.assemble()`
  before moot deliberation starts and passes the brief as additional context

**Explicitly excluded:**

- Changes to the knowledge package itself (it's built)
- New entity types or connection types
- Moot deliberation changes beyond accepting the context brief input field
- Embedding model changes (current pluggable async function interface stays)

---

## Failure Modes

**Job failures:**
- `knowledge-watch` fails because moot.db is unavailable (moot not running) → log
  warning; job retries per standard breadmin retry policy; no alert unless consecutive
  failures exceed threshold
- `knowledge-correlate` fails because embedding function is not configured → log error;
  skip gracefully; do not retry endlessly; alert operator once
- `knowledge-bootstrap` finds moot library tables empty → complete successfully with
  0 entities created; log informational message

**Context injection failures:**
- Context assembly exceeds token budget → ContextAssembler truncates by design; brief
  is passed as-is with pruning stats logged
- Context assembly fails entirely → deliberation proceeds without knowledge context;
  log warning; do not block the deliberation
- Moot is unavailable when context injection is attempted → deliberation proceeds without
  context; log warning

## Interface Contracts

**REST API (new endpoints under breadmin server):**

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/knowledge/entities` | GET | List entities with optional `?domain=` and `?type=` filters |
| `/knowledge/entities/<id>` | GET | Single entity with connections |
| `/knowledge/context` | GET | Assemble context brief; `?seeds=<labels>&domains=<d>&budget=<N>` |
| `/knowledge/bootstrap` | POST | Trigger a bootstrap run (async, returns job ID) |

**CLI:**

```
breadmin knowledge bootstrap            # Run bootstrap
breadmin knowledge query <label>        # Look up entity by label
breadmin knowledge context <seeds...>   # Assemble context brief for seeds
```

**Deliberation context injection schema:**

```json
{
  "context_brief": {
    "sections": [
      {
        "entity_label": "<string>",
        "domain": "<string>",
        "summary": "<string>",
        "connections": ["<label> (<relationship>, strength=<float>)"]
      }
    ],
    "token_count": 1200,
    "pruned_sections": 3
  }
}
```

## Constraints

- Knowledge bootstrap and watch jobs are `mod:knowledge` scope; REST API and CLI
  additions are `mod:core` scope
- Knowledge store path configured via env var, not hardcoded
- `LibraryReader` reads from moot's `library_summaries` SQLite view — never directly
  from `library_*` tables; the view is the stable contract maintained by moot
- Context injection is **push** — jobman assembles a `ContextBrief` and appends it to
  the deliberation request before calling moot's API; moot does not call jobman back for
  context; moot's deliberation input gains an optional `context_brief` field
- All new code is type-annotated, async-compatible, and passes `ruff check`
- No changes to moot code beyond accepting the optional `context_brief` field in
  deliberation input

## Key Unknowns

~~1. **[P1]** Push vs pull for context injection?~~ **Resolved: push.** Jobman assembles
   the `ContextBrief` and appends it to the deliberation request before calling moot's
   API. Moot does not call jobman. This keeps moot ignorant of the coordination layer —
   the dependency arrow runs moot → jobman at the package level; reversing it via a pull
   call would create a runtime circular dependency (jobman schedules moot; moot calls
   jobman). Design stage should proceed with push.

1. **[P2]** Which embedding provider should be used for `EmbeddingIndex`? The package
   has a pluggable async function interface. `text-embedding-3-small` (OpenAI) is
   referenced in the HLD. This must be wired to breadmin-llm's embedding support.
   Research stage should confirm whether breadmin-llm exposes an embeddings method.

## Non-Goals

- Changes to `breadmin-knowledge` internals
- New entity types or connection types
- Full-text search (separate from embedding similarity)
- Multi-user knowledge graph isolation
