# breadmin-jobman v0.2.0 — Verdict Bridge

**Theme:** Financial council integration — portfolio state delivery to moot and verdict
routing back to breadwinner.

---

## Overview

Moot's trading-floor council (v0.6.0) needs two things from breadmin-jobman:

1. **Portfolio state delivery**: a scheduled job that extracts breadwinner's current
   options and crypto positions and delivers them to moot so the financial council has
   live portfolio context before deliberating.

2. **Verdict routing**: a job (or webhook receiver) that picks up financial council
   verdicts from moot and delivers actionable signals to breadwinner.

breadmin-jobman is the coordination layer between the two isolated product repos. Neither
moot nor breadwinner imports the other — all state flows through breadmin-jobman jobs and
the defined contract schemas.

**Prerequisite:** moot v0.6.0-trading-floor must be in progress; the verdict handler
interface (moot side) must be defined before the jobman bridge is built.

---

## Success Criteria

- [ ] A `breadwinner-state-snapshot` job runs on a schedule (e.g., every 30 minutes
  during market hours) and writes current options positions and crypto holdings to a
  portfolio state store that moot can read
- [ ] The portfolio state store schema matches the contract defined in moot
  v0.6.0-trading-floor (versioned, schema_version field)
- [ ] A `moot-verdict-receiver` job runs on a schedule or webhook trigger and reads
  pending financial council verdicts from moot; for each verdict, it dispatches a
  structured signal to breadwinner via the breadwinner job transport
- [ ] A breadwinner job handler (`breadwinner-council-signal`) receives the verdict
  signal and logs it to breadwinner's audit trail (execution is breadwinner's concern)
- [ ] Deep-pass trigger criteria are evaluated by the `breadwinner-state-snapshot` job:
  if the pending decision exceeds thresholds, the job flags it for full council
  deliberation before breadwinner acts
- [ ] No direct imports between moot and breadwinner anywhere in breadmin-jobman source
- [ ] `make test-core && make lint` pass on mainline

---

## Scope

**Included:**

- `breadwinner-state-snapshot` job handler: calls breadwinner CLI to extract options
  and crypto positions; writes to portfolio state store (SQLite in breadmin data dir)
- Portfolio state store: SQLite table with versioned schema; readable by moot via the
  contract defined in moot v0.6.0-trading-floor Key Unknown #1
- `moot-verdict-receiver` job handler: polls moot's verdict queue (REST API or DB read);
  dispatches signals to breadwinner via breadmin HTTP or CLI transport
- `breadwinner-council-signal` CLI handler (in breadwinner's handler registry): receives
  verdict signal, logs to audit trail
- Deep-pass evaluation logic: compares pending decision size/type against configured
  thresholds; flags decisions requiring council deliberation before execution

**Explicitly excluded:**

- Financial council implementation (moot v0.6.0-trading-floor)
- Breadwinner order execution changes
- Real-time market data feeds
- Verdict storage in moot (moot's concern)

---

## Failure Modes

**State snapshot failures:**
- Breadwinner CLI call fails → log error; skip snapshot for this cycle; do not block
  breadwinner execution
- Portfolio state store write fails → log error; retry once; if still failing, alert
  operator; do not lose the snapshot silently

**Verdict routing failures:**
- Moot verdict queue unavailable → log warning; skip this poll cycle; retry next cycle
- Breadwinner handler unreachable → log error; dead-letter the verdict for manual review;
  do not retry indefinitely
- Contract version mismatch between jobman and breadwinner → log error with version info;
  refuse to deliver rather than misdelivering

**Deep-pass failures:**
- Threshold config missing or malformed → raise at startup; do not proceed with
  unconfigured thresholds

## Interface Contracts

**Portfolio state store schema (written by jobman, read by moot):**

```json
{
  "schema_version": "1",
  "as_of": "<iso8601>",
  "options_positions": [
    {
      "ticker": "<string>",
      "type": "CSP | CC | <other>",
      "strike": 0.0,
      "expiry": "<YYYY-MM-DD>",
      "delta": 0.0,
      "pnl_pct": 0.0
    }
  ],
  "crypto_holdings": [
    { "coin": "<string>", "quantity": 0.0, "value_usd": 0.0 }
  ]
}
```

**Verdict signal (jobman → breadwinner):**

Exact schema TBD pending moot v0.6.0-trading-floor Key Unknown #1 definition. Minimum:

```json
{
  "schema_version": "1",
  "council_id": "financial",
  "outcome": "approve | reject | defer",
  "rationale": "<string>",
  "confidence": "high | medium | low",
  "deep_pass_used": true,
  "subject": "<string describing the decision>",
  "issued_at": "<iso8601>"
}
```

---

## Constraints

- No direct imports between moot and breadwinner in any breadmin-jobman code
- Portfolio state store schema is versioned; jobman must not write a schema_version it
  hasn't tested against moot's reader
- Deep-pass thresholds are configured via env or YAML config, not hardcoded
- All handlers are `mod:core` scope (job infrastructure); portfolio state store is also
  `mod:core`

---

## Key Unknowns

1. **[P1]** What is the exact verdict handler interface from moot's side? Moot
   v0.6.0-trading-floor Key Unknown #1 must be resolved first. The jobman side can only
   be designed once moot defines what fields a financial verdict contains and how it
   exposes them (REST endpoint, DB table, or event bus). The shared research output
   is committed to `breadmin-jobman/docs/contracts/verdict-signal-v1.json` — a design
   stage deliverable that all three repos (moot, jobman, breadwinner) consume.

2. **[P1]** How does breadwinner expose its current options and crypto positions for
   extraction? Does breadwinner have a CLI command or REST endpoint that jobman can call,
   or does jobman need to read breadwinner's SQLite DB directly? Direct DB reads couple
   the schema; a CLI/API call is cleaner. Research stage must answer this.

---

## Non-Goals

- Financial council implementation (moot scope)
- Breadwinner order execution
- Real-time streaming (all polling-based in this milestone)
- Multi-council verdict routing (financial council only)
