# breadmin-jobman v0.4.0 — Signal Grid

**Theme:** Wire breadmin-youtube channel scanning as a scheduled breadmin job, and
extend the signals store to track multi-source video queue state alongside existing
Reddit and portfolio signals.

---

## Overview

`breadmin-youtube v0.1.0-prospector` delivers a standalone `breadmin-youtube scan`
command and queue store. What it does not deliver is scheduled, unattended execution —
that is jobman's role. This milestone registers a `youtube-scan` job in breadmin's
scheduler (same pattern as `reddit-sync` from v0.3.0-signals-pipeline), adds
queue-state visibility to the signals store, and wires a `youtube-queue-drain` job
that automatically calls `bw ingest` for each pending video on a configured schedule.

At completion, the full loop runs unattended: channels are scanned on a cron schedule
→ qualifying videos land in the queue → `bw ingest` is called for each → validated
tickers appear in Alpaca watchlists — without any manual intervention beyond initial
channel configuration.

**Prerequisite:** breadmin-youtube v0.1.0-prospector must complete (queue store schema
and `scan` / `drain` CLI must be stable); breadwinner v0.5.0-ticker-tape must complete
(`bw ingest --from-queue` must be available).

---

## Success Criteria

- [ ] A `youtube-scan` job is registered in breadmin's scheduler; it calls
  `breadmin-youtube scan` on a configurable cron schedule (default: every 4 hours);
  success/failure is logged to the signals store with a `source: youtube` entry
- [ ] A `youtube-queue-drain` job runs after each successful `youtube-scan` and calls
  `bw ingest` for each `pending` queue entry; drained entries are marked `done` or
  `error` in the queue store
- [ ] When `youtube-scan` fails (quota exceeded, yt-dlp blocked), the job logs a
  `scan_failed` event to the signals store and backs off by 2× on the next scheduled
  run (max backoff: 24 hours)
- [ ] When `youtube-queue-drain` encounters a `bw ingest` error for a video, that
  entry is marked `error` in the queue store; the drain continues to the next entry;
  no partial-drain aborts
- [ ] The signals store freshness record for `youtube` is updated after each scan;
  moot's context layer can detect staleness using the same schema as `reddit` and
  `breadwinner`
- [ ] Queue depth (pending count) is visible in `breadmin status` output alongside
  Reddit and portfolio signal freshness
- [ ] `make test-youtube && make test-core && make lint` pass on mainline

---

## Scope

**Included:**

- `youtube-scan` job handler: thin wrapper around `breadmin-youtube scan`; writes
  result to signals store; implements exponential backoff on quota/block failures
- `youtube-queue-drain` job handler: iterates the queue store for `pending` entries;
  calls `bw ingest --source=youtube --url={url} --watchlist={channel_slug}` for each;
  updates queue entry status; writes drain summary to signals store
- Schedule config for both jobs (YAML, same format as v0.3.0)
- Signals store extension: `youtube` source row in the freshness table; queue depth
  metric written after each scan
- `breadmin status` output extension: youtube scan freshness + pending queue depth
- Alerting: Telegram notification (if configured) when `youtube-scan` fails 3
  consecutive times or queue depth exceeds a configurable threshold (default: 50)

**Explicitly excluded:**

- Changes to breadmin-youtube package internals (that repo's scope)
- Changes to `bw ingest` CLI (breadwinner v0.5.0 scope)
- Automatic Alpaca trade execution triggered by ingestion results
- Queue review UI (queue is inspectable via `breadmin-youtube queue list`)

---

## Failure Modes

**youtube-scan:**
- `breadmin-youtube scan` exits non-zero (quota, block, config error) → log failure
  to signals store; apply exponential backoff; alert after 3 consecutive failures
- Scan succeeds but discovers 0 new videos → normal; update `last_scanned_at`; no alert

**youtube-queue-drain:**
- `bw ingest` exits non-zero for a video → mark entry `error`; log error detail;
  continue drain; do not retry in same drain cycle
- `bw ingest` exits non-zero for all videos in a drain cycle → log warning; alert
  operator (likely a breadwinner or Alpaca API issue)
- Queue store unavailable → abort drain; log error; do not mark any entries

## Interface Contracts

**Signals store freshness record (youtube source — same schema as v0.3.0):**

```json
{
  "source": "youtube",
  "last_updated": "<iso8601>",
  "is_stale": false,
  "stale_threshold_hours": 8,
  "extra": {
    "queue_depth_pending": 0,
    "last_scan_video_count": 0,
    "backoff_until": null
  }
}
```

**Schedule config additions (YAML):**

```yaml
youtube-scan:
  schedule: "0 */4 * * *"         # Every 4 hours
  active_hours_only: false
  handler: breadmin_youtube.cli:scan
  args: []
  max_retries: 1
  alert_after_failures: 3
  backoff_multiplier: 2.0
  backoff_max_hours: 24

youtube-queue-drain:
  schedule: "30 */4 * * *"        # 30 min after scan
  active_hours_only: false
  handler: breadmin_youtube.cli:queue_drain
  args: ["--exec", "bw ingest --source=youtube --url={url} --watchlist={channel_slug}"]
  max_retries: 0
```

---

## Constraints

- `youtube-queue-drain` must never execute trades or call anything beyond `bw ingest`
- Backoff state is persisted in the signals store — not in memory — so it survives
  scheduler restarts
- No direct imports from breadmin-youtube package internals; job handlers invoke CLI
  subprocess (same pattern as `reddit-sync`)
- Queue drain is idempotent: re-running the drain on already-`done` entries is a no-op

---

## Key Unknowns

1. **[P2]** Should `youtube-queue-drain` run `bw ingest` as a subprocess or invoke
   breadwinner as a library? Subprocess is consistent with how jobman runs other
   breadwinner commands, but introduces process-spawn overhead per video. If queue depth
   is routinely > 20 entries, a batch mode may be preferable. Research stage should
   check whether breadwinner v0.5.0 exposes a batch ingest path.

---

## Non-Goals

- YouTube content streaming or real-time polling (scan cadence is cron-based only)
- Auto-trading from ingested watchlists
- Queue drain UI beyond CLI
- Multi-platform signal sources beyond YouTube (separate future milestones)
