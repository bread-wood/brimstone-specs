# moot v0.4.0 ‚ÄĒ Council Spine

**Theme:** Establish the `CouncilPlugin` protocol and restructure the package around it.

---

## Overview

Moot has grown to ~15,600 lines with council logic (Prism, Inquiry, future RGB) tangled
into shared infrastructure (identity, debate, routing, Mirror). This milestone defines a
`CouncilPlugin` protocol ‚ÄĒ the interface contract all councils must satisfy ‚ÄĒ restructures
`src/moot/` so each council is an isolated plugin, removes stale OAuth/Fitbit stubs and
the Rampart agent, and refines the CLAUDE.md module table so brimstone can dispatch
isolated agents per module.

No new council is built in this milestone. Existing councils (Prism, Inquiry) are migrated
to the new interface.

## Scope

**Included:**

- `CouncilPlugin` protocol definition in `src/moot/councils/base.py`
- Migration of Prism and Inquiry to implement `CouncilPlugin`
- Removal of Gmail OAuth stub (`src/moot/integrations/gmail.py`, `src/moot/email/`)
- Removal of Calendar OAuth stub (`src/moot/integrations/calendar.py`)
- Removal of Fitbit integration (`src/moot/integrations/fitbit.py`)
- Removal of Rampart agent references
- Merge of `src/moot/procedures/` into `src/moot/councils/`
- CLAUDE.md module table update (11 fine-grained labels)
- Router evaluation: determine whether GPT-5 Nano router stays or moves to Claude
- Migration of moot's scheduling layer to breadmin-jobman: `src/moot/scheduler.py` is
  deleted; the morning, evening, and weekly cycles become breadmin-jobman job handlers
  (`moot-heartbeat`, `moot-morning-cycle`, `moot-evening-cycle`, `moot-weekly-cycle`);
  the standalone CLI entry point for `src/moot/heartbeat.py` is removed (logic stays in
  moot, invocation ownership moves to jobman)
- Removal of moot's budget/cost tracking system: `check_budget()` and `estimate_cost()`
  in `heartbeat.py` are deleted; breadmin-llm's `token_usage` table is the single source
  of truth for LLM cost tracking
- Create a stable `library_summaries` SQLite view in moot's DB schema; this view is the
  read contract for jobman's `LibraryReader` ‚ÄĒ replacing raw `library_*` table coupling

**Explicitly excluded:**

- New council implementation (RGB, Trading Floor)
- Personality seed changes
- Identity model changes
- Any new integration modules

## Failure Modes

**Migration failures:**
- Prism or Inquiry fails to satisfy `CouncilPlugin` ‚Üí block merge; do not ship partial protocol
  compliance
- `procedures/` merge breaks routing behavior ‚Üí write regression tests for routing paths
  before deleting the directory
- Stale import in an unexpected file after stub removal ‚Üí ruff and mypy will catch; fix
  before merge

**Structural failures:**
- `CouncilPlugin` interface too broad ‚Üí councils pass the check but the interface is
  meaningless; Key Unknown #1 research must answer this before design begins
- Module table granularity wrong ‚Üí revise after impl if agent isolation proves
  too coarse or too fine in practice

## Interface Contracts

**`CouncilPlugin` protocol (to be fully specified in design stage):**

Minimum required surface ‚ÄĒ exact signatures determined by Key Unknown #1 research:

| Method / attribute | Direction | Notes |
|--------------------|-----------|-------|
| `council_id: str` | property | Unique identifier (e.g. `"prism"`, `"inquiry"`, `"moot"`) |
| `run(topic: Topic) -> CouncilOutput` | async method | Primary entry point |
| Lifecycle hooks (TBD) | TBD | `init`, `session_start`, `session_end` as needed |

## Constraints

- `CouncilPlugin` must be a `typing.Protocol` (structural subtyping) ‚ÄĒ no forced
  inheritance for existing councils
- Router decision (GPT-5 Nano vs Claude) must be made before impl closes ‚ÄĒ flagged
  as Key Unknown #2
- All changes land on mainline via PRs; no direct commits

---

## Key Unknowns

1. **[P1]** What should the `CouncilPlugin` interface look like? Specifically: what inputs
   and outputs does the system layer pass to a council, what lifecycle hooks does it expose
   (`init`, `session_start`, `run`, `session_end`), and what does it return? The answer
   determines whether the protocol is a thin wrapper or a richer coordination surface.
   Research stage must answer this before design begins.

2. **[P1]** Should the router stay on GPT-5 Nano (breadmin-llm) or be replaced with a
   Claude call? GPT-5 Nano is external, small, and fast; Claude is internal to our stack
   and easier to prompt-tune but adds latency and cost. The answer affects the
   `mod:router` cleanup scope and whether any router tests need updating.
