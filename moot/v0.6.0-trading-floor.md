# moot v0.6.0 — Trading Floor

**Theme:** Financial deliberation council for breadwinner decisions.

---

## Overview

Moot needs a council specialized for financial deliberation — trading decisions,
allocation changes, position sizing. This milestone builds the financial council as a
`CouncilPlugin`, defines financial-domain agent personas (including new Meridian agent),
adds reference classes for base-rate anchoring, implements deep-pass trigger criteria
for high-stakes decisions, and builds the moot-to-breadwinner verdict handler that
translates council output into actionable breadwinner signals.

A heartbeat scan is added to surface stale or urgent docket items.

**Prerequisite:** v0.4.0-council-spine must complete; the financial council is
implemented as a `CouncilPlugin` from the start.

- [ ] Heartbeat correctly surfaces docket issues stale longer than the configured

- [ ] Rampart is not present in any config or source file
- [ ] Meridian agent persona is defined and loaded as part of the financial council roster

## Scope

**Included:**

- Financial council manifest (YAML config + `CouncilPlugin` implementation)
- Agent personas: `financial_analyst`, `risk_officer`, `macro_strategist`,
  `sector_specialist`, `quant_researcher`, `meridian` (systems integrator)
- Financial domain reference classes: `FinancialDecision`, `PortfolioAction`, `RiskEvent`
- Deep-pass trigger logic: configurable thresholds for position size, drawdown %, and
  strategy change flag
- Verdict handler: structured outcome (`approve` / `reject` / `defer`) with rationale
- Heartbeat: checks for docket issues stale > N days or marked urgent
- Roster update: Rampart removed, Meridian added

**Explicitly excluded:**

- Reddit integration (v0.7.0)
- Options/crypto state context (v0.7.0)
- Direct moot ↔ breadwinner imports (all coordination through a defined contract)
- Real-time market data feeds

## Failure Modes

**Verdict handler boundary:**
- breadwinner endpoint unreachable → log error; do not retry automatically; surface
  in heartbeat as an undelivered verdict
- Verdict payload rejected by breadwinner → log error with response body; file issue
  for manual review
- Contract version mismatch → log error with version info; refuse to deliver rather
  than silently misdelivering

**Deep-pass failures:**
- Model upgrade call fails (Sonnet→Opus) → retry once; if still failing, fall back to
  Sonnet with a logged warning; do not block the deliberation
- Deep-pass threshold misconfigured (zero or negative value) → raise at startup, not
  at trigger time

**Council failures:**
- No agents available in financial council manifest → raise at startup
- Agent persona YAML malformed → raise at load time; do not proceed with partial roster

## Interface Contracts

**Financial council verdict (output from council, input to verdict handler):**

```json
{
  "outcome": "approve | reject | defer",
  "rationale": "<string>",
  "dissents": ["<agent_id>: <brief position>"],
  "confidence": "high | medium | low",
  "deep_pass_used": true
}
```

**Verdict handler → breadwinner signal (exact schema TBD in research stage):**

See Key Unknown #1 — research stage must define the interface before design begins.

**Deep-pass trigger criteria (configurable):**

| Trigger | Default threshold |
|---------|------------------|
| Position size change | > 10% of portfolio (TBD in research) |
| Drawdown | > configured drawdown % |
| Strategy change flag | Any |

## Constraints

- Financial council is a `CouncilPlugin` — no special-casing in the routing layer
- No direct imports from `breadwinner` in `moot` source
- Deep-pass thresholds are configurable via environment or config file, not hardcoded
- All new code is type-annotated, async-compatible, and passes `ruff check`

## Key Unknowns

1. **[P1]** What should the moot-to-breadwinner verdict handler interface look like?
   Specifically: what fields does a financial council verdict contain that breadwinner can
   act on, and what does breadwinner expose for moot to call? The answer determines the
   contract schema, the transport mechanism, and the error-handling surface. Research
   stage must answer this before the verdict handler is designed. The research output
   must be committed as `breadmin-jobman/docs/contracts/verdict-signal-v1.json` — the
   canonical schema home that moot, jobman, and breadwinner all reference. This file is
   a design stage deliverable.

2. **[P1]** What are the deep-pass trigger criteria for financial decisions? Specifically:
   what position size threshold, drawdown percentage, and other signals warrant switching
   from Sonnet to Opus for a full council deliberation? The answer affects both the
   trigger logic implementation and the configuration schema.

## Agent Identity

Financial council agents (Meridian, financial_analyst, risk_officer, macro_strategist,
sector_specialist, quant_researcher) are **stateless** — they do not use the six-layer
identity model defined for RGB council agents in v0.5.0. No personality seeds persist
across sessions; no journal or relationship map is maintained; Mirror integration is not
available for financial council agents. Financial agents reason from their persona
definition and the deliberation context, not from accumulated history.

If this decision is revisited, it belongs in a future milestone spec — it must not be
made implicitly by individual impl agents.
