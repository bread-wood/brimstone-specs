# breadforge v0.2.0 — Phase 2

**Theme:** Production hardening — cross-repo blocking, full research pipeline, credential proxy, GHA, multi-backend.

---

## Overview

breadforge v0.1.0 shipped a working DAG executor: plan → build → merge → readme, with
repair agents on CI failure and crash recovery. Phase 2 adds the features that make it
production-grade and complete the pipeline:

- **Cross-repo wait nodes** (2B): milestones blocked by foreign-repo releases dispatch
  a wait node instead of proceeding immediately
- **Full research pipeline** (2F): the plan handler emits two kinds of research nodes —
  web research (Key Unknowns) and codebase/design research (architecture questions) —
  both feeding into plan-refine, which synthesizes a richer PlanArtifact. Build nodes
  execute the plan artifact; they do not consume research findings directly.
- **Credential proxy** (2A): raw API keys are never in subprocess env; a loopback proxy
  holds the real key and issues scoped session tokens per agent
- **GHA deployment** (2C): breadforge runs unattended in CI, triggered by GitHub issues
  labeled `pipeline`, without a local Claude session
- **Multi-backend** (2D): research and plan nodes can use Gemini or GPT-4.1; build nodes
  always use Claude Code
- **Cross-model consensus** (2E): P0 unknowns run through two backends; contradictions
  are flagged `[DISPUTED]` before plan-refine sees the findings

## Success Criteria

- [ ] A milestone with `milestone_blocked_by` in its CampaignBead emits a `wait` node
  that polls `gh release view`; build dispatch holds until the release appears
- [ ] A spec with `novel-domain`, `security`, or `multi-module-coordination` risk flag
  emits design-flavored research nodes alongside web-research nodes; both feed
  plan-refine; build nodes execute from the resulting PlanArtifact only
- [ ] `ANTHROPIC_API_KEY` does not appear in any agent subprocess environment; health

  runner log shows `backend=gemini`

  contradictions `[DISPUTED]`; plan-refine node uses annotated output

## Scope

**2B — Cross-Repo Wait Nodes:**
- New node type `"wait"` in `NodeType`
- `WaitHandler`: polls `gh release view {tag} --repo {repo}` every 60s; resolves when release exists
- `graph/builder.py`: before plan node, check `CampaignBead.milestone_blocked_by`; emit wait node(s) with plan depending on them
- Status display: wait nodes show `waiting for {repo}@{tag}`

**2F — Full Research Pipeline:**

Research nodes are first-class nodes that already exist. The plan handler already emits
them for low-confidence unknowns and plan-refine already consumes their findings. 2F
extends this to cover design-stage questions too — no new node type needed.

Two research subtypes, distinguished by `context["task"]`:
- `task="research"` (existing): agent uses `WebSearch` + `WebFetch`; answers Key
  Unknowns from spec; findings stored as markdown
- `task="design"` (new): agent uses `Read` + `Glob` + `Bash` on the cloned repo;
  answers architecture, interface, and schema questions from the codebase itself; no web
  access; findings stored as markdown at the same path pattern

Both types feed into plan-refine via `_gather_research_findings()` — no change to that
function. Plan-refine synthesizes all findings into a richer PlanArtifact. Build nodes
execute from the plan artifact; they never read research findings directly.

Plan handler expansion logic (updated):
```
if confidence < 0.6 and unknowns:
    emit web-research nodes (task=research) for Key Unknowns
    if risk_flags has novel-domain | security | multi-module-coordination:
        also emit design research node (task=design)
    emit plan-refine depending on all research nodes
elif risk_flags has novel-domain | security | multi-module-coordination:
    emit design research node (task=design)
    emit plan-refine depending on design research node
else:
    emit build nodes directly
```

`ResearchHandler.execute()` branches on `context.get("task", "research")`:
- `task="research"`: `RESEARCH_PROMPT`, tools=`["WebSearch", "WebFetch"]`
- `task="design"`: `DESIGN_PROMPT`, tools=`["Read", "Glob", "Bash", "Edit"]`,
  passes `repo_local_path` and `plan_artifact` in prompt context

**Files changed:**
- `graph/node.py` — add `"wait"` to `NodeType`
- `graph/handlers/research.py` — branch on `context.get("task", "research")`
- `graph/handlers/plan.py` — add `_emit_design_research_node()`, `_needs_design()`,
  update expansion logic to emit design research node when warranted
- `graph/handlers/wait.py` — new: `WaitHandler`
- `graph/executor.py` — register `WaitHandler` in `make_handlers()`
- `agents/prompts.py` — add `DESIGN_PROMPT` template

**2A — Credential Proxy:**
- `agents/proxy.py`: loopback HTTP server (`localhost:{port}`); intercepts Anthropic base URL; holds real key in memory; issues scoped session tokens per agent; revokes on node completion
- `agents/runner.py`: `_build_env()` injects `ANTHROPIC_BASE_URL=http://localhost:{port}` + session token instead of raw key
- `config.py`: `proxy_port: int = 9473`
- `health.py`: proxy health check

**2C — GHA Deployment:**
- `.github/workflows/breadforge.yml`: triggered on `issues` event (label `pipeline`); ephemeral runner; reads `BREADFORGE_CONFIG` secret
- `cli.py`: `--ci` flag disables Rich live display, emits plain log output
- `health.py`: `check_gh_token()`, `check_no_tty()` for CI context

**2D — Multi-Backend:**
- `agents/backend.py`: `Backend` protocol; `ClaudeBackend` (wraps run_agent subprocess); `GeminiBackend` / `OpenAIBackend` (SDK direct; research + plan nodes only); `BackendRegistry` maps model name → backend
- Build nodes always use `ClaudeBackend`
- `config.py`: `research_model`, `plan_model` config fields (override per node type)

**2E — Cross-Model Consensus:**
- New node type `"consensus"` in `NodeType`
- `ConsensusHandler`: reads two research findings docs; calls LLM to diff; marks disagreements `[DISPUTED]`; writes merged findings
- `plan.py`: for P0 unknowns, emit two research nodes (one per backend) + one consensus node; plan-refine depends on consensus node

## Failure Modes

**Wait node:**
- Release never appears → wait node abandons after `max_retries` (configurable, default 1440 = 24h at 60s intervals)
- `gh release view` call fails → retry; log warning; do not abandon on transient failure

**Design research node:**
- Agent fails to produce usable findings → retry up to `max_retries=3`; if exhausted,
  plan-refine proceeds with only web-research findings; do not block the pipeline

**Credential proxy:**
- Proxy fails to start → health check fails; `breadforge run` exits with error

**GHA:**
- `BREADFORGE_CONFIG` secret missing → exit 1 with clear error

## Interface Contracts

**Wait node context:**
```json
{ "wait_repo": "owner/repo", "wait_tag": "v0.6.0", "poll_interval_seconds": 60 }
```

**Research node context (task=design):**
```json
{
  "milestone": "v0.4.0",
  "task": "design",
  "repo": "owner/repo",
  "repo_local_path": "/tmp/clone",
  "plan_artifact": { "...": "..." },
  "unknowns": ["What should the CouncilPlugin interface look like?"]
}
```

Design research findings are stored identically to web-research findings:
`~/.breadforge/beads/{owner}/{repo}/research/{node-id}.md`

Plan-refine reads both via `_gather_research_findings(research_node_ids, store)` —
no changes needed to that function or to how build nodes receive their context.

## Constraints

- 2B and 2F have no external dependencies — ship first
- 2A is prerequisite for 2C and 2D
- No new node type for design — reuse `"research"` with `task` field in context
- Build nodes consume only the PlanArtifact — no research findings passed to build context
- Design research nodes use codebase tools only; no WebSearch/WebFetch
- Build nodes always use `ClaudeBackend` (Claude Code subprocess)
